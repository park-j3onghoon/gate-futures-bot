1 포지셔닝: 정직하면서 강점 강조

하지 말아야 할 것: 경력 부풀리기

"Kotlin/Spring 3년 경력"으로 포장하면 즉시 들킵니다. 면접관은 30초 대화로 구분. 대신 "Python/Django 3년 경력 + 최근 Kotlin/Spring 학습 + 사이드 프로젝트"로 정직하게 제시. 티니어가 원하는 건 "학습 능력 + 문제 해결 + 프로덕트 마인드"지, 경력 년수 자체가 아닙니다.

이렇게 말하세요

"Python/Django로 3년간 백엔드를 개발했고, 최근 Kotlin/Spring 생태계로 전환을 준비 중입니다. gate-futures-bot이라는 사이드 프로젝트를 통해 Kotlin 언어, Spring Boot DI/설정, Kotlin 코루틴 기반 동시성, JPA 없이 sealed class로 도메인 모델링 등을 실전으로 익혔습니다. 프로덕션 경험은 Python이지만 Kotlin/Spring 이해는 경력 2-3년차 수준이라고 생각하고, 빠른 온보딩이 가능합니다."

2 프로젝트 스토리: 왜 만들었고 무엇을 배웠나

Why: 왜 이 프로젝트를 만들었는가

"Kotlin/Spring을 배우기 위해"는 약한 답변. 더 강한 서사:

"Gate.io 선물 거래를 수동으로 하다가 자동화의 필요성을 느꼈습니다. 마침 Kotlin/Spring 스택 전환을 준비 중이었고, 단순 예제가 아닌 실제 돈이 오가는 제약이 있는 프로젝트로 배우면 '진짜' 요구사항(정확성, 안전성, 관측성)을 고민하게 되어 빠르게 성장할 수 있겠다 판단했습니다. 학습용이지만 실제로 동작해야 한다는 제약이 모든 설계 선택의 기준이 됐습니다."

진행 타임라인 (실제 커밋 기준)

1단계
프로젝트 세팅 + 매수/매도 기본 — Gate.io SDK 래퍼(GateClient), sealed class 예외 계층, FuturesTrader. MockK 기반 테스트 도입.
2단계
포지션 청산 — close=true 플래그로 시장가 청산. 롱/숏 방향 자동 감지. 테스트 9개.
3단계
차트 데이터 + 지표 — Candle 모델, SMA/EMA/RSI top-level 순수 함수. 경계값 테스트(빈 리스트, period=0, 모든 상승).
4단계
자동 진입 + 동시성 — EntryCondition, TradingStrategy, 코루틴 워커, supervisorScope, Token Bucket RateLimiter. 에러 격리 설계.
5단계
자동 청산 — ExitCondition sealed class(TakeProfit/StopLoss/IndicatorExit), PnL 계산(롱/숏 부호 대칭), 전체 사이클 통합 테스트.
6-7단계
운영 준비 — BotRunner로 Spring 생명주기 연결, StrategyProperties로 외부 설정, @Validated 런타임 검증, MDC 구조화 로깅, MAINNET 경보.
리팩터링
architect 리뷰 기반 B+ → A-(데드 필드/중복 제거, WorkerConfig 추출) → A+(설정 검증, SRP 개선, 통합 테스트 추가).

3 STAR 패턴으로 설명하기

프로젝트 경험을 말할 때 구조화된 패턴을 사용하면 설득력이 올라갑니다.

S · Situation

문제 상황/배경

T · Task

맡은 과제

A · Action

취한 행동/결정

R · Result

결과/배운점

예시 1: 에러 격리 설계

  • S: 여러 코인을 병렬로 거래할 때, 하나의 코인 API 에러(rate limit, 네트워크 등)가 다른 코인 처리까지 중단시키는 문제
  • T: 한 워커 장애가 다른 워커에 전파되지 않는 동시성 구조 설계
  • A: Kotlin supervisorScope로 워커를 launch, 각 워커 내부는 try-catch로 CancellationException 외 모든 예외 격리. 공통 API rate limit은 Mutex 기반 Token Bucket으로 모든 워커가 공유. 테스트에선 BTC 워커에 예외를 던지고 ETH 워커가 정상 동작하는지 검증.
  • R: 한 코인 장애가 전체를 멈추지 않음. 이 과정에서 구조화된 동시성과 supervisor vs regular coroutineScope의 차이를 깊이 이해. 실무에선 여기에 Circuit Breaker를 추가할 것.

예시 2: 아키텍처 리팩터링

  • S: 초기 구현 완료 후 구조적 개선 여지가 있는지 점검 필요
  • T: "외부 시각"으로 전체를 평가하고 합리적 리팩터링
  • A: ①데드 필드 제거(BotProperties.takeProfitPct) ②EntryCondition과 ExitCondition.IndicatorExit의 평가 로직 중복을 evaluateIndicator로 추출 ③CoinWorker 10개 파라미터를 WorkerConfig로 묶음 ④StrategyFactory → bootstrap.StrategyAssembler(application-layer 명시) ⑤@Validated로 부팅 시 설정 검증 ⑥MDC 기반 구조화 로깅
  • R: 테스트 커버리지 82% 유지, 구조가 더 명확. 교훈: "처음부터 완벽"보다 "반복 개선이 가능한 구조"가 실용적. YAGNI를 적용해 "지금 필요없는 추상화"는 의도적으로 미룸.

4 이 프로젝트로 이야기할 수 있는 주제들

면접관이 "Kotlin/Spring 관련 구체적인 예시" 요청시 꺼낼 카드.

4.1 Kotlin 언어

  • Null safety: getPosition()이 Position? 반환, ?.let { ... }로 null-safe 체이닝
  • sealed class: ExitCondition(TakeProfit/StopLoss/IndicatorExit), ExitSignal(None/Close)로 닫힌 타입 계층 + exhaustive when
  • data class + companion object: 설정 DTO(OrderResult 등)와 팩토리 분리
  • 확장 함수 + 고차 함수: GateClient.callApi(context) { action() }로 try-catch + retry 공통화 (Python 데코레이터와 유사)
  • top-level 순수 함수: Indicators.kt의 SMA/EMA/RSI (클래스 불필요)
  • inline function: callApi는 inline으로 람다 객체 할당 제거

4.2 Kotlin 코루틴

  • suspend 함수: RateLimiter.acquire(), WorkerOrchestrator.runAll()
  • supervisorScope: 워커 간 에러 격리
  • Mutex + delay: Token Bucket에서 lock 안에서 대기 계산, lock 밖에서 delay
  • CancellationException rethrow: graceful shutdown 관용구
  • Dispatcher 선택: IO pool (HTTP 대기 많음)
  • runBlocking bridge: @PreDestroy(동기 콜백)에서 코루틴 cancel + join 대기

4.3 Spring Boot

  • @ConfigurationProperties: ApiProperties, BotProperties, StrategyProperties로 타입 안전 설정
  • @Validated + jakarta.validation: @NotBlank, @Positive로 부팅 시 검증
  • @Bean + @Configuration: Gate.io Java SDK의 FuturesApi를 @Bean으로 감싸 DI 가능하게
  • 생성자 주입: 모든 @Component/@Service가 val + 생성자 주입
  • @EventListener(ApplicationReadyEvent): 앱 시작 시 오케스트레이터 launch
  • @PreDestroy: 종료 시 graceful shutdown
  • Profile 분리: application.yml 기본 + dev/prod 파일, 민감 설정은 환경변수

4.4 설계 원칙

  • SDK 격리: gate-api 직접 사용은 GateClient에만 (3rd party 교체/mock 용이)
  • Strategy 순수성: TradingStrategy는 Spring 의존 없는 순수 클래스, 데이터를 파라미터로 받음 (백테스트 가능)
  • DDD 레이어링: config → bootstrap → domain → infrastructure 방향
  • Entry/Exit 의미 분리: 필드 비슷해도 별도 sealed 타입 (타입으로 혼용 방지)
  • YAGNI: RateLimiter interface는 유지(Redis 전환 대비 + 테스트 mock), model/ 분할은 거부(파일 3개뿐)

4.5 테스트

  • JUnit 5 + MockK: 103개 테스트, 커버리지 82%
  • 경계값 테스트: RSI 50/100/0, TakeProfit 정확 5% 경계, 빈 캔들
  • 코루틴 테스트: runTest(virtual time) / runBlocking(실시간) 혼용
  • 통합 테스트: @SpringBootTest로 StrategyProperties YAML 바인딩 검증
  • 전체 사이클 테스트: 진입 → 포지션 → 청산 플로우

4.6 운영 관점

  • MDC 구조화 로깅: contract를 MDC에 주입 → 로그 grep 용이
  • 환경 배너: 부팅시 MAINNET 감지 경보 (실거래 사고 방지)
  • no-op 경고: 전략 미정의 contract는 경고 로그
  • 향후 확장: Micrometer 커스텀 메트릭 추가 시 시나리오 (주문 성공률, 워커 지연, 포지션 게이지)

6 라이브 코딩 대응

자주 출제되는 유형

  • 알고리즘/자료구조: Kotlin 문법으로 풀기 (LeetCode easy/medium)
  • 간단한 API 구현: "Spring Boot로 /users CRUD 만들어보세요"
  • 주어진 코드 리팩터링: 버그 수정 + 테스트 작성
  • 프로젝트 확장: "이 프로젝트에 X 기능을 추가한다면?"

코딩 중 대화 팁

  • 생각을 말로: "먼저 빈 입력 케이스를 고려하고..." 면접관이 힌트 주기 쉬움
  • 제약 먼저 확인: "입력 크기는? null 허용? 성능 요구사항?"
  • naive 먼저: O(N²)라도 일단 동작 → 개선
  • 테스트 케이스 먼저: 입출력 예시 직접 작성
  • 막히면 솔직히: "이 부분을 잘 모르겠어요. 힌트 주시면 도움 될 것 같습니다"

Kotlin 관용구 연습

// Python 습관: for + if
val result = mutableListOf<Int>()
for (x in nums) {
    if (x > 0) result.add(x * 2)
}

// Kotlin 관용: 체이닝
val result = nums.filter { it > 0 }.map { it * 2 }

// Python 습관: if-else 문
val msg = if (score >= 60) "pass" else "fail"

// Python 습관: dict 접근
val value = map.get(key) ?: 0
// Python의 map.get(key, 0) 대응

// Kotlin 관용: 구조 분해
val (key, value) = entry
for ((k, v) in map) { /* ... */ }

7 시스템 설계 면접 대응

단계별 접근

  1. 요구사항 명확화 (5분): Functional + Non-functional (QPS, 지연, 데이터 크기, 일관성)
  2. 용량 추산 (3분): DAU, peak TPS, 저장 용량 대략 계산
  3. High-level 아키텍처 (10분): 박스+화살표로 Client → CDN → API → Cache → DB → Queue
  4. API 설계 (5분): 핵심 endpoint 3-5개, 요청/응답
  5. 데이터 모델 (5분): 주요 테이블/컬렉션, 관계, 인덱스
  6. Deep dive (10분): 면접관이 지적한 병목을 어떻게 해결
  7. trade-off: 내가 선택한 것과 대안 비교, 한계 인정

자주 나오는 시스템

  • 채팅 앱 (WhatsApp) — WebSocket, 메시지 순서, 오프라인 동기화
  • URL shortener — 해시 충돌, 분산 ID 생성, 리다이렉트 캐싱
  • 뉴스피드 (Twitter/Instagram) — fanout, ranking, 캐싱 전략
  • 레이팅/카운터 (좋아요, 조회수) — Redis INCR, 배치 동기화
  • 광고 시스템 — 실시간 경매, 빈도 제어, 집계
티니어 관련 예상 주제

날씨/번역/키보드 같은 유틸리티 앱의 백엔드 설계. 예시: "전 세계 6천만 MAU에게 현재 위치의 날씨를 빠르게 제공하는 시스템". 힌트: CDN 캐시(격자 기준 위도/경도 반올림으로 cache key) + 지역별 데이터 파티셔닝 + 주기적 prewarm.

8 약점 솔직하게 인정하기

"약점이 무엇인가요?"는 단골. 솔직 + 개선 노력 = 좋은 답변.

Bad: "저는 완벽주의라 시간이 오래 걸립니다" (가짜 약점)
Good: "프로덕션 Kotlin/Spring 경험은 아직 없습니다. 사이드 프로젝트로 주요 개념을 익혔지만, 실제 대규모 트래픽 운영 시 발생하는 문제들(GC 튜닝, 장애 대응 실전, 팀 협업 속 리뷰)은 경험이 부족합니다. 그래서 입사 초 3-6개월은 시니어의 리뷰를 적극 받고 페어 프로그래밍으로 배우고 싶습니다. 대신 Python/Django 3년 경험에서 온 백엔드 기본기(DB 설계, API, 운영 의식)는 자산이라고 생각합니다."
