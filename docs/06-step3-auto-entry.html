1 왜 전략을 순수 클래스로 만드나

TradingStrategy는 Spring 의존 없는 순수 클래스입니다. @Component도 없고, 다른 Spring Bean을 주입받지 않습니다.

이유
  • 테스트 용이: TradingStrategy(longEntries = ...)로 즉시 생성
  • 백테스트 가능: 동일 클래스를 과거 데이터로 평가
  • 파라미터로 데이터 받음: evaluateEntry(candles) — strategy가 직접 API 호출 금지

2 EntryCondition 설계

진입 조건 하나는 아래 4가지로 표현:

  • Indicator: 어떤 지표를 볼 것인가 (RSI/SMA/EMA/PRICE)
  • ComparisonOp: 어떤 비교 (LT/LTE/GT/GTE)
  • value: 비교할 임계값
  • period: 지표의 기간 (기본 14)
enum class Indicator { RSI, SMA, EMA, PRICE }
enum class ComparisonOp { LT, LTE, GT, GTE }

data class EntryCondition(
    val indicator: Indicator,
    val operator: ComparisonOp,
    val value: Double,
    val period: Int = 14
) {
    init { require(period > 0) { "period must be positive: $period" } }

    fun evaluate(prices: List<Double>): Boolean {
        val current: Double = when (indicator) {
            Indicator.RSI -> calculateRsi(prices, period) ?: return false
            Indicator.SMA -> calculateSma(prices, period) ?: return false
            Indicator.EMA -> calculateEma(prices, period) ?: return false
            Indicator.PRICE -> prices.lastOrNull() ?: return false
        }
        return when (operator) {
            ComparisonOp.LT -> current < value
            ComparisonOp.LTE -> current <= value
            ComparisonOp.GT -> current > value
            ComparisonOp.GTE -> current >= value
        }
    }
}
when 식의 exhaustiveness

Kotlin when은 enum/sealed class의 모든 분기를 커버하면 else가 필요 없습니다. 나중에 Indicator에 값을 추가하면 컴파일 에러로 누락을 알려줍니다 (안전장치).

3 TradingStrategy.evaluateEntry

롱/숏 각각의 조건 리스트를 받습니다. 각 side 내부는 AND로 결합 — 모든 조건이 충족되어야 시그널 발생. 

class TradingStrategy(
    private val longEntries: List<EntryCondition> = emptyList(),
    private val shortEntries: List<EntryCondition> = emptyList(),
    private val exitConditions: List<ExitCondition> = emptyList()
) {
    fun evaluateEntry(candles: List<Candle>): EntrySignal {
        if (candles.isEmpty()) return EntrySignal.None
        val prices = candles.map { it.closePrice }

        if (longEntries.isNotEmpty() && longEntries.all { it.evaluate(prices) }) {
            return EntrySignal.Long(longEntries)
        }
        if (shortEntries.isNotEmpty() && shortEntries.all { it.evaluate(prices) }) {
            return EntrySignal.Short(shortEntries)
        }
        return EntrySignal.None
    }
}
Long 우선

둘 다 충족되면 Long이 선택됩니다. 이는 "전략 설계 오류" 신호일 수 있으므로 KDoc에 명시. 보통 Long/Short 조건은 상호 배타적으로 설계합니다 (예: RSI<30 vs RSI>70).

4 EntrySignal sealed class

반환 타입은 3가지뿐. sealed class로 컴파일 타임에 모든 경우를 강제합니다. 

sealed class EntrySignal {
    data object None : EntrySignal()
    data class Long(val matched: List<EntryCondition>) : EntrySignal()
    data class Short(val matched: List<EntryCondition>) : EntrySignal()
}

사용처는 when으로 분기:

when (val signal = strategy.evaluateEntry(candles)) {
    is EntrySignal.Long -> trader.openLong(contract, orderSize, leverage)
    is EntrySignal.Short -> trader.openShort(contract, orderSize, leverage)
    EntrySignal.None -> logger.debug("시그널 없음")
}
data object vs object

Kotlin 1.9+에서 data object는 자동으로 toString()/equals()/hashCode()를 적절히 구현. object None만 쓰면 기본 Object identity 비교라 로그 가독성이 낮습니다.

5 application.yml 외부 설정

전략을 코드에 하드코딩하지 않고 application.yml에서 주입합니다. 

strategy:
  contracts:
    BTC_USDT:
      long-entries:
        - indicator: RSI
          operator: LT
          value: 30.0
          period: 14
      short-entries:
        - indicator: RSI
          operator: GT
          value: 70.0
          period: 14

@ConfigurationProperties로 바인딩:

@ConfigurationProperties(prefix = "strategy")
data class StrategyProperties(
    val contracts: Map<String, ContractStrategySpec> = emptyMap()
)

@Component
class StrategyFactory(private val properties: StrategyProperties) {
    fun forContract(contract: String): TradingStrategy {
        val spec = properties.contracts[contract] ?: return TradingStrategy()
        return TradingStrategy(
            longEntries = spec.longEntries.map { it.toEntryCondition() },
            shortEntries = spec.shortEntries.map { it.toEntryCondition() },
            exitConditions = spec.exitConditions.map { it.toExitCondition() }
        )
    }
}

4단계 마무리

Strategy는 순수 판단 레이어. 데이터 조회는 Worker가, 주문은 Trader가. 각 레이어의 책임이 분리되어 각자를 독립적으로 테스트할 수 있습니다.
