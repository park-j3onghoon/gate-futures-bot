# ADR-0002 — Gate.io SDK 격리 (단일 진입점 GateClient)

- **Status**: Accepted
- **Date**: 2026-05-19 (회고 작성)

## Context

`io.gate:gate-api` SDK는 Gate.io 응답 DTO(`FuturesOrder`, `FuturesCandlestick`, `Position` 등)를 Java 빈으로 노출한다.
이를 모든 레이어(전략, 워커, 포지션 추적기 등)가 직접 임포트해 쓰면:

1. SDK 버전 업그레이드 시 광범위한 변경
2. SDK가 가진 (가변, nullable, 단위 불명확) 타입 특성이 도메인 전체로 누출
3. SDK 호출 실패/timeout/Rate limit 처리가 분산
4. 거래소 교체(Gate.io → Binance 등) 시 사실상 전부 재작성

## Decision

**Gate.io SDK 임포트는 `client/GateClient`에만 허용**한다.

- `GateClient`는 SDK DTO → 도메인 타입(`Candle`, `OrderResult`, `Position`)으로 **변환 책임**을 갖는다
- 나머지 레이어(전략, 워커, 포지션, 알림)는 **도메인 타입만 다룬다**
- 이 경계는 ArchUnit 테스트로 강제 (ADR-0005 참조)

## Alternatives Considered

- **SDK 타입을 그대로 도메인에서 사용**: 변환 비용 없음. → ❌ 위 Context의 4개 문제 모두 발생. 거래소 격리 0.
- **각 모듈이 필요한 SDK 호출을 직접**: 빠른 작성. → ❌ Rate limit/에러 처리/로깅이 분산됨.

## Consequences

### Positive
- SDK 변경 영향이 `client/` 한 디렉토리로 국한
- 도메인 타입은 의도가 명확한 데이터 클래스(필드명·단위 통일)
- Mock하기 쉬움 → 전략/워커 단위 테스트 단순화
- 거래소 추가(Hexagonal Port — ADR-0008)로 자연스럽게 확장

### Negative
- SDK가 새 필드를 노출해도 도메인 타입에 매핑해야 사용 가능 (한 단계 추가)
- 매핑 코드 자체가 일종의 boilerplate

### Neutral
- 변환 로직은 단일 파일(`GateClient`)에 집중되므로 검토 용이

## Follow-up

- Hexagonal `ExchangePort` 도입 (ADR-0008)으로 이 결정의 가치가 한 단계 더 살아남
- 거래소 추가 시 `client/` 아래 새 Adapter + 도메인 타입 매핑 추가
