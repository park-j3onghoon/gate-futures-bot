1 Gradle이란

Python 개발에서는 여러 도구를 조합합니다: pip으로 의존성 설치, pytest로 테스트, Makefile로 자동화. Gradle은 이 모든 걸 하나로 합친 도구입니다.

역할 Python 세계 Gradle
의존성 관리 pip install -r requirements.txt dependencies { } 블록
패키지 저장소 PyPI (pypi.org) Maven Central (mavenCentral())
빌드/컴파일 필요 없음 (인터프리터 언어) compileKotlin 태스크
테스트 실행 pytest ./gradlew test
자동화 스크립트 Makefile tasks { } 블록
패키징 python setup.py bdist_wheel ./gradlew bootJar
설정 파일 형식 pyproject.toml / setup.cfg build.gradle.kts (Kotlin 스크립트)
Tip: .kts는 뭔가

build.gradle.kts의 .kts는 "Kotlin Script"의 약자입니다. 빌드 설정을 Kotlin 문법으로 작성합니다. 예전에는 Groovy 언어로 build.gradle을 썼는데, 요즘은 Kotlin DSL(.kts)이 표준입니다. 타입 추론과 자동완성이 되는 장점이 있습니다.

2 build.gradle.kts 한 줄씩 해부

이 프로젝트의 실제 build.gradle.kts 파일을 블록별로 분해해서 설명합니다.

plugins 블록: "이 프로젝트에 어떤 기능을 켤지"

plugins {
    kotlin("jvm") version "2.1.20"              // (1)
    kotlin("plugin.spring") version "2.1.20"    // (2)
    id("org.springframework.boot") version "3.5.3"  // (3)
    id("io.spring.dependency-management") version "1.1.7"  // (4)
    id("jacoco")                                // (5)
}
# 플러그인 하는 일 Python 비유
(1) kotlin("jvm") Kotlin 소스를 JVM 바이트코드로 컴파일 Python 인터프리터 자체에 해당
(2) kotlin("plugin.spring") Kotlin 클래스를 Spring이 쓸 수 있게 open 처리 없음 (Python 클래스는 기본적으로 상속 가능)
(3) org.springframework.boot 실행 가능한 JAR 생성, bootRun 태스크 추가 manage.py runserver 가능하게 하는 것
(4) io.spring.dependency-management Spring 관련 라이브러리 버전 자동 맞춤 requirements.txt에서 버전 충돌 해결
(5) jacoco 코드 커버리지 측정 pytest --cov
Tip: kotlin("plugin.spring")이 필요한 이유

Kotlin 클래스는 기본적으로 final(상속 불가)입니다. 하지만 Spring은 내부적으로 프록시(상속 기반)를 만들어 DI를 처리합니다. 이 플러그인은 Spring 관련 클래스를 자동으로 open(상속 가능)으로 변환합니다. Python에서는 클래스가 기본적으로 상속 가능하므로 이런 문제가 없습니다.

설정 값 Python 비유
group "com.parkj3onghoon" setup.py의 author / 패키지 네임스페이스
version "0.0.1-SNAPSHOT" setup.py의 version
toolchain JDK 21 python_requires='>=3.12'
Tip: SNAPSHOT이란

SNAPSHOT은 "아직 정식 릴리스가 아닌 개발 중인 버전"이라는 표시입니다. Python에서 0.0.1.dev0 또는 0.0.1a1 같은 프리릴리스 버전과 같은 개념입니다. 정식 출시 시 0.1.0으로 바꿉니다.

각 의존성이 하는 일

# 라이브러리 역할 Python 비유
(1) spring-boot-starter Spring Boot 핵심 (DI, 설정, 로깅) django 패키지 자체
(2) configuration-processor 설정 바인딩 시 자동완성/검증 지원 django-environ
(3) kotlin-reflect Kotlin 리플렉션 (런타임 타입 검사) inspect 모듈
(4) kotlinx-coroutines-core 코루틴 라이브러리 asyncio (내장이 아닌 외부 라이브러리)
(5) gate-api Gate.io 거래소 API SDK ccxt 또는 거래소 공식 SDK
(6) spring-boot-starter-test 테스트 프레임워크 묶음 (JUnit 5 등) pytest + django.test
(7) kotlin-test-junit5 Kotlin 테스트 유틸 + JUnit 5 연동 pytest 확장
(8) mockk Kotlin용 모킹 라이브러리 unittest.mock
(9) kotlinx-coroutines-test 코루틴 테스트 유틸 pytest-asyncio
(10) junit-platform-launcher JUnit 테스트 실행 엔진 pytest 내부 러너

kotlin 블록: 컴파일러 옵션

kotlin {
    compilerOptions {
        freeCompilerArgs.addAll("-Xjsr305=strict")
    }
}

-Xjsr305=strict는 Java 라이브러리의 null 어노테이션을 Kotlin이 엄격하게 해석하도록 합니다. 예를 들어 Java 라이브러리가 @NonNull을 썼으면 Kotlin에서도 null 불가로 취급합니다.

Tip

이 설정은 Java ↔ Kotlin 상호운용에서 null safety를 더 강화하는 옵션입니다. 순수 Kotlin 코드만 쓴다면 큰 영향은 없지만, Spring Boot 내부는 Java로 작성되어 있으므로 이 설정이 유용합니다.

tasks 블록: 빌드 자동화 설정

// 테스트 실행 시 JUnit 5 플랫폼 사용
tasks.withType<Test> {
    useJUnitPlatform()
}

// JaCoCo 커버리지 리포트 설정
tasks.jacocoTestReport {
    dependsOn(tasks.test)      // 테스트 먼저 실행 후 리포트 생성
    reports {
        xml.required = true     // CI에서 읽을 XML 리포트
        html.required = true    // 브라우저로 볼 HTML 리포트
    }
}
Python: Makefile
# Makefile
test:
    pytest --cov=myapp --cov-report=html

lint:
    flake8 myapp/

all: lint test
Gradle: tasks 블록
// build.gradle.kts
tasks.withType<Test> {
    useJUnitPlatform()
}

tasks.jacocoTestReport {
    dependsOn(tasks.test)
    reports {
        xml.required = true
        html.required = true
    }
}

3 Gradle Wrapper 깊이 이해

sequenceDiagram participant 개발자 participant gradlew as gradlew (쉘 스크립트) participant jar as gradle-wrapper.jar participant cache as ~/.gradle/ participant internet as services.gradle.org 개발자->>gradlew: ./gradlew build gradlew->>jar: Gradle 있나 확인 alt 캐시에 Gradle 없음 (첫 실행) jar->>internet: Gradle 8.14.4 다운로드 internet-->>jar: gradle-8.14.4-bin.zip jar->>cache: 저장 (~/.gradle/wrapper/dists/) end jar->>cache: Gradle 8.14.4 로드 cache-->>개발자: 빌드 태스크 실행!

파일별 역할

파일 역할 Git에 포함?
gradlew Mac/Linux용 실행 스크립트 (bash) Yes
gradlew.bat Windows용 실행 스크립트 Yes
gradle-wrapper.jar Gradle 다운로더 (작은 Java 프로그램, ~60KB) Yes
gradle-wrapper.properties 사용할 Gradle 버전 URL Yes
~/.gradle/wrapper/dists/ 다운로드된 Gradle 바이너리 캐시 No (로컬)
핵심 요약

Gradle Wrapper 파일들은 반드시 Git에 커밋합니다. 이렇게 하면 프로젝트를 clone한 누구나 ./gradlew build 한 줄로 정확히 같은 버전의 Gradle로 빌드할 수 있습니다. Gradle을 시스템에 별도 설치할 필요가 없습니다.

4 주요 명령어

Gradle 명령어 하는 일 Python 대응
./gradlew build 컴파일 + 테스트 + JAR 생성 (전부) pip install -e . && pytest && python setup.py bdist_wheel
./gradlew test 테스트만 실행 pytest
./gradlew bootRun 애플리케이션 실행 python manage.py runserver
./gradlew clean build/ 디렉토리 삭제 (처음부터 다시) rm -rf __pycache__ .pytest_cache build dist
./gradlew dependencies 의존성 트리 출력 pip freeze (더 상세함)
./gradlew tasks 사용 가능한 모든 태스크 목록 make help
./gradlew jacocoTestReport 테스트 커버리지 리포트 생성 pytest --cov --cov-report=html

명령어 실행 예시

# 빌드 (가장 많이 씀)
./gradlew build

# 테스트만
./gradlew test

# 앱 실행
./gradlew bootRun

# 클린 빌드 (캐시 문제 시)
./gradlew clean build

# 의존성 트리 확인
./gradlew dependencies --configuration runtimeClasspath

# 특정 테스트만 실행
./gradlew test --tests "com.parkj3onghoon.gatefuturesbot.MyTest"

# 테스트 커버리지
./gradlew jacocoTestReport
# → build/reports/jacoco/test/html/index.html 을 브라우저로 열기
Tip: clean build는 언제 쓰나

빌드가 이상하게 실패할 때 ./gradlew clean build를 실행하면 대부분 해결됩니다. clean은 build/ 디렉토리를 통째로 삭제하고 처음부터 빌드합니다. Python에서 __pycache__를 삭제하고 다시 실행하는 것과 같습니다.

build = compileKotlin + processResources + test + jar + bootJar + check

Tip: JAR이란

JAR (Java ARchive)은 컴파일된 코드 + 리소스를 하나의 ZIP 파일로 묶은 것입니다. bootJar로 생성된 JAR은 java -jar gate-futures-bot-0.0.1-SNAPSHOT.jar로 바로 실행할 수 있습니다. Python의 .whl 파일과 비슷하지만, 실행까지 가능한 점이 다릅니다.
