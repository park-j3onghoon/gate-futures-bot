1 왜 코루틴인가

여러 코인을 동시에 거래하려면 동시성이 필요합니다. 3가지 선택지:

접근메모리에러 격리공유 상태학습 가치
별도 JVM 프로세스 ~150MB × 코인수 OS 수준 (완벽) IPC 필요 (Redis/File) 낮음
스레드 (Thread) ~1MB / 스레드 없음 (직접 구현) 같은 메모리 (락 필요) 중간
코루틴 (채택) ~수 KB / 코루틴 supervisorScope 같은 메모리 (Mutex) 높음
코루틴 = suspend 함수 + Dispatcher

suspend fun은 "중간에 멈췄다 나중에 재개할 수 있는 함수". delay(1000)은 스레드를 블록하지 않고 다른 코루틴에 양보합니다. 수만 개의 코루틴이 수십 개의 스레드 위에서 돌 수 있습니다.

2 supervisorScope: 에러 격리

기본 coroutineScope는 자식 코루틴 하나가 실패하면 형제 모두 취소됩니다.
supervisorScope는 자식 실패가 형제에 전파되지 않습니다 — 매매 봇에 필수.

        botProperties.contracts.forEach { contract ->
            launch { createWorker(contract, strategyProvider(contract)).run() }
        }
    }

3 Token Bucket RateLimiter

Gate.io는 초당 API 호출 수 제한이 있습니다. 모든 워커가 공유하는 토큰 버킷을 사용합니다.

Token Bucket 알고리즘
  • 버킷에 토큰이 capacity개 있음
  • 초당 refillPerSec개씩 채워짐
  • API 호출 전 acquire()로 토큰 1개 소비
  • 토큰이 없으면 생길 때까지 delay로 대기
@Component
class InMemoryRateLimiter(
    private val capacity: Int = 20,
    private val refillPerSec: Double = 5.0,
    private val nowNanos: () -> Long = System::nanoTime
) : RateLimiter {

    private val mutex = Mutex()
    private var tokens: Double = capacity.toDouble()
    private var lastRefillNanos: Long = nowNanos()

    override suspend fun acquire() {
        while (true) {
            val waitMillis = mutex.withLock {
                refill()
                if (tokens >= 1.0) {
                    tokens -= 1.0
                    return
                }
                val needed = 1.0 - tokens
                (needed / refillPerSec * 1_000).toLong().coerceAtLeast(1L)
            }
            delay(waitMillis)  // mutex 밖에서 delay (다른 코루틴 진입 허용)
        }
    }

    private fun refill() {
        val now = nowNanos()
        val elapsed = now - lastRefillNanos
        if (elapsed <= 0) return
        val refilled = elapsed.toDouble() / 1_000_000_000.0 * refillPerSec
        tokens = (tokens + refilled).coerceAtMost(capacity.toDouble())
        lastRefillNanos = now
    }
}
Mutex 내부에서 delay 금지

withLock { ... delay(...) }는 다른 코루틴의 acquire를 블록합니다. 이 구현은 lock 안에서 대기 시간만 계산하고, lock 밖에서 delay.

4 CoinWorker 루프

각 코인마다 하나의 코루틴. 캔들 캐시 + 주기적 체크:

suspend fun run() {
    try {
        while (currentCoroutineContext().isActive) {
            runOnce()
            delay(checkIntervalMillis)
        }
    } finally {
        logger.info("워커 종료: contract={}", contract)
    }
}

internal suspend fun runOnce() {
    try {
        rateLimiter.acquire()
        val position = trader.getCurrentPosition(contract)
        updateCandles()
        if (position != null) evaluateAndClose(position) else evaluateAndEnter()
    } catch (e: CancellationException) {
        throw e  // 취소 신호는 rethrow
    } catch (e: Exception) {
        logger.error("워커 iteration 에러", e)  // 다른 에러는 로깅 후 계속
    }
}

캔들 캐시 — 델타 조회

매 iteration마다 전체 캔들을 받으면 API 부담이 큽니다. 첫 조회만 전체, 이후는 fromSec 파라미터로 델타. 캐시 크기 상한(maxCacheSize)을 두어 OOM 방지. 

internal fun updateCandles() {
    val lastTs = candleCache.lastOrNull()?.timestamp
    val fetched = if (lastTs == null) {
        marketData.getCandles(contract, interval, limit = 100)
    } else {
        marketData.getCandles(contract, interval, fromSec = lastTs)
    }
    val existingTs = candleCache.mapTo(HashSet()) { it.timestamp }
    val fresh = fetched.filter { it.timestamp !in existingTs }
    candleCache.addAll(fresh)
    trimCache()
}

5 WorkerOrchestrator

BotProperties의 contracts 리스트를 받아 각 코인 워커를 launch. supervisorScope로 감싸 에러를 격리. 

@Component
class WorkerOrchestrator(
    private val botProperties: BotProperties,
    private val marketData: MarketDataService,
    private val trader: FuturesTrader,
    private val rateLimiter: RateLimiter,
    private val interval: Interval = Interval.MIN_1
) {
    fun createWorker(contract: String, strategy: TradingStrategy) = CoinWorker(
        contract = contract,
        interval = interval,
        strategy = strategy,
        marketData = marketData,
        trader = trader,
        rateLimiter = rateLimiter,
        orderSize = botProperties.orderSize,
        leverage = botProperties.leverage,
        checkIntervalMillis = botProperties.checkIntervalSec * 1_000L
    )

    suspend fun runAll(strategyProvider: (String) -> TradingStrategy) = supervisorScope {
        botProperties.contracts.forEach { contract ->
            launch { createWorker(contract, strategyProvider(contract)).run() }
        }
    }
}

6 BotRunner: Spring 생명주기 연결

Spring 앱이 완전히 시작된 시점(ApplicationReadyEvent)에 Orchestrator를 구동. 앱 종료 시 @PreDestroy로 모든 워커를 정리. 

@Component
class BotRunner(
    private val orchestrator: WorkerOrchestrator,
    private val strategyFactory: StrategyFactory
) {
    private val scope = CoroutineScope(Dispatchers.Default + SupervisorJob())
    private var job: Job? = null

    @EventListener(ApplicationReadyEvent::class)
    fun start() {
        if (job != null) return
        job = scope.launch {
            orchestrator.runAll { contract -> strategyFactory.forContract(contract) }
        }
    }

    @PreDestroy
    fun stop() {
        val current = job ?: return
        runBlocking { current.cancelAndJoin() }
        scope.cancel()
        job = null
    }
}
왜 runBlocking

@PreDestroy는 동기 콜백입니다. 여기서 코루틴을 비동기로 cancel하고 바로 리턴하면 워커가 완전히 정리되기 전에 Spring이 종료합니다. runBlocking으로 완료까지 블록.
