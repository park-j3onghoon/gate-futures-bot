1 확장의 기본

Stateless 서비스는 쉽게 수평 확장. Stateful(DB, 세션)은 공유 스토리지 또는 샤딩으로 해결.

Stateless 전환의 핵심

  • 세션: 서버 메모리 → Redis(Spring Session)
  • 파일: 로컬 디스크 → S3
  • 캐시: 로컬 맵 → Redis
  • 스케줄러: 단일 서버 cron → 분산 잡 큐 (Quartz cluster, AWS EventBridge)

2 CAP 정리

분산 시스템은 Consistency(일관성), Availability(가용성), Partition Tolerance(분할 내성) 중 2가지만 보장 가능. 네트워크는 언제든 분할될 수 있으므로 실전에선 CP vs AP 선택.

Consistency

모든 노드가 동일 데이터

RDBMS (PostgreSQL)

Availability

장애시에도 응답

DynamoDB, Cassandra

Partition Tolerance

네트워크 분할 견딤

모든 분산 시스템 필수
실전 의미

"네트워크가 끊어졌을 때 일관성을 희생하고 응답할지(AP), 가용성을 희생하고 거부할지(CP)"의 선택. 은행 송금은 CP(잘못된 잔고 응답 금지), 상품 조회는 AP(약간 stale해도 응답). 시스템 내 여러 컴포넌트가 각자 다른 선택을 할 수 있음.

3 계층형 캐싱 전략

지연 감소의 핵심. 데이터 소스에 가까울수록 빠르고 비싸고, 멀수록 느리고 저렴.

① Browser Cache (HTTP Cache-Control, localStorage) ~0ms (로컬)
② CDN (CloudFront, CloudFlare) — 정적 파일 + cacheable API ~10ms (edge)
③ API Gateway Cache (AWS API Gateway, Nginx) ~5ms (region)
④ Application Local Cache (Caffeine) — 같은 JVM 내 hot data ~0.1ms
⑤ Distributed Cache (Redis/ElastiCache) ~1ms
⑥ Database (PostgreSQL, DynamoDB) ~20~100ms

Local + Distributed Cache 조합 (2-tier)

// Caffeine (로컬) + Redis (공유)
@Configuration
class CacheConfig {
    @Bean
    fun cacheManager(redisTemplate: RedisTemplate<String, Any>): CacheManager {
        val caffeine = CaffeineCacheManager().apply {
            setCaffeine(Caffeine.newBuilder()
                .maximumSize(10_000)
                .expireAfterWrite(Duration.ofMinutes(1))  // 로컬: 1분
                .recordStats()
            )
        }

        val redis = RedisCacheManager.builder(redisTemplate.connectionFactory!!)
            .cacheDefaults(
                RedisCacheConfiguration.defaultCacheConfig()
                    .entryTtl(Duration.ofMinutes(10))    // Redis: 10분
            ).build()

        // CompositeCacheManager: 로컬 miss시 Redis 조회
        return CompositeCacheManager(caffeine, redis)
    }
}
2-tier 캐시의 장단점

장점: Redis 왕복도 생략 → 0.1ms, 네트워크 부하 절감.
단점: 인스턴스 간 동기화 어려움. TTL 짧게(1분) + 이벤트 기반 invalidation (Redis pub/sub)으로 완화.
핫 데이터(카테고리, 설정, 환율)에 효과적.

4 메시지 큐와 이벤트 기반

동기 호출의 단점(응답 기다림, 커플링, 장애 전파)을 비동기 메시지로 분리.

동기 vs 비동기 비교

// ❌ 동기: 주문 API가 이메일/알림/통계 모두 대기 (1.5초)
@PostMapping("/orders")
fun create(req: CreateOrderRequest): OrderResponse {
    val order = orderService.save(req)       // 50ms
    emailService.sendConfirmation(order)      // 300ms
    pushService.notifyMerchant(order)         // 500ms
    analyticsService.track(order)             // 300ms
    recommendationService.update(order)       // 400ms
    return OrderResponse.from(order)
    // 총: 1550ms, 이메일 실패시 주문도 실패
}

// ✅ 비동기: 주문은 빠르게 응답, 나머지는 이벤트로 처리
@PostMapping("/orders")
fun create(req: CreateOrderRequest): OrderResponse {
    val order = orderService.save(req)
    eventPublisher.publishEvent(OrderCreatedEvent(order.id))  // 1ms
    return OrderResponse.from(order)
    // 총: 51ms
}

@Component
class OrderEventHandlers {
    @Async
    @EventListener(OrderCreatedEvent::class)
    fun sendEmail(event: OrderCreatedEvent) { /* ... */ }

    @Async
    @EventListener(OrderCreatedEvent::class)
    fun notifyMerchant(event: OrderCreatedEvent) { /* ... */ }
}

Kafka 예시 (Spring Kafka)

// Producer
@Service
class OrderService(private val kafka: KafkaTemplate<String, OrderCreatedEvent>) {
    fun placeOrder(req: CreateOrderRequest): Order {
        val order = save(req)
        kafka.send("orders.created", order.id.toString(), OrderCreatedEvent.from(order))
        return order
    }
}

// Consumer
@Component
class OrderConsumer {
    @KafkaListener(topics = ["orders.created"], groupId = "email-service")
    fun onOrderCreated(event: OrderCreatedEvent) {
        emailService.send(event)
    }
}
이벤트 기반 설계 원칙
  • Idempotency(멱등성): 같은 이벤트 재처리해도 결과 동일해야 함 (at-least-once 가정)
  • Outbox 패턴: DB 커밋과 이벤트 발행을 같은 트랜잭션으로 보장
  • Dead Letter Queue: 계속 실패하는 메시지 격리
  • 이벤트 스키마 관리: Avro/Protobuf + Schema Registry
  • Saga 패턴: 분산 트랜잭션 → 보상 트랜잭션 연쇄

Outbox 패턴

// 문제: save + publish가 원자적이지 않으면 불일치
@Transactional
fun placeOrder(req: CreateOrderRequest): Order {
    val order = orderRepo.save(...)       // ✅ DB 커밋
    kafka.send(...)                        // ❌ 발행 실패시 DB엔 주문있는데 이벤트 없음
    return order
}

// 해결: Outbox — 이벤트를 DB에 먼저 저장, 별도 publisher가 읽어 발행
@Transactional
fun placeOrder(req: CreateOrderRequest): Order {
    val order = orderRepo.save(...)
    outboxRepo.save(OutboxEvent(
        topic = "orders.created",
        payload = OrderCreatedEvent.from(order)
    ))
    return order
    // 커밋시 order + outbox 원자적
}

@Scheduled(fixedDelay = 1000)
fun publishOutbox() {
    outboxRepo.findPending().forEach { event ->
        kafka.send(event.topic, event.payload)
        event.markPublished()
    }
    // 실패시 다음 주기에 재시도 — at-least-once
}

5 분산 락

여러 인스턴스가 공유 자원에 동시 접근하는 것을 막기.

Redis 기반 분산 락 (Redisson)

// build.gradle.kts
implementation("org.redisson:redisson-spring-boot-starter:3.34.1")

@Service
class OrderService(private val redisson: RedissonClient) {

    fun processPayment(orderId: Long) {
        val lock = redisson.getLock("order:payment:$orderId")
        try {
            // 최대 5초 대기, 30초 lease
            if (lock.tryLock(5, 30, TimeUnit.SECONDS)) {
                // Critical section
                doProcessPayment(orderId)
            } else {
                throw ConflictException("다른 요청이 처리 중")
            }
        } finally {
            if (lock.isHeldByCurrentThread) lock.unlock()
        }
    }
}

단순 SET NX 방식 (Raw Redis)

val lockKey = "lock:order:$orderId"
val token = UUID.randomUUID().toString()

// acquire: SET if not exists + expire 원자적
val acquired = redisTemplate.opsForValue().setIfAbsent(
    lockKey, token, Duration.ofSeconds(30)
) ?: false

if (acquired) {
    try {
        // Critical section
    } finally {
        // release: 내 토큰일 때만 삭제 (Lua script로 원자적)
        val luaScript = """
            if redis.call('get', KEYS[1]) == ARGV[1] then
                return redis.call('del', KEYS[1])
            else
                return 0
            end
        """.trimIndent()
        redisTemplate.execute(
            DefaultRedisScript(luaScript, Long::class.java),
            listOf(lockKey), token
        )
    }
}
분산 락 주의사항
  • TTL 필수: 인스턴스 crash시 락 영원히 걸림 방지
  • 비즈니스 처리 시간 < TTL: 락 만료 중 작업 완료시 다른 인스턴스와 충돌
  • Redlock: 단일 Redis 장애 대비. 여러 Redis 인스턴스에 과반수 락
  • 락 대신 idempotency가 가능하면 우선 고려 (멱등 키 + Redis SET NX)

6 장애 격리 패턴 (Resilience)

Circuit Breaker

외부 서비스 연속 실패시 호출 차단. "일단 실패 반환"으로 장애 전파 방지. Closed→Open→Half-Open.

Bulkhead

리소스(스레드풀/커넥션) 분리. 한 기능 폭주해도 다른 기능 영향 X.

Retry + Backoff

일시적 실패 재시도. 지수 백오프 + jitter로 thundering herd 방지.

Timeout

모든 외부 호출에 타임아웃. 느린 의존성이 스레드 고갈시키는 것 방지.

Resilience4j 적용

// build.gradle.kts
implementation("io.github.resilience4j:resilience4j-spring-boot3:2.2.0")

// application.yml
resilience4j:
  circuitbreaker:
    instances:
      paymentApi:
        failureRateThreshold: 50        # 50% 실패율 넘으면 Open
        waitDurationInOpenState: 30s
        slidingWindowSize: 10
  retry:
    instances:
      paymentApi:
        maxAttempts: 3
        waitDuration: 500
        exponentialBackoffMultiplier: 2
  timelimiter:
    instances:
      paymentApi:
        timeoutDuration: 2s

@Service
class PaymentService(private val client: PaymentClient) {

    @CircuitBreaker(name = "paymentApi", fallbackMethod = "fallback")
    @Retry(name = "paymentApi")
    @TimeLimiter(name = "paymentApi")
    fun charge(req: ChargeRequest): CompletableFuture<PaymentResult> =
        CompletableFuture.supplyAsync { client.charge(req) }

    fun fallback(req: ChargeRequest, t: Throwable): CompletableFuture<PaymentResult> =
        CompletableFuture.completedFuture(
            PaymentResult.failure("결제 서비스 일시 불가")
        )
}
이 프로젝트의 유사 패턴

gate-futures-bot의 supervisorScope + RateLimiter가 소형 bulkhead + timeout 역할. CoinWorker 에러가 다른 워커에 전파되지 않는 격리 + Gate.io API rate limit 방어. 실무 프로덕션에선 여기에 Circuit Breaker + Retry를 추가합니다.

7 일관성 모델

Strong vs Eventual Consistency

  • Strong Consistency: 쓰기 후 즉시 모든 읽기가 최신 값 반환 (RDBMS).
  • Eventual Consistency: 시간이 지나면 일치 (DynamoDB, DNS, 캐시). Stale read 허용.
  • Read-Your-Writes: 내가 쓴 것만은 즉시 보이게 (세션 고정 or primary read).

선택 기준

영역일관성 요구방식
결제/잔고Strong 필수RDBMS + 트랜잭션
재고Strong (오버셀 방지)RDBMS row lock 또는 Redis atomic
상품 조회Eventual OK (수초 지연 허용)캐시 + TTL
조회수/좋아요Eventual OKRedis INCR + 배치 동기화
피드/추천Eventual OKMaterialized view, 비동기 갱신

지연 최적화 전략

  • CDN 캐시: 정적 파일 + cacheable API 응답(상품, 카테고리) → 엣지 5~30ms
  • Regional 데이터: 사용자와 가까운 region에 read replica
  • Write는 중앙, Read는 로컬: Aurora Global의 cross-region 복제 <100ms
  • Connection pooling + HTTP/2: keep-alive로 TCP 핸드셰이크 절감
  • 비동기 처리: 사용자 응답은 빠르게, 나머지는 SQS로 뒤로

10 실전 설계 연습: 광고 미디에이션

티니어 채용공고의 "자체 광고 미디에이션과 서비스 대시보드의 백엔드" 주제를 가정한 설계 연습.

요구사항 정리

  • 광고 요청 TPS 10,000+
  • 여러 광고 네트워크(AdMob, Facebook 등) 중 최적 선택
  • 응답 시간 p99 < 200ms (유저 경험)
  • 노출/클릭/전환 이벤트 실시간 집계
  • 설정(광고 우선순위, 지역별 노출 룰) 핫 리로드
  • 글로벌 배포

아키텍처 개략

Mobile App
    ↓ HTTPS (region별 latency routing)
CloudFront / Regional ALB
    ↓
Ad Mediation API (ECS Fargate, auto-scale)
    ├─ Redis: 광고 설정 캐시 (1s TTL + pub/sub invalidate)
    ├─ Redis: 사용자 최근 노출 기록 (Rate cap)
    ↓
광고 네트워크 순위 결정 (비즈니스 로직)
    ├─ Waterfall: 1순위 네트워크 호출 → 실패시 2순위
    └─ Bidding: 여러 네트워크에 병렬 호출 → 최고가 선택
    ↓
이벤트 발행 (Kafka or SQS)
    ├─ 실시간 집계 (Kinesis Analytics, ClickHouse)
    └─ 배치 집계 (EMR, Athena)
    ↓
대시보드 API (Spring Boot)
    ├─ ClickHouse: OLAP 쿼리
    └─ PostgreSQL: 광고주 설정, 캠페인

구현 포인트

  • Non-blocking: 여러 광고 네트워크 병렬 호출은 WebFlux 또는 코루틴 async + awaitAll
  • Circuit Breaker: 한 네트워크가 느려지면 제외, 주기적으로 재시도
  • Timeout: 각 네트워크 호출에 150ms timeout (전체 p99 200ms 위해)
  • Rate Cap: Redis INCR + EXPIRE로 사용자당 "분당 광고 노출 횟수" 제한
  • 설정 핫 리로드: Spring Cloud Config + Redis pub/sub으로 무배포 설정 변경
  • 이벤트 중복: 클라이언트 생성 UUID + Redis SET NX로 멱등 처리
  • A/B 테스트: Feature flag(LaunchDarkly) 또는 설정 기반

11 예상 면접 질문

Q. 트래픽이 10배 증가했을 때 어떻게 대응하시겠어요?
A. 단계적 접근: ①측정: CloudWatch/Prometheus로 병목 식별(CPU? DB? 외부 API?). ②Auto scaling: 이미 설정된 ECS/K8s HPA로 수평 확장. 부족하면 인스턴스 타입 up. ③DB 읽기 분산: Read replica 추가, 애플리케이션 레벨 라우팅(@Transactional(readOnly = true)). ④캐싱 추가: 핫 데이터를 Redis로, 정적 응답은 CDN. ⑤비동기화: 동기 호출 중 후처리 가능한 것을 큐로 분리(이메일, 통계, 알림). ⑥CQRS/분리: 읽기 모델을 별도 DB/캐시로(오래된 해결). ⑦샤딩: 그래도 안되면 DB 샤딩(최후의 수단, 복잡).
Q. 분산 트랜잭션이 필요한 상황을 설명하고 해결책을 제시해보세요.
A. 예: "주문 생성 + 재고 차감 + 결제"를 여러 서비스가 담당할 때. 2가지 접근: ①2PC (Two-Phase Commit): 강한 일관성, but 성능 낮고 coordinator 장애 취약. 실무에선 거의 안 씀. ②Saga 패턴: 각 로컬 트랜잭션을 순차 실행, 실패시 보상 트랜잭션(취소) 연쇄. Choreography(이벤트 기반) 또는 Orchestration(중앙 조정자). 실무 실용 솔루션: ①Outbox 패턴으로 DB + 이벤트 원자성 보장. ②멱등성 + 재시도로 부분 실패 복구. ③가능하면 eventual consistency 수용(UX 허용 범위 내). 완벽한 일관성 대신 "복구 가능한 불일치"를 설계.
Q. 캐시와 DB 사이 일관성을 어떻게 유지하시나요?
A. 4가지 전략 조합: ①TTL: 짧은 TTL(10~60s)이면 stale이 금방 해소. 단순, 기본. ②Write-through: 쓰기 시 DB + 캐시 동시 갱신. 쓰기 비용 2배. ③Cache-aside + evict: 쓰기 후 캐시 삭제. 다음 read가 재로드. 가장 흔함. ④이벤트 기반 invalidate: DB 변경 이벤트를 Kafka/Redis pub/sub으로 발행, 각 인스턴스가 로컬 캐시 갱신. 주의: "DB 커밋 전 캐시 갱신 → 롤백되면 불일치". Transaction synchronization으로 AFTER_COMMIT 단계에서 캐시 갱신하거나 Outbox 사용. 완벽 일관성은 어려우므로 "허용 가능 stale 시간"을 비즈니스와 합의.
Q. MSA에서 서비스 간 통신 방식을 어떻게 선택하시나요?
A. 2대 축: 동기 vs 비동기, REST vs gRPC vs 이벤트. 동기 REST: 단순, 디버깅 쉬움. 강한 커플링, 장애 전파. 동기 gRPC: 성능 좋음(binary), 타입 안전(proto), HTTP/2 스트리밍. 사내 서비스 간. 비동기 이벤트(Kafka): 느슨한 커플링, 확장 쉬움. 즉시 응답 불가, 복잡도 증가. 실전 기준: "쓰기 후 결과가 즉시 필요한가". 예: 로그인 검증은 동기(gRPC), 주문 생성 후 추천 업데이트는 비동기(Kafka). 읽기는 대개 REST/GraphQL, 쓰기는 경우에 따라 혼합. 이벤트 주도 아키텍처가 확장성엔 유리하지만 디버깅/추적 복잡성이 큰 비용.
Q. 글로벌 서비스 구축시 주요 고려사항은?
A. ①지연: CDN + 지역별 배포(ap-northeast-2, us-west-2, eu-west-1). Route53 latency routing. ②데이터 일관성: Aurora Global(cross-region 복제) 또는 region별 분리(GDPR 등 규제). ③i18n/l10n: 언어별 리소스, 시간대(UTC 저장), 통화(ISO 4217). ④규제: GDPR(EU), CCPA(CA), 개인정보법(한국). 데이터 지역화. ⑤결제: 현지 결제 수단 통합 (Stripe, Adyen, 카카오페이 등). ⑥장애 대응: region failover 계획 + 훈련(Chaos Engineering). ⑦CDN 캐시 전략: user-agent/locale 기반 variation, Vary header. ⑧비용: data transfer cost가 region 간 비쌈. 설계시 주의.
Q. Circuit Breaker는 언제 쓰고 어떤 상태가 있나요?
A. 외부 서비스(결제/이메일/추천 API 등) 연속 실패시 호출을 잠시 차단하여 장애 전파와 스레드 고갈을 방지. 3가지 상태: ①Closed(정상): 호출 통과. 실패율 집계. ②Open(차단): 실패율 임계값(예: 50%) 초과시 전환. 즉시 실패 반환 또는 fallback 호출. ③Half-Open(시험): 일정 시간(예: 30초) 후 소수 호출 허용. 성공하면 Closed, 실패하면 Open 복귀. Resilience4j/Hystrix로 구현. Retry와 함께 쓰면 주의 — retry가 circuit breaker 통계를 오염시킬 수 있어 순서와 scope 잘 설정.
핵심 요약
  • Scale-Out 우선 + Stateless 설계 (세션/파일을 외부 스토리지로)
  • CAP: 네트워크 분할 대비 CP(일관성) vs AP(가용성) 선택
  • 계층형 캐싱: CDN → Gateway → Local(Caffeine) → Redis → DB
  • 이벤트 기반으로 동기 호출 분리, Outbox로 원자성, Saga로 분산 트랜잭션
  • 장애 격리: Circuit Breaker + Bulkhead + Retry + Timeout
  • MSA는 성숙도 있을 때. Modular Monolith → 쪼개기
  • 글로벌: CDN + region별 배포 + Aurora Global + DynamoDB Global Tables
