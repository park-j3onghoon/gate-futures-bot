1 JDK란? JDK vs JRE 차이

Python을 실행하려면 Python 인터프리터가 필요하듯, Kotlin/Java를 실행하려면 JDK가 필요합니다.

Python 세계 Java/Kotlin 세계
실행 환경 Python 인터프리터 JRE (JVM + 표준 라이브러리)
개발 환경 Python + pip + venv JDK (JRE + 컴파일러 + 개발 도구)

핵심 요약

JRE = "실행만" 가능 (서버에 배포할 때).
JDK = "개발 + 실행" 가능 (개발할 때 필요). 우리는 JDK를 설치합니다.

2 왜 JDK 21인가 (최신이 있는데 왜?)

Java는 6개월마다 새 버전이 나옵니다. 하지만 모든 버전이 같지 않습니다.

timeline title JDK 버전과 LTS section LTS 버전 (장기 지원) JDK 8 (2014) : 8년+ 지원 JDK 11 (2018) : 8년+ 지원 JDK 17 (2021) : 8년+ 지원 JDK 21 (2023) : 8년+ 지원, 우리가 사용 section 일반 버전 (단기 지원) JDK 22 (2024) : 6개월 지원 JDK 23 (2024) : 6개월 지원 JDK 24 (2025) : 6개월 지원 JDK 25 (2025) : 6개월 지원
LTS 버전 (Long-Term Support) 일반 버전
지원 기간 최소 8년 6개월 (다음 버전 나오면 끝)
보안 패치 오래 제공됨 6개월 후 없음
프로덕션 사용 적합 실험/테스트 용도
라이브러리 호환성 대부분 지원 일부 미지원 가능
Tip

Python으로 비유하면: Python 3.12(안정) vs Python 3.14(베타) 중 프로덕션에 뭘 쓸지의 문제입니다. JDK 21은 가장 최신 LTS이므로, 안정성과 최신 기능을 모두 갖췄습니다. Spring Boot 3.5도 JDK 21을 공식 지원합니다.

3 brew install 명령어 의미 분해

brew install --cask temurin@21
부분 의미
brew macOS 패키지 매니저 (Homebrew)
install 패키지를 설치
--cask GUI 앱이나 대형 바이너리를 설치하는 모드. JDK는 단순 CLI 도구가 아니라 시스템에 등록되는 런타임이라 cask 사용
temurin Eclipse Adoptium이 배포하는 무료 오픈소스 JDK. Oracle JDK의 무료 대안
@21 JDK 21 버전 지정
Tip: JDK 배포판이 여러 개인 이유

Python은 공식 배포판 하나지만, Java는 여러 회사가 배포합니다 (Oracle, Temurin, Amazon Corretto, Azul 등). 모두 같은 소스 코드(OpenJDK)를 빌드한 것이라 기능은 동일합니다. Temurin은 커뮤니티가 무료로 배포하는 가장 대중적인 선택입니다.

설치 후 확인:

# Python 버전 확인하듯이
java -version
# 출력 예: openjdk version "21.0.x" ...

# JAVA_HOME 확인 (Python의 sys.executable 같은 것)
/usr/libexec/java_home -V

4 Gradle Wrapper (gradlew) 개념

Python 프로젝트에서 팀원마다 다른 pip 버전을 쓰면 문제가 생기듯, Gradle도 버전 차이 문제가 있습니다. Gradle Wrapper는 이 문제를 해결합니다.

Gradle Wrapper 파일들

gate-futures-bot/ ├── gradlew ← Unix/Mac용 실행 스크립트 (bash) ├── gradlew.bat ← Windows용 실행 스크립트 └── gradle/ └── wrapper/ ├── gradle-wrapper.jar ← Gradle 다운로더 (작은 Java 프로그램) └── gradle-wrapper.properties ← 사용할 Gradle 버전 지정

gradle-wrapper.properties 내용:

# 이 한 줄이 핵심! Gradle 8.14.4 버전을 사용하겠다는 선언
distributionUrl=https\://services.gradle.org/distributions/gradle-8.14.4-bin.zip
핵심 요약

Gradle Wrapper = 프로젝트에 Gradle 버전을 고정하는 장치.
./gradlew를 실행하면 지정된 버전의 Gradle을 자동 다운로드 & 실행합니다.
시스템에 Gradle을 따로 설치할 필요가 없습니다. Python의 pipx가 도구 버전을 격리하는 것과 비슷합니다.

5 Spring Initializr란

Django에서 django-admin startproject myproject로 프로젝트를 생성하듯, Spring Boot는 Spring Initializr라는 웹 도구로 프로젝트를 생성합니다.

Python / Django
# 프로젝트 생성
django-admin startproject myproject

# 구조:
# myproject/
#   manage.py
#   myproject/
#     settings.py
#     urls.py
#     wsgi.py
Kotlin / Spring Boot
# start.spring.io 에서 설정 선택 후 다운로드
# 또는 CLI로:
# curl https://start.spring.io/starter.zip \
#   -d language=kotlin \
#   -d type=gradle-project-kotlin \
#   -o project.zip

# 구조: (아래 섹션에서 자세히)

Spring Initializr에서 선택한 것들

설정 항목 선택한 값 의미

Language Kotlin 애플리케이션 개발 언어
Spring Boot 3.5.3 프레임워크 버전

Tip: Group ID란

com.parkj3onghoon처럼 도메인을 뒤집은 형태입니다. Java/Kotlin 세계에서는 패키지 충돌을 방지하기 위해 이런 컨벤션을 사용합니다. Python의 pip install mypackage와 달리, Java에서는 group:artifact:version으로 라이브러리를 식별합니다.

Python 세계에서 비슷한 과정
# 의존성 설치
pip install -r requirements.txt

# 테스트 실행
pytest

# 패키징 (선택)
python setup.py bdist_wheel

# → 이 3개를 한 번에 하는 게
#   ./gradlew build
Gradle이 한 번에 하는 것
# 이것 하나로 전부!
./gradlew build

# 결과:
# build/
#   classes/    ← 컴파일된 클래스 파일
#   libs/       ← JAR 파일 (배포용)
#   reports/    ← 테스트 리포트
#   resources/  ← 복사된 리소스
주의: 첫 실행이 느린 이유

첫 빌드 시 Gradle 자체와 모든 의존성을 인터넷에서 다운로드합니다. ~/.gradle/caches/에 캐시되므로 두 번째부터는 훨씬 빠릅니다. Python에서 pip install이 처음에 오래 걸리는 것과 같습니다.

7 프로젝트 디렉토리 구조

모든 파일/폴더가 무엇인지 하나씩 설명합니다:

핵심 요약
  • src/main/kotlin/ = 프로덕션 코드가 들어가는 곳
  • src/main/resources/ = 설정 파일 (application.yml)
  • src/test/kotlin/ = 테스트 코드
  • build.gradle.kts = 모든 빌드 설정의 중심
  • gradlew = Gradle 실행 명령어 (시스템 설치 불필요)
  • build/ = 빌드 산출물 (자동 생성, 커밋하지 않음)
Tip: 왜 패키지 경로가 이렇게 깊은가

src/main/kotlin/com/parkj3onghoon/gatefuturesbot/ — 이 깊은 경로는 Java/Kotlin 세계의 관례입니다. 패키지명이 디렉토리 구조와 일치해야 합니다. Python에서 from mypackage.module import Class를 쓸 때 실제 디렉토리 구조가 mypackage/module.py인 것과 같은 원리이지만, 도메인 역순 네이밍까지 적용됩니다.
