2 Docker 이미지 만들기

기본 Dockerfile (multi-stage)

# 1단계: 빌드
FROM eclipse-temurin:21-jdk AS builder
WORKDIR /build

# 의존성만 먼저 복사해서 캐시 레이어 최적화
COPY gradle gradle
COPY gradlew build.gradle.kts settings.gradle.kts ./
RUN ./gradlew dependencies --no-daemon

# 소스 복사 후 빌드
COPY src src
RUN ./gradlew bootJar --no-daemon -x test

# 2단계: 실행 (이미지 슬림)
FROM eclipse-temurin:21-jre-alpine
WORKDIR /app

# non-root user
RUN addgroup -S app && adduser -S app -G app
USER app

COPY --from=builder /build/build/libs/*.jar app.jar

EXPOSE 8080

ENTRYPOINT ["java", \
    "-XX:MaxRAMPercentage=75.0", \
    "-XX:+UseG1GC", \
    "-Dspring.profiles.active=${SPRING_PROFILE:prod}", \
    "-jar", "app.jar"]
최적화 포인트
  • Multi-stage: 빌드 이미지와 런타임 이미지 분리 (런타임엔 JRE만, JDK 불필요)
  • 레이어 캐싱: 의존성 변화 없으면 ./gradlew dependencies 단계 재사용
  • Alpine: 이미지 크기 100MB 이하 (기본 ubuntu는 400MB+)
  • non-root: 보안 (컨테이너 탈출시 영향 최소화)
  • JVM 튜닝: MaxRAMPercentage로 컨테이너 메모리 제한에 맞춤

Spring Boot의 더 좋은 방법: Buildpack

# Dockerfile 없이 Spring Boot가 자동 생성
./gradlew bootBuildImage --imageName=myapp:1.0.0

# 내부적으로 Cloud Native Buildpacks + 레이어 분리(JIT 최적화, 기동 빠름)
# 대형 서비스는 여전히 커스텀 Dockerfile 선호 (세밀 제어)

3 GitHub Actions

# .github/workflows/ci.yml
name: CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - uses: actions/setup-java@v4
        with:
          distribution: 'temurin'
          java-version: '21'

      - uses: gradle/actions/setup-gradle@v3
        with:
          cache-read-only: ${{ github.ref != 'refs/heads/main' }}

      - name: Run tests
        run: ./gradlew test --no-daemon

      - name: Coverage
        run: ./gradlew jacocoTestReport jacocoTestCoverageVerification

      - name: Upload test results
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: test-results
          path: build/reports/tests/

      - name: SonarQube
        env:
          SONAR_TOKEN: ${{ secrets.SONAR_TOKEN }}
        run: ./gradlew sonar --no-daemon

  build-and-push:
    needs: test
    if: github.ref == 'refs/heads/main'
    runs-on: ubuntu-latest
    permissions:
      id-token: write  # AWS OIDC 용
      contents: read
    steps:
      - uses: actions/checkout@v4

      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions
          aws-region: ap-northeast-2

      - uses: aws-actions/amazon-ecr-login@v2
        id: ecr

      - name: Build and push image
        run: |
          IMAGE="${{ steps.ecr.outputs.registry }}/my-service:${{ github.sha }}"
          docker build -t $IMAGE .
          docker push $IMAGE
          docker tag $IMAGE "${{ steps.ecr.outputs.registry }}/my-service:latest"
          docker push "${{ steps.ecr.outputs.registry }}/my-service:latest"

      - name: Deploy to ECS (dev)
        run: |
          aws ecs update-service \
            --cluster my-cluster \
            --service my-service-dev \
            --force-new-deployment
CI/CD 베스트 프랙티스
  • PR별 테스트 + 메인 브랜치에서만 배포
  • 이미지 태그는 git sha: latest는 재현성 없음
  • AWS OIDC 권장: 정적 access key 대신 페더레이션
  • 캐시 활용: Gradle, Docker layer, 의존성 모두
  • 병렬화: 테스트를 shard로 분산, matrix build
  • Gradle Enterprise 또는 Remote build cache로 팀 단위 가속

4 배포 전략

Rolling Update

인스턴스를 1대씩 순차 교체. 다운타임 없음. ECS/K8s 기본. 롤백 느림(순차 되돌림). 대부분 상황에 적합.

Blue/Green

구/신 환경 동시 유지 후 LB 전환. 즉시 롤백 가능(트래픽 되돌리기만). 비용 2배 일시적, 상태 공유(DB/세션) 주의.

Canary

소수 트래픽(5%)을 신버전에 보내 검증 후 점진 확대. 메트릭 기반 자동 promote/rollback. 실험적 변경에 안전.

Recreate (다운타임)

전체 종료 후 새 버전 시작. 다운타임 발생. 배치 작업, 내부 도구 등 사용자 없는 시간에만.

Blue/Green의 함정: DB 스키마

-- ❌ 위험한 스키마 변경 (Blue와 Green이 동시에 DB를 봐야 함)
ALTER TABLE orders DROP COLUMN old_field;
-- Blue가 아직 old_field를 참조하면 깨짐

-- ✅ Expand/Contract 패턴 (2단계 배포)
-- 1단계: 컬럼 추가만 (Blue와 Green 둘 다 호환)
ALTER TABLE orders ADD COLUMN new_field VARCHAR(100);
-- 앱 배포: old_field 읽고 new_field 기록

-- 2단계: old_field 참조 제거한 앱 배포 + 나중에 컬럼 삭제
ALTER TABLE orders DROP COLUMN old_field;

5 AWS Compute

EC2

가상 서버. 유연하지만 OS/패치/스케일링 직접 관리. Auto Scaling Group + Load Balancer 조합.

ECS (Fargate)

컨테이너 오케스트레이터. Fargate는 서버리스(서버 관리 없음). 소/중 규모 팀에 최적.

EKS

Kubernetes 관리형. 표준 k8s API. 복잡하지만 유연. 대규모/MSA에 적합.

Lambda

서버리스 함수. 이벤트 트리거(API Gateway, S3, SQS). Cold start 주의. 간헐적 작업에 최적.

Elastic Beanstalk

"플랫폼 서비스". Heroku 같은 경험. 빠른 시작 but 세밀 제어 어려움.

App Runner

컨테이너 최신 서버리스. Fargate 더 간단 버전. 소규모 서비스 적합.

6 AWS Data

RDS

관리형 RDBMS. PostgreSQL/MySQL/Oracle. Multi-AZ(HA), Read Replica, 자동 백업.

Aurora

Amazon 자체 RDBMS. MySQL/PostgreSQL 호환. 성능 5배+, 빠른 복구. 비용 RDS보다 비쌈.

DynamoDB

서버리스 NoSQL. 수평 확장. 지연 한 자릿수 ms. 복잡한 쿼리 불가.

ElastiCache

관리형 Redis/Memcached. Cluster 모드 + Multi-AZ.

S3

오브젝트 스토리지. 정적 파일, 백업, 로그. presigned URL로 직접 업/다운로드.

SQS / SNS

SQS=큐(메시지 저장), SNS=pub/sub. 비동기 처리/이벤트 분산.

Spring Boot + RDS 설정

# application-prod.yml
spring:
  datasource:
    url: jdbc:postgresql://${DB_HOST}:5432/${DB_NAME}
    username: ${DB_USER}
    password: ${DB_PASSWORD}
    hikari:
      maximum-pool-size: 20        # 인스턴스당 pool size
      minimum-idle: 5
      connection-timeout: 3000
      idle-timeout: 600000         # 10분
      max-lifetime: 1800000        # 30분 (RDS idle timeout 고려)
      leak-detection-threshold: 60000

  jpa:
    hibernate:
      ddl-auto: validate
    properties:
      hibernate.jdbc.batch_size: 50
      hibernate.order_inserts: true
커넥션 풀 크기 계산

공식: pool_size = (core_count × 2) + effective_spindle_count. 대부분은 10~30 사이. 너무 많으면 DB context switch 증가. 전체 서버 인스턴스 수 × pool_size ≤ DB max_connections. 예: 인스턴스 20개 × pool 20 = 400 → RDS max_connections 500 이상 필요.

7 AWS Network

VPC

격리된 가상 네트워크. Subnet(public/private), Route table, NAT, IGW.

ALB / NLB

ALB=L7(HTTP 경로/헤더 라우팅), NLB=L4(TCP, 고성능). ECS/EKS 앞단.

CloudFront

CDN. 전세계 엣지 캐시. 정적 파일 + 동적 요청 proxy.

Route53

DNS. 헬스체크 + failover, 지역별 라우팅.

API Gateway

REST/WebSocket API 관리. rate limit, 인증, 변환.

WAF

Web Application Firewall. SQL injection, XSS 차단.

글로벌 서비스 아키텍처 (티니어 맥락)

전 세계 6천만 다운로드 규모를 고려한 배포:

Client (전세계)
    ↓
Route53 (latency routing: 한국→서울, US→오레곤)
    ↓
CloudFront (CDN: 정적 파일 + 캐시 가능한 API 응답)
    ↓
ALB (region별)
    ↓
ECS Fargate 서비스 (region별)
    ↓
RDS (Global Database - 읽기 전용 복제본을 각 region에)
ElastiCache (region별)
S3 (cross-region replication)

9 시크릿 관리

절대 코드/yml에 시크릿을 커밋하지 마세요. 3가지 방법:

9.1 AWS Secrets Manager

spring:

  "secrets": [
    {"name": "DB_PASSWORD", "valueFrom": "arn:aws:secretsmanager:ap-northeast-2:xxx:secret:db-password"}

}
# → 컨테이너 시작시 Secrets Manager에서 값 조회, 환경변수로 주입

9.2 AWS Parameter Store (SSM)

# spring-cloud-starter-aws-parameter-store-config 또는 수동
# /myservice/prod/db-password에서 자동 로드

9.3 Kubernetes Secrets

apiVersion: v1
kind: Secret
metadata:
  name: db-credentials
type: Opaque
stringData:
  password: "xxx"

# Deployment에서

  - name: DB_PASSWORD

        name: db-credentials

시크릿 운영 원칙
  • 최소 권한: 각 서비스는 자신의 시크릿만 접근
  • 자동 rotation: DB 비밀번호는 주기적으로 교체 (Secrets Manager 자동)
  • 감사 로그: CloudTrail로 시크릿 접근 기록
  • Git secret scanning: 실수로 커밋된 시크릿 탐지 (GitHub Advanced Security)

10 IaC (Infrastructure as Code)

인프라를 코드로 관리. 재현성, 리뷰 가능, 롤백 용이.

# Terraform 예시
resource "aws_ecs_service" "api" {
  name            = "my-service"
  cluster         = aws_ecs_cluster.main.id
  task_definition = aws_ecs_task_definition.api.arn
  desired_count   = 3

  deployment_configuration {
    minimum_healthy_percent = 100
    maximum_percent         = 200
  }

  network_configuration {
    subnets          = module.vpc.private_subnets
    security_groups  = [aws_security_group.api.id]
    assign_public_ip = false
  }

  load_balancer {
    target_group_arn = aws_lb_target_group.api.arn
    container_name   = "app"
    container_port   = 8080
  }
}
Terraform vs AWS CDK

Terraform: HCL(선언적). AWS 외 모든 클라우드 지원. 큰 생태계. AWS CDK: TypeScript/Python/Kotlin 등 프로그래밍 언어로. AWS 전용. 루프/추상화 강력. 팀이 AWS에만 집중이면 CDK, multi-cloud면 Terraform. Kotlin 백엔드 팀은 CDK Kotlin binding도 선택지.

11 예상 면접 질문

Q. CI/CD 파이프라인을 설계한다면 어떻게 구성하시나요?
A. 단계: ①PR 트리거: lint + unit test + 커버리지 체크. 빠르게 피드백(5분 이내 목표). ②main 머지 트리거: integration test(Testcontainers) + security scan(Trivy) + SonarQube. ③이미지 빌드: multi-stage Dockerfile로 slim 이미지 생성, ECR push. 태그는 git sha. ④dev 자동 배포: 매 머지마다 dev 환경에 배포 + smoke test. ⑤staging/prod는 수동 approval 후 rolling update. 메트릭 기반 자동 rollback. 핵심 원칙: 재현성(동일 커밋 → 동일 산출물), 빠른 피드백, 안전한 롤백.
Q. Blue/Green과 Canary의 차이는?
A. Blue/Green은 구/신 환경을 동시에 유지하고 LB 전환으로 트래픽을 100%로 한번에 바꿉니다. 롤백이 즉각적(전환 되돌리기). 비용은 일시적으로 2배. Canary는 트래픽을 점진적으로 이동(5% → 25% → 100%). 단계별로 메트릭을 보며 자동/수동 promote. 문제 발견시 영향받은 사용자 최소화. 단계 관리 복잡. 실무: risk가 낮은 변경은 Blue/Green(단순), risk 있는 변경/큰 변경은 Canary. K8s에선 Istio/Argo Rollouts로 쉽게 구현.
Q. ECS Fargate vs EKS 선택 기준은?
A. ECS Fargate: AWS 전용, 간단함, 서버리스(노드 관리 없음). 소/중 규모 서비스에 최적. 학습 곡선 낮음. EKS: 표준 Kubernetes API, multi-cloud 이전 가능, 복잡한 오케스트레이션(Service Mesh, Operator). 대규모 MSA에 적합. 학습 곡선 높고 운영 비용 큼. 실무 관용: 팀이 10명 미만이고 AWS 고정이면 Fargate, DevOps 전담이 있고 복잡한 아키텍처면 EKS. 티니어처럼 성장기 회사는 Fargate로 시작 → 필요시 EKS 이전이 일반적.
Q. DB 마이그레이션을 무중단으로 어떻게 하시나요?
A. Expand/Contract 패턴(2단계): Expand 단계: 신규 컬럼/테이블을 추가만. 앱은 기존 + 신규 둘 다 읽고, 쓰기는 신규에도 복제. Blue(구 앱)와 Green(신 앱) 모두 동작 가능. Contract 단계: 앱이 신규만 사용하는 버전 배포 → 안정화 후 기존 컬럼/테이블 삭제. 금지: 하나의 배포로 컬럼 rename/delete. 구 앱이 남아있으면 깨짐. 도구: Flyway/Liquibase로 버전 관리, pt-online-schema-change로 대용량 alter 무중단.
Q. AWS에서 비용 최적화 경험이 있나요?
A. 5가지 관점: ①Right sizing: CloudWatch로 CPU/memory 사용률 확인, 작은 인스턴스로 이전. ②Spot/Savings Plan: Fargate Spot은 30% 할인(워커/배치에 적합). 안정 워크로드는 1년/3년 Savings Plan. ③Auto Scaling: 시간대별 트래픽 패턴에 맞춰 Target Tracking(p95 응답 시간 기반). ④S3 Lifecycle: 오래된 로그/백업을 Glacier로 자동 이전. ⑤RDS: Read Replica로 primary 부하 분산, Aurora Serverless v2로 off-peak 비용 절감. ⑥모니터링: Cost Explorer + Budgets로 이상 탐지.
Q. 장애 대응 프로세스는 어떻게 가져가시나요?
A. ①탐지: SLO 기반 Alert (PagerDuty). ②초기 대응 (5분 내): on-call이 acknowledge, runbook 확인. 심각도 결정. ③완화: 원인 찾기 전에 서비스 복구 우선. 롤백, 트래픽 우회, 기능 플래그 off. ④커뮤니케이션: Status page 업데이트, 내부 Slack incident 채널. ⑤해결: 원인 수정 + 배포. ⑥Post-mortem (blameless): 48시간 내 문서화 — 타임라인, 근본 원인(5 Whys), action item. 사람 비난 X, 시스템 개선 O. ⑦재발 방지: 테스트 추가, 모니터링 강화, 프로세스 개선.
