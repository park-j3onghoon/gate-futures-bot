1 왜 DI인가 (Django엔 없는 이유)

Django는 함수/클래스를 직접 import해서 호출합니다. 의존 관리는 개발자 책임. Spring은 Bean 컨테이너(ApplicationContext)가 객체 생성과 의존 연결을 대신 합니다. 

# Django: 서비스를 직접 인스턴스화
# views.py
from myapp.services import EmailService, UserRepo

def send_welcome(request, user_id):
    repo = UserRepo()
    user = repo.get(user_id)

    email = EmailService(
        api_key=settings.SENDGRID_KEY
    )
    email.send(user.email, "Welcome!")
    return ...

# 문제:
# - EmailService를 테스트할 때 Mock 주입 어려움
# - SENDGRID_KEY를 여기저기서 import
# - 구현체 교체시 모든 호출부 수정
// Spring: Bean 컨테이너가 의존 주입
@RestController
class UserController(
    private val userRepo: UserRepository,
    private val emailService: EmailService
) {
    @GetMapping("/welcome/{id}")
    fun sendWelcome(@PathVariable id: Long) {
        val user = userRepo.findById(id)
        emailService.send(user.email, "Welcome!")
    }
}

// 장점:
// - Mock EmailService를 테스트에 주입 쉬움
// - SENDGRID_KEY는 EmailService 생성시 1회만
// - EmailService 구현 교체해도 Controller 변화 없음

2 IoC: 제어의 역전

Inversion of Control — "객체를 누가 만들고 관리하느냐"의 주체가 개발자 코드에서 프레임워크로 넘어감. DI는 IoC를 구현하는 한 방식.

용어 정리

  • IoC 컨테이너 = Spring의 ApplicationContext. Bean을 생성/보관/주입
  • Bean = IoC 컨테이너가 관리하는 객체
  • DI (Dependency Injection) = 필요한 Bean을 생성자/필드로 자동 주입
  • @Component = "이 클래스를 Bean으로 만들어라"는 표시

3 Bean이란

Bean으로 등록하는 3가지 방법:

방법 1: 스테레오타입 어노테이션 (가장 흔함)

@Component      // 일반 컴포넌트
class RateLimiter

@Service        // 비즈니스 로직
class OrderService

@Repository     // 데이터 접근
class UserRepository

@RestController // 웹 API
class UserController

@Configuration  // 수동 Bean 정의
class AppConfig

// 모두 내부적으로 @Component. 의도를 표현할 뿐

방법 2: @Bean 메서드 (3rd party 라이브러리)

@Configuration
class RedisConfig {
    @Bean
    fun redisTemplate(connectionFactory: RedisConnectionFactory): RedisTemplate<String, String> {
        return RedisTemplate<String, String>().apply {
            setConnectionFactory(connectionFactory)
            keySerializer = StringRedisSerializer()
            valueSerializer = StringRedisSerializer()
        }
    }

    @Bean
    fun objectMapper(): ObjectMapper = jacksonObjectMapper().apply {
        registerModule(JavaTimeModule())
        disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS)
    }
}

방법 3: 자동 구성 (Spring Boot Starter)

# - DataSource (HikariCP)
# - EntityManagerFactory
# - PlatformTransactionManager
# - JpaRepository 구현체

4 주입 방식 3가지

4.1 생성자 주입 (⭐ 권장)

@Service
class OrderService(
    private val repository: OrderRepository,
    private val paymentClient: PaymentClient,
    private val eventPublisher: ApplicationEventPublisher
) {
    fun placeOrder(cmd: OrderCommand) { /* ... */ }
}

// Spring이 찾아서: new OrderService(orderRepo, paymentClient, eventPublisher)
왜 생성자 주입인가 (3가지 이유)
  1. 불변성: val로 선언 가능 → 재할당 방지
  2. 필수 의존성 명시: 빠뜨리면 컴파일 에러. 순환 의존도 Spring 부팅 시 감지
  3. 테스트 용이: OrderService(mockRepo, mockClient, mockPublisher)로 Spring 없이 생성 가능

4.2 필드 주입 (사용 금지 권장)

// ❌ 비권장
@Service
class OrderService {
    @Autowired lateinit var repository: OrderRepository
    @Autowired lateinit var paymentClient: PaymentClient
}

// 문제:
// - val 불가 (반드시 var + lateinit)
// - Spring 없이 인스턴스화 불가 → 테스트 어려움
// - 숨겨진 의존성 (IDE에서 생성자로 파악 불가)
// - final 필드가 아니라 리플렉션으로 주입 → 약간의 오버헤드

4.3 세터 주입 (특수 상황)

@Service
class OrderService {
    private var optionalLogger: AuditLogger? = null

    @Autowired(required = false)
    fun setAuditLogger(logger: AuditLogger) {
        this.optionalLogger = logger
    }
}

// 언제 쓰나: 선택적 의존성, 런타임 교체가 필요한 경우 (드물다)

5 Bean 스코프

Bean이 언제 생성되고 얼마나 공유되는지.

singleton (기본)

컨테이너에 1개만 존재. 모든 주입 지점이 같은 인스턴스. 99%의 Bean이 singleton. 무상태(stateless) 또는 thread-safe해야 함.

prototype

주입할 때마다 새 인스턴스. 상태를 가진 객체, 동시성 충돌 우려가 있을 때. 드묾.

request (web)

HTTP 요청당 1개. 요청 scope 데이터(사용자 컨텍스트 등).

session (web)

HTTP 세션당 1개. 사용자별 상태. 요즘은 JWT로 대체되는 추세.

@Component
@Scope("prototype")
class OrderBuilder { /* ... */ }

@Component
@Scope(value = WebApplicationContext.SCOPE_REQUEST, proxyMode = ScopedProxyMode.TARGET_CLASS)
class RequestContext { /* ... */ }
Singleton에 상태를 넣으면 동시성 버그
// ❌ 위험
@Service
class OrderService {
    private var currentUser: User? = null  // 모든 스레드가 공유!

    fun process(user: User) {
        currentUser = user  // race condition
        // ...
    }
}

// ✅ 파라미터로 전달 또는 thread-local/scoped
@Service
class OrderService {
    fun process(user: User) { /* user를 파라미터로 */ }
}

6 @Configuration과 수동 Bean 등록

3rd party 라이브러리 객체를 Bean으로 등록할 때 사용. 메서드 이름이 Bean 이름.

@Configuration
class HttpClientConfig {

    @Bean
    fun httpClient(): HttpClient = HttpClient.newBuilder()
        .connectTimeout(Duration.ofSeconds(5))
        .build()

    @Bean
    fun objectMapper(): ObjectMapper = jacksonObjectMapper().apply {
        registerModule(JavaTimeModule())
    }

    // 다른 Bean에 의존할 수도 있음
    @Bean
    fun weatherApi(httpClient: HttpClient, objectMapper: ObjectMapper): WeatherApi {
        return WeatherApiImpl(httpClient, objectMapper)
    }
}
프로젝트 사용 예: Gate.io SDK Bean 등록
src/main/kotlin/.../config/GateApiConfig.kt 
@Configuration
class GateApiConfig {
    @Bean
    fun futuresApi(apiProperties: ApiProperties): FuturesApi {
        val apiClient = ApiClient().apply {
            basePath = apiProperties.host
            setApiKeySecret(apiProperties.key, apiProperties.secret)
        }
        return FuturesApi(apiClient)
    }
}

Gate.io SDK의 FuturesApi는 Java 라이브러리라 우리가 @Component 붙일 수 없음. @Bean 메서드로 감싸서 Spring에 등록 + 설정값 주입.

7 동일 타입이 여러 개일 때

같은 인터페이스 구현체가 2개 이상이면 Spring이 혼란스러워합니다. 해결책 4가지:

// 두 개의 EmailSender 구현
interface EmailSender { fun send(to: String, msg: String) }

@Component
class SesEmailSender : EmailSender { /* ... */ }

@Component
class SmtpEmailSender : EmailSender { /* ... */ }

// 주입 시도 → NoUniqueBeanDefinitionException
@Service
class Notifier(private val sender: EmailSender) { /* 에러! */ }

해결 1: @Primary — "기본 후보"

@Component
@Primary
class SesEmailSender : EmailSender { /* ... */ }

@Component
class SmtpEmailSender : EmailSender { /* ... */ }

// 자동으로 SesEmailSender 주입

해결 2: @Qualifier — 이름으로 지정

@Component("ses")
class SesEmailSender : EmailSender { /* ... */ }

@Component("smtp")
class SmtpEmailSender : EmailSender { /* ... */ }

@Service
class Notifier(
    @Qualifier("ses") private val sender: EmailSender,
    @Qualifier("smtp") private val backup: EmailSender
) { /* ... */ }

해결 3: @Profile — 환경별 선택

@Component
@Profile("prod")
class SesEmailSender : EmailSender { /* ... */ }

@Component
@Profile("!prod")  // prod 아닐 때
class ConsoleEmailSender : EmailSender { /* ... */ }

해결 4: List/Map으로 모두 주입

@Service
class MultiChannelNotifier(
    private val senders: List<EmailSender>  // 모든 구현 주입
) {
    fun broadcast(msg: String) {
        senders.forEach { it.send("admin@x.com", msg) }
    }
}

// Map<String, EmailSender> — Bean 이름을 key로

8 조건부 Bean

환경/설정에 따라 Bean 등록 여부를 결정. Spring Boot Auto-configuration이 이것으로 동작.

// 1. 특정 클래스가 classpath에 있을 때만
@Configuration
@ConditionalOnClass(name = ["redis.clients.jedis.Jedis"])
class RedisConfig

// 2. 특정 속성이 있을 때만
@Component
@ConditionalOnProperty(
    prefix = "feature.experimental",
    name = ["enabled"],
    havingValue = "true",
    matchIfMissing = false
)
class ExperimentalFeature

// 3. 특정 Bean이 없을 때만 (기본 구현 제공)
@Bean
@ConditionalOnMissingBean(EmailSender::class)
fun defaultEmailSender(): EmailSender = ConsoleEmailSender()

// 4. 표현식 기반
@Component
@ConditionalOnExpression("'\${app.mode}' == 'PROD'")
class ProductionMonitor

9 순환 의존

A → B → A 형태의 의존. Spring Boot 2.6+부터 기본 에러로 처리 (예전엔 자동 해결).

❌ 순환 의존

@Service
class OrderService(val userService: UserService)

@Service
class UserService(val orderService: OrderService)
// 생성자 주입 + singleton → 서로를 생성 못 함
// BeanCurrentlyInCreationException

✅ 해결: 공통 로직 추출

@Service
class OrderService(
    val userQuery: UserQueryService
)

@Service
class UserService(
    val userQuery: UserQueryService
)

@Service
class UserQueryService(
    val userRepo: UserRepository
)
// 공통 로직을 제3 컴포넌트로

순환 의존 해결 방법

  1. 설계 재검토: 대부분 책임이 잘못 분리된 것. 공통 서비스 추출(위 예시)
  2. ApplicationEventPublisher: A → event → B 방향으로 간접 호출
  3. @Lazy: 한쪽을 lazy 프록시로 감싸 순환을 끊음 (임시 방편)
  4. 세터 주입: 권장하지 않음 (불변성 손실)

@Validated

11 예상 면접 질문

Q. 왜 생성자 주입을 권장하나요?
A. 3가지 이유: ①불변성: val(또는 Java final)로 의존성 고정. 런타임에 바뀌지 않음. ②필수 의존성 명시: 생성자 시그니처로 "무엇이 필요한지" 명확. Spring 없이도 new OrderService(mockRepo, mockClient) 테스트 가능. ③순환 의존 감지: 부팅 시 에러로 드러남. 필드 주입은 런타임까지 감춤. 또한 생성자가 너무 길면 "클래스가 너무 많은 책임을 가졌다"는 신호 — SRP 위반 감지에도 유용.
Q. Singleton Bean에 상태를 가지면 안 되나요?
A. 안 됩니다. Singleton은 여러 스레드가 동시 접근하는 공유 객체라 가변 상태는 race condition 유발. 상태가 필요하면: ① 파라미터로 전달(가장 좋음), ② scope를 prototype/request로 변경, ③ thread-safe 자료구조 사용(AtomicReference, ConcurrentHashMap). Singleton의 프로퍼티는 불변 설정값이거나 자기 자신도 thread-safe한 컴포넌트(repository, client)만.
Q. @Component와 @Bean은 언제 각각 쓰나요?
A. @Component는 내가 작성하는 클래스에 붙여 자동 Bean 등록. @Bean은 @Configuration 내부의 팩토리 메서드로, 3rd party 라이브러리 객체처럼 내가 수정할 수 없는 클래스를 Bean으로 등록할 때 사용. 또 한 가지 객체에 대한 복잡한 초기화 로직(HttpClient에 interceptor 추가 등)이 있을 때도 @Bean 메서드가 적합. 동일 역할이지만 대상과 용도가 다릅니다.
Q. 순환 의존을 만났을 때 어떻게 해결하나요?
A. 일반적으로 설계 오류 신호로 보고 해결합니다. 3가지 접근: ①공통 책임을 제3 컴포넌트로 추출(권장). 예: UserService ↔ OrderService가 서로 필요하면 UserQueryService를 분리해 둘이 이것에만 의존. ②이벤트 기반 분리: ApplicationEventPublisher로 직접 호출을 간접 호출로 바꿈. pub/sub. ③@Lazy: 한쪽을 프록시로 감싸 순환을 끊음 (설계 문제를 숨기는 임시 방편). Spring Boot 2.6+는 기본적으로 순환 허용 안 함(spring.main.allow-circular-references=false).
Q. @Qualifier와 @Primary는 언제 각각 쓰나요?
A. 둘 다 동일 타입 Bean이 여러 개일 때의 선택 전략이지만: @Primary는 "기본 후보"를 지정 — 호출자 쪽에서 아무것도 명시 안 하면 이게 주입. 하나의 "주력"이 명확한 경우에 깔끔. @Qualifier는 호출 지점에서 명시적으로 선택 — 여러 구현을 용도별로 다 쓸 때 적합. 조합도 가능: Primary로 기본을 정하고 일부 지점에서만 Qualifier로 override.
Q. Bean의 생성 시점은 언제인가요?
A. 기본적으로 애플리케이션 시작 시 한 번에 eager 생성됩니다 (singleton + eager init). 이점은 설정 오류나 순환 의존을 부팅 시점에 바로 발견한다는 것. @Lazy를 붙이면 처음 사용할 때 생성(lazy init). 부팅 속도가 중요한 경우(많은 Bean + 개발 환경)에 고려하지만, 프로덕션은 eager가 기본. prototype scope는 주입할 때마다 새로 생성.
핵심 요약
  • DI = Bean 컨테이너가 객체 생성/주입을 대신 → 결합도 낮추고 테스트 용이
  • 주입 방식: 생성자 주입이 정답. 불변 + 명시적 + 테스트 쉬움
  • Bean 스코프: singleton이 기본. 상태 금지, thread-safe 필수
  • @Component(내 코드) vs @Bean(3rd party). 스테레오타입(@Service/@Repository)은 의도 표현
  • 동일 타입 충돌은 @Primary / @Qualifier / @Profile / List 주입으로 해결
  • 조건부 Bean(@ConditionalOn*)은 Spring Boot Auto-configuration의 엔진
  • 순환 의존은 설계 재검토가 1순위
