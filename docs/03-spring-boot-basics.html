1 Spring Boot란

Spring Boot = Django 같은 프레임워크이지만, 범위가 더 넓습니다. Django가 주로 웹 애플리케이션에 특화되어 있다면, Spring Boot는 웹, 배치, 메시징, 스케줄링 등 모든 종류의 애플리케이션을 만들 수 있습니다.

관점 Django Spring Boot
철학 "Batteries included" (다 들어있음) "Convention over Configuration" + "Starter"로 조합
주요 용도 웹 (REST API, 템플릿 렌더링) 웹, 배치, 메시징, CLI 등 범용
ORM Django ORM (내장) JPA / Hibernate (별도 Starter 추가)
설정 파일 settings.py application.yml
의존성 주입 없음 (직접 import) 핵심 기능 (IoC 컨테이너)
프로젝트 구조 app 단위 분리 (urls.py, views.py) 패키지 단위 자유 구성
이 프로젝트에서 - 웹 서버 없이 봇으로 사용 (CLI 앱)
Tip: 왜 웹 서버가 없는데 Spring Boot를 쓰나

이 프로젝트(gate-futures-bot)는 HTTP 요청을 받는 웹 서버가 아닙니다. 그런데도 Spring Boot를 쓰는 이유: DI(의존성 주입), 설정 관리, 로깅, 테스트 지원 등의 인프라가 매우 편리하기 때문입니다. Django도 manage.py로 커맨드를 만들어 웹 서버 없이 쓸 수 있는 것과 비슷합니다.

3 DI (의존성 주입) — Spring의 핵심 개념

Django에는 없는 개념이지만, Spring에서 가장 중요한 개념입니다. 먼저 "왜 필요한지"를 Django 코드로 설명합니다.

DI가 없는 Django 방식

Django: 직접 import해서 사용
# services/order_service.py
from gateio_sdk import GateClient

class OrderService:
    def __init__(self):
        # 직접 생성! 테스트할 때 곤란...
        self.client = GateClient(
            api_key="real_key",
            api_secret="real_secret"
        )

    def place_order(self, symbol, size):
        return self.client.create_order(
            symbol=symbol, size=size
        )

# 사용하는 쪽
from services.order_service import OrderService
service = OrderService()  # 직접 생성
service.place_order("BTC_USDT", 1)
Spring Boot: 주입받아서 사용
// Spring이 객체를 만들어서 주입!
@Service
class OrderService(
    private val client: GateClient  // 주입!
) {
    fun placeOrder(symbol: String, size: Int) {
        client.createOrder(
            symbol = symbol, size = size
        )
    }
}

// 사용하는 쪽도 주입받음
@Component
class TradingBot(
    private val orderService: OrderService
) {
    fun execute() {
        orderService.placeOrder("BTC_USDT", 1)
    }
}

Spring의 DI 어노테이션

어노테이션 의미 Django 비유
@Component "이 클래스를 Spring이 관리해라" (일반) 없음 (모든 클래스를 직접 관리)
@Service "이 클래스는 비즈니스 로직 담당" (@Component의 특화) 서비스 계층 클래스
@Configuration "이 클래스는 설정/Bean 정의를 담당" settings.py
@Bean "이 메서드의 반환값을 Spring이 관리하는 객체로 등록" 없음
@Autowired "이 필드/매개변수에 알맞은 객체를 주입해라" 없음

생성자 주입 (Kotlin 스타일)

Kotlin에서는 생성자 주입이 가장 깔끔하고 권장되는 방식입니다.

// Kotlin에서는 생성자 매개변수 = 자동 주입!
// @Autowired 어노테이션도 필요 없음!

@Service
class OrderService(
    private val gateClient: GateClient,    // 자동 주입
    private val config: BotConfig          // 자동 주입
) {
    fun placeOrder(symbol: String) {
        gateClient.createOrder(
            symbol = symbol,
            size = config.orderSize,
            leverage = config.leverage
        )
    }
}

// Spring이 하는 일:
// 1. GateClient 객체를 찾거나 생성
// 2. BotConfig 객체를 찾거나 생성
// 3. OrderService(gateClient, botConfig) 호출
// 4. 생성된 OrderService를 다른 곳에서도 사용할 수 있게 보관
핵심 요약

DI = "내가 필요한 객체를 직접 만들지 않고, Spring에게 달라고 요청"
장점: 테스트 시 가짜 객체를 쉽게 주입, 객체 간 결합도 낮춤, 라이프사이클 자동 관리
Kotlin에서는 생성자 매개변수에 쓰기만 하면 자동 주입됩니다. 매우 간결합니다.

Tip: ${GATE_API_KEY:}에서 콜론 뒤가 비어있는 이유

${변수명:기본값} 형식에서 : 뒤가 비어있으면 기본값이 빈 문자열입니다. Python의 os.environ.get('GATE_API_KEY', '')와 동일합니다. 환경변수가 설정되지 않았을 때 앱이 크래시하지 않고 빈 값으로 시작합니다.

@ConfigurationProperties: 타입 안전한 설정 바인딩

Django에서는 settings.GATE_API_KEY처럼 문자열 키로 접근합니다. Spring Boot에서는 설정을 Kotlin 클래스에 자동 매핑할 수 있습니다.

Django: 문자열 키로 접근
# settings.py 직접 import

# 오타 나도 런타임까지 모름!

# settings.GATE_API_KEI  ← 오타! 런타임 에러

# 타입 검증 없음
leverage = settings.BOT_LEVERAGE
# leverage가 int인지 str인지 보장 안 됨

@ConfigurationProperties(prefix = "bot")

    val stopLossPct: Double
)

// 사용: 타입 안전!
@Service
class TradingBot(
    private val config: BotConfig  // 주입!
) {
    fun run() {
        // config.leverage는 항상 Int
        // config.contracts는 항상 List<String>
        // 오타 → 컴파일 에러!
    }
}
핵심 요약

@ConfigurationProperties = YAML 설정을 Kotlin data class에 자동 바인딩.
kebab-case(order-size) → camelCase(orderSize) 자동 변환.
오타 → 컴파일 에러. 타입 불일치 → 시작 시 에러. Django의 settings.XXX보다 훨씬 안전합니다.

Django: 직접 관리

client = GateClient()

# 초기화
client.connect()

Spring Boot: 자동 관리
@Service
class GateApiClient(
    private val config: GateConfig
) {
    // 생성 후 자동 호출
    @PostConstruct
    fun init() {
        println("API 클라이언트 초기화!")
    }

    fun placeOrder(symbol: String) { ... }

    // 종료 시 자동 호출
    @PreDestroy
    fun cleanup() {
        println("연결 정리!")
    }
}
// Spring이 생성 → init() → 사용 → cleanup() → 제거
// 모두 자동!
Tip: 왜 Spring이 객체 생명주기를 관리하나

싱글톤 보장: Bean은 기본적으로 하나만 생성됩니다. 여러 곳에서 OrderService를 주입받아도 같은 인스턴스입니다. Python에서 전역 변수나 모듈 패턴으로 싱글톤을 만드는 것을 프레임워크가 자동으로 합니다.

정리 보장: DB 연결, API 세션 등 리소스를 @PreDestroy로 확실히 정리합니다. Python의 atexit 모듈이나 context manager(with)와 비슷하지만 더 체계적입니다.

6 로깅

Django / Python
import logging

logger = logging.getLogger(__name__)

class OrderService:
    def place_order(self, symbol, size):
        logger.info(f"주문: {symbol} x {size}")
        try:
            result = self.client.create_order(...)
            logger.info(f"성공: {result.id}")
            return result
        except Exception as e:
            logger.error(f"실패: {e}", exc_info=True)
            raise
Spring Boot / Kotlin
import org.slf4j.LoggerFactory

@Service
class OrderService(
    private val client: GateClient
) {
    private val log = LoggerFactory
        .getLogger(javaClass)

    fun placeOrder(symbol: String, size: Int) {
        log.info("주문: {} x {}", symbol, size)
        try {
            val result = client.createOrder(...)
            log.info("성공: {}", result.id)
        } catch (e: Exception) {
            log.error("실패: {}", e.message, e)
            throw e
        }
    }
}
Python logging SLF4J + Logback 비고
logging.getLogger(__name__) LoggerFactory.getLogger(javaClass) 로거 생성
logger.debug() log.debug() 디버그 레벨
logger.info() log.info() 정보 레벨
logger.warning() log.warn() 경고 레벨
logger.error() log.error() 에러 레벨
f"값: {value}" "값: {}", value 플레이스홀더 방식 차이
Tip: {} 플레이스홀더 vs f-string

Python은 f-string으로 문자열을 먼저 만들지만, SLF4J는 "주문: {} x {}", symbol, size처럼 로그 레벨이 비활성화되면 문자열을 아예 만들지 않습니다. DEBUG 로그가 수천 개 있어도 INFO 레벨에서는 성능 저하가 없습니다.

Django / pytest
import pytest
from unittest.mock import MagicMock

class TestOrderService:
    def test_place_order_success(self):
        """주문 성공 테스트"""
        # Mock 생성
        mock_client = MagicMock()
        mock_client.create_order.return_value = {

        result = service.place_order("BTC_USDT", 1)

        assert result["id"] == "123"
        mock_client.create_order.assert_called_once()

    def test_place_order_failure(self):
        """주문 실패 시 예외 발생"""

        mock_client.create_order.side_effect = \

        every {
            mockClient.createOrder(any(), any())
        } throws ApiException("잔액 부족")

        val service = OrderService(mockClient)
        assertFailsWith<ApiException> {
            service.placeOrder("BTC_USDT", 1)
        }
    }
}

핵심 요약
  • 단위 테스트: @Test + mockk<T>() — DI 덕분에 Mock 주입이 자연스러움
  • 통합 테스트: @SpringBootTest — 전체 Spring 컨테이너를 로드
  • 실행: ./gradlew test
  • 커버리지: ./gradlew jacocoTestReport
  • Kotlin의 백틱(`` ` ``)을 쓰면 테스트 메서드명을 한글로 작성 가능
Tip: 테스트 구조 권장 사항

테스트 파일은 src/test/kotlin/ 아래에 동일한 패키지 구조로 배치합니다.
src/main/.../OrderService.kt → src/test/.../OrderServiceTest.kt
Django에서 app/tests/test_services.py를 만드는 것과 같은 패턴입니다.
