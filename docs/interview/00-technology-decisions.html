① 언어: Kotlin vs Java vs 대안

백엔드 언어 선택 Kotlin
후보장점단점
Kotlin 간결, null safety, JVM 생태계 100% 활용, 코루틴, data/sealed class Java 대비 컴파일 다소 느림, 일부 라이브러리는 Java-first
Java 가장 큰 생태계, 시니어 인력 풀 최대, 도구 지원 최고 verbose, null 처리 수동(Optional), 레코드 나오기 전 value class 불편
Scala 강력한 타입 시스템, 함수형 패러다임, Spark 학습 곡선 극단적, 컴파일 느림, 인력 수급 어려움
Go 빠른 기동, 작은 바이너리, goroutine 단순 제네릭 뒤늦음, 에러 처리 verbose, 성숙한 DI/ORM 부재
Python (FastAPI) 빠른 개발, ML 통합, 기존 Python 인력 GIL로 진정한 병렬 어려움, 런타임 에러 많음, 대규모 리팩터링 위험
Node.js (TypeScript) 프론트와 동일 언어, 빠른 IO, npm 생태계 단일 스레드 CPU 바운드 약함, 메모리 누수 디버깅 어려움
우리의 상황: 티니어는 글로벌 6천만 MAU 백엔드. 안정성 + 대규모 트래픽 + 장기 유지보수 + 한국 시니어 인력 풀. 광고 미디에이션 등 정확성 중요 서비스.
선택 근거: Kotlin

①JVM의 안정성: 10년+ 검증된 GC, 프로파일러, APM 도구 풀 활용. ②Java 생태계 그대로: Spring, Hibernate, Kafka client 등 모든 라이브러리 사용 가능. ③null safety: 글로벌 서비스에서 NPE 줄이기는 직접적 비용 절감. ④Python 경력과의 갭 최소: 간결한 문법, 함수형, data class 등 Python 개발자 친화적. ⑤한국 채용 시장: 토스, 카카오, 쿠팡, 배민, 당근 등이 Kotlin 중심 → 인력 수급 유리. Java만 쓰던 10년 전이라면 Java지만, 2025년 기준 신규 JVM 백엔드는 Kotlin이 거의 기본.

② 웹 프레임워크: Spring Boot vs 대안

JVM 백엔드 프레임워크 Spring Boot
후보장점단점
Spring Boot 생태계 지배적, Auto-configuration, Starter, 한국 시장 표준, 방대한 자료 기동 시간 10초+, 메모리 무거움, 리플렉션 오버헤드
Quarkus GraalVM native image 지원, 기동 0.1초, 낮은 메모리, Kubernetes-native 생태계 작음, 리플렉션 제약, 국내 레퍼런스 부족
Micronaut 컴파일타임 DI (리플렉션 X), 빠른 기동, Kotlin friendly 생태계 작음, 튜토리얼 부족, Spring 이식 비용
Ktor Kotlin-first, 코루틴 네이티브, 경량, JetBrains 공식 DI 자체 없음(Koin 별도), ORM 없음, 엔터프라이즈 기능 부족
Spring WebFlux 리액티브, 높은 동시성, 백프레셔 러닝커브 높음, 디버깅 어려움, 블로킹 라이브러리와 궁합 나쁨
우리의 상황: 팀 규모 중간, 빠른 기능 개발 필요, 기동 시간은 초당 trade-off 가능, 국내 채용 중심. Lambda로 돌리는 게 아니라 상시 구동되는 ECS/EKS 서비스.
선택 근거: Spring Boot

①생태계가 압도적: Spring Data JPA, Spring Security, Spring Cloud 등 "해결된 문제"가 많음. ②채용/온보딩: 국내 Kotlin/Spring 개발자 풀이 가장 크고 학습 리소스 풍부. ③성숙도: 15년 이상 검증된 프레임워크. 장애 사례/해결책이 문서화됨. ④기동 시간이 문제가 되는 경우(Lambda 등)에만 Quarkus/Micronaut 고려. 대부분 백엔드 서비스는 상시 구동되므로 기동 10초는 한 번 비용.

MVC vs WebFlux Spring MVC + 코루틴

Spring Boot 내부에서도 선택이 있습니다.

후보장점단점
Spring MVC + 코루틴 서블릿 모델(이해 쉬움), 대부분 라이브러리와 호환, Kotlin suspend 지원(Spring 6+) 스레드-per-요청 모델의 한계(수만 동시 접속시)
Spring WebFlux (Reactor) 리액티브, 수만 동시 접속 감당, 백프레셔 자동 Mono/Flux 러닝커브, 블로킹 코드 섞이면 쉽게 깨짐, 스택트레이스 읽기 어려움
선택 근거

수만 동시 접속 + 대부분 IO 대기(채팅, 게이트웨이, 스트리밍) 시나리오에만 WebFlux. 일반 REST API는 Spring MVC + 코루틴이 현실적:
① 팀원 전체가 리액티브 사고를 이해할 필요 없음.
② Kotlin suspend를 Controller에서 직접 지원(Spring 6+).
③ 기존 JPA, JDBC 등 블로킹 라이브러리 그대로 사용.
④ 디버깅/스택트레이스 명확.

③ 빌드 도구: Gradle vs Maven

JVM 빌드 도구 Gradle (Kotlin DSL)
후보장점단점
Gradle (Kotlin DSL) 빠른 incremental build, build cache, Kotlin 타입 안전, 유연한 커스텀 로직 학습 곡선 있음, build script도 "프로그램"이라 복잡해질 수 있음
Gradle (Groovy DSL) 기존 프로젝트 호환, 커뮤니티 예제 많음 IDE 지원 약함(동적 타입), Kotlin 프로젝트에서 부자연스러움
Maven XML로 선언적, IDE 지원 최고, 학습 쉬움, Dependabot 등 도구 지원 XML verbose, 커스텀 로직 어려움, 빌드 속도 Gradle보다 느림
우리의 상황: Kotlin 프로젝트, 중간 규모, 빌드 속도 중요, 커스텀 태스크 가끔 필요.
선택 근거: Gradle Kotlin DSL

①Kotlin DSL: 빌드 스크립트도 Kotlin이라 IDE 자동완성/리팩터링 지원. ②incremental + build cache: 변경된 모듈만 재빌드, 팀 단위 remote cache로 CI 가속. ③Kotlin 프로젝트와 일관성: 소스와 빌드가 같은 언어. ④ Maven은 레거시 유지보수에만. 신규 Kotlin 프로젝트엔 Gradle Kotlin DSL이 표준. ⑤ Gradle 설정 복잡도가 문제되면 Version Catalog (libs.versions.toml)로 중앙 관리.

④ 동시성 모델: 코루틴 vs 대안

JVM 동시성 처리 Kotlin Coroutines
후보장점단점
Kotlin Coroutines 경량(수KB/코루틴), 구조화 동시성, suspend 함수로 순차 코드 모양, Flow 코루틴과 블로킹 API 섞으면 주의 필요, 약간의 런타임 비용
Thread + Executor 가장 단순한 모델, 모든 Java 라이브러리와 호환 스레드당 ~1MB, 수천 개 안됨, Future/CompletableFuture 중첩 지옥
Project Reactor (WebFlux) 리액티브 스트림, 백프레셔, 고동시성 Mono/Flux 체인 익히기 어려움, 블로킹 API 섞기 어려움
Virtual Threads (Java 21+, Project Loom) 기존 스레드 코드 그대로, JVM 네이티브 경량 스레드 아직 초기 단계, synchronized + pinning 이슈, 라이브러리 대응 중
RxJava Java 리액티브 사실상 표준 (구세대) Reactor가 Spring 표준화 이후 밀림
우리의 상황: Kotlin 기반, IO 바운드(HTTP/DB 호출 많음), 팀이 리액티브 사고 경험 없음, 여러 작업 병렬 처리 필요(광고 미디에이션은 여러 네트워크 병렬 호출).
선택 근거: Coroutines

①구조화된 동시성: GlobalScope 금지 → 부모 스코프가 자식 관리 → leak/취소 누락 방지. ②suspend 함수는 "순차 코드처럼 읽힘": Future/Mono 체인보다 디버깅 쉬움. ③Spring 6 통합: Controller에서 suspend 직접 지원. ④경량: 수천 코루틴도 문제 없음 → 글로벌 서비스의 동시 요청에 대응. ⑤ Virtual Threads는 아직 이른 편. Java 21이 LTS지만 생태계 대응이 완료된 2-3년 후 고려.

⑤ ORM/DB 접근

데이터 접근 기술 Spring Data JPA (기본) + JOOQ (복잡 쿼리)
후보장점단점
Spring Data JPA (Hibernate) Repository 자동 생성, 영속성 컨텍스트, 생산성 높음, 표준 N+1 함정, LazyInit 이슈, 복잡 쿼리 표현 제약, 러닝커브
JOOQ 타입 안전 SQL, 복잡 쿼리 가독성, DB 기능 그대로 사용 상용 라이선스(상용 DB), 코드 생성 설정 필요
JDBC Template 가벼움, SQL 그대로, 성능 예측 가능 RowMapper 반복, 관계 매핑 수동
MyBatis XML mapper로 SQL/Java 분리, 레거시 호환, 국내 레거시 많음 XML 관리 번거로움, 컴파일 타임 타입 체크 약함
Exposed (JetBrains) Kotlin-first DSL, 경량 Spring 통합 약함, 문서/커뮤니티 작음, Repository 추상화 없음
R2DBC (리액티브) non-blocking DB, WebFlux와 짝 JPA 기능 없음, 라이브러리 미성숙, 디버깅 어려움
QueryDSL JPA 기반 타입 안전 쿼리, JPA와 공존 APT 설정 복잡, Q-class 생성 관리, Spring Boot 3에서 호환 이슈(버전 선택 주의)
우리의 상황: 대부분 CRUD + 일부 복잡 집계(통계/리포트). 스키마 관계 있음(사용자-주문-결제). 팀이 JPA에 익숙.
선택 근거: JPA (기본) + JOOQ (필요시)

①JPA로 간단한 CRUD 80%를 생산성 있게 처리. JpaRepository 인터페이스 선언만으로 기본 동작. ②QueryDSL로 동적 검색 쿼리(검색 필터 등). ③JOOQ 또는 Native SQL로 복잡 집계(window 함수, CTE 등). JPQL로 표현 불가능한 것들. ④R2DBC는 아직 성숙도 부족. WebFlux 전용일 때만 고려.

대표 안티패턴 피하기: Entity를 API 응답에 직접 노출 금지 → DTO 변환. 모든 관계는 LAZY 기본 → N+1은 fetch join/@EntityGraph로 해결.

⑥ 캐싱

분산 캐시 서버 Redis
후보장점단점
Redis 자료구조 풍부(String/Hash/List/Set/ZSet/Stream), Pub/Sub, Lua, persistence, 커뮤니티 최대 단일 스레드(쓰기), 메모리 제약, 복잡한 쿼리 없음
Memcached 극단적 단순, 멀티 스레드, 약간 더 낮은 지연 String만 지원, Pub/Sub 없음, 복제 없음
Hazelcast JVM 내장 분산 맵, 연산 분산 Redis 대비 생태계 작음, 비용 모델
선택 근거: Redis

단순 KV 캐시라면 Memcached도 충분하지만, 대부분 서비스는 금방 캐시 이상의 기능이 필요해집니다: 세션 저장, 분산 락(SET NX), rate limit(INCR + EXPIRE), 랭킹(ZSet), Pub/Sub, queue(List/Stream). Redis는 "다용도 in-memory 도구"로 자리잡아 사실상 표준. AWS에선 ElastiCache Redis로 관리형 이용.

Redis 클라이언트 Lettuce
후보장점단점
Lettuce Netty 기반 non-blocking, 단일 연결 공유, 리액티브 지원, Spring Boot 기본 블로킹 API 쓸 땐 차이 없음
Jedis 단순하고 안정적, 오래됨 연결 풀 필수(블로킹), 동시성에 풀 크기 제약
Redisson Java 자료구조 추상화(RLock, RMap), 분산 락 쉬움, 엔터프라이즈 기능 추가 학습, 일부 유료 기능
선택 근거: Lettuce (기본) + Redisson (분산 락 필요시)

①Lettuce가 Spring Boot 기본 — 설정 없이 동작. ②분산 락이 필요한 특수 케이스는 Redisson의 RLock이 Raw SET NX 직접 구현보다 안전(자동 lease 갱신, Redlock 지원).

캐시 추상화 Spring Cache (@Cacheable)
후보장점단점
Spring Cache (@Cacheable) 선언적, 캐시 백엔드 교체 용이, 어노테이션으로 끝 프록시 기반 제약(내부 호출), 복잡한 로직 표현 제한
RedisTemplate 직접 세밀 제어, Lua 스크립트, 특수 자료구조 보일러플레이트 많음
선택 근거: 둘 다 필요

①단순 "결과 캐싱"은 @Cacheable — 가독성 좋고 캐시 이름으로 TTL 관리. ②복잡한 로직(증분 카운터, 랭킹, 분산 락, pub/sub)은 RedisTemplate 또는 Redisson 직접 사용. 두 방식 공존.

⑦ 메시지 브로커

메시지/이벤트 시스템 AWS SQS + Kafka (병용)
후보장점단점
Kafka 대용량 스트림, 순서 보장(파티션), 재처리 가능(로그 기반), 연합 consumer group 운영 복잡(ZooKeeper/KRaft), topic 설계 학습, MSK 비용
AWS SQS 완전 관리형, 간단, 무한 확장, DLQ 내장 순서 제한(FIFO 큐의 처리량 낮음), 스트리밍 아님, 메시지 보존 최대 14일
RabbitMQ 강력한 라우팅(exchange), AMQP 표준, 개발 쉬움 처리량 Kafka 대비 낮음, 클러스터 운영 복잡
AWS SNS + SQS (fan-out) SNS topic → 여러 SQS 큐, 관리형 순서 보장 없음
Redis Pub/Sub 간단, 지연 낮음 구독자 오프라인이면 손실(at-most-once)
NATS 매우 낮은 지연, 단순 persistence 약함(JetStream 쓰지 않으면), 생태계 작음
우리의 상황: 서비스 간 비동기 이벤트 + 실시간 광고 노출 로그 스트림. AWS 기반 인프라.
선택 근거: SQS + Kafka 병용

①간단한 작업 큐 → SQS: 이메일 전송, 알림, 썸네일 생성 등. 관리형이라 운영 부담 적음. ②이벤트 스트리밍 → Kafka: 광고 노출/클릭 이벤트(수만 TPS), 실시간 집계(Kinesis Analytics/Flink). ③ 둘 다 필요한 건 다른 문제를 해결하기 때문. SQS는 "작업 처리", Kafka는 "이벤트 로그". ④ 초기엔 SQS만으로 시작, 분석/스트리밍 수요 생기면 Kafka 추가가 실전 패턴.

⑭ 이 프로젝트(gate-futures-bot)의 구체적 선택들

동시성: 별도 프로세스 vs 스레드 vs 코루틴 Kotlin Coroutines
후보장점단점
별도 JVM 프로세스 (코인마다) OS 수준 에러 격리, 메모리 완전 분리 프로세스당 ~150MB, IPC(Redis 등) 필요, 배포 복잡
스레드 + ExecutorService Java 표준, 모든 라이브러리 호환 수천 스레드 불가, Future 체인 어색
Kotlin Coroutines 경량(수KB), supervisorScope로 에러 격리, suspend 함수의 순차 코드 모양 Java 라이브러리의 블로킹 호출시 주의 필요
선택 근거: Coroutines

학습 목적 + 현대 Kotlin 핵심 기술이라는 의도적 선택. 별도 프로세스가 "더 안전"하지만, 단일 JVM에서 코루틴으로도 충분한 격리(supervisorScope) 달성 가능. 메모리 효율 월등하고, 학습 가치 높음.

Rate Limiting 알고리즘 Token Bucket
후보장점단점
Token Bucket burst 허용(capacity 내), 평균 rate 제어, 단순 긴 idle 후 갑작스런 burst 대응 주의
Leaky Bucket 출력 rate 고정(smoothing 효과) burst 완전 차단(사용자 경험 나쁨)
Fixed Window Counter 구현 단순(Redis INCR) window 경계에서 2배 burst 가능
Sliding Window Log 정확한 rate 측정 메모리/CPU 비용 큼
Sliding Window Counter 정확도 + 효율의 절충 구현 약간 복잡
선택 근거: Token Bucket

Gate.io API는 "초당 N개 + 약간의 burst 허용" 정책. Token Bucket이 이 의미와 정확히 일치: capacity = burst 상한, refill rate = 평균 허용 rate. Leaky Bucket은 burst를 막아 앱 시작시 캔들 대량 조회가 느려짐.

Rate Limiter 저장소 In-memory (단일 프로세스)
후보장점단점
In-memory (Kotlin Mutex) 0 지연, 의존성 0, 단일 프로세스에 충분 프로세스 여러 대면 전체 rate 제어 불가
Redis (SET + Lua) 분산 환경, 여러 인스턴스 합산 Redis 의존, 네트워크 왕복(~1ms)
선택 근거 + 확장 대비

단일 프로세스에서 동작하는 개인 봇이라 in-memory가 최적. 하지만 RateLimiter interface를 추출해 Redis 구현으로 교체 용이하게 설계. "지금 필요 없지만 교체 지점을 만들어 둠"이 YAGNI와의 타협점.

ExitCondition 표현: sealed class vs enum + fields sealed class
후보장점단점
sealed class 계층 각 타입 고유 필드(TakeProfitPct는 pct, IndicatorExit는 indicator/period 등), exhaustive when YAML 바인딩이 번거로움(별도 Spec data class 필요)
enum + nullable fields YAML 바인딩 자연스러움 IndicatorExit일 때 pct=null, TakeProfitPct일 때 indicator=null 등 의미 없는 조합 허용 → 런타임 검증 필요
선택 근거: sealed class (도메인), Spec data class (YAML 바인딩)

도메인 계층은 sealed class로 타입 안전성 확보(when이 모든 케이스를 컴파일 타임 검증). 설정(YAML) 계층은 data class + type 필드로 YAML 바인딩 용이. StrategyAssembler가 Spec → sealed class 변환하며 validation 수행. 타입 계층 × 설정 계층 분리가 관용적 해법.

설정 주입 방식 @ConfigurationProperties + @Validated
후보장점단점
@ConfigurationProperties + @Validated 타입 안전, 기본값 data class로 표현, 부팅 시 검증(BindException), IDE 자동완성 class 선언 필요
@Value("${...}") 개별 값 주입 단순 타입 변환 수동, 그룹화 불가, 검증 수동
System.getenv() 환경변수만 Spring 기능 못씀, 하드코딩 느낌
선택 근거

설정은 여러 관련 값이 그룹화되는 게 일반적(bot.contracts, bot.leverage, bot.orderSize...). 타입 안전 + 부팅 시 검증으로 잘못된 설정으로 부팅되지 않음을 보장. @Value는 단일 값 또는 실험적 feature flag 정도에만.

예외 처리: sealed class vs 일반 Exception sealed class 계층
후보장점단점
sealed GateFuturesException 도메인별 예외 타입(Auth/Insufficient/Order/RateLimit/Position/MarketData), catch에서 exhaustive when 가능 초기 설계 비용
단일 RuntimeException + 메시지 단순 catch에서 구분 못함, 로깅/응답 매핑 어려움
Result<T, Error> 패턴 타입으로 실패 표현, try/catch 불필요 Spring AOP/트랜잭션 롤백과 궁합 약함, 모든 호출자가 처리 강제
선택 근거: sealed class

Spring 생태계에선 예외 기반이 표준(@Transactional 롤백, @ControllerAdvice 전역 처리). sealed class로 도메인 예외를 정의하면 계층 명확 + when 안전. 완전 순수 함수형 코드라면 Result 패턴도 좋지만, 실무 Spring 코드엔 sealed Exception이 정답.

  4. 우리 상황 매칭: "우리는 [제약]이 있었기 때문에"
  5. 선택과 근거: "Z를 선택했고 이유는 [구체적 이점]"
  6. 미래 대비: "만약 [변화]가 생기면 [대안]으로 전환 가능"

이 구조로 대답하면 면접관에게 "고민해서 선택했다"는 신호가 확실히 전달됩니다. "Spring 써서요"가 아니라 "Quarkus, Micronaut도 봤지만 팀 역량과 생태계 고려해서 Spring Boot로 갔습니다".
