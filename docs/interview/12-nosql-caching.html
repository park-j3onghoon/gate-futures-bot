1 언제 무엇을 쓰나

채용공고의 "RDBMS 및 NoSQL 데이터 모델링 경험"은 각 저장소의 특성을 이해하고 선택할 수 있는지를 봅니다.

PostgreSQL (RDB)

트랜잭션 + 관계
사용자, 주문, 결제, 정산.
ACID 필수, JOIN 많음, 스키마 엄격.
예: users, orders, payments

Redis (Key-Value)

캐시 + 실시간 카운터 + 큐
세션, 캐시, 랭킹, 분산 락, pub/sub.
인메모리 + TTL + 빠른 R/W.
예: session:xxx, cache:user:42

MongoDB (Document)

유연한 스키마 + 중첩
상품 카탈로그, 로그/이벤트, 사용자 프로필(복잡).
JSON 그대로 저장, 스키마 진화 자유.
예: product 문서에 스펙 중첩

실무 관용: "RDB 기본 + Redis 캐시 + 특수 목적 Mongo"

대부분의 서비스는 PostgreSQL로 시작하고, 캐시가 필요하면 Redis를 추가합니다. MongoDB는 "스키마가 진짜 유연해야 하는 특수 영역"(광고 설정, 사용자별 feature flag 등)에만. 티니어 같은 글로벌 앱도 핵심 데이터는 RDB + Redis 캐시 조합.

2 왜 캐싱인가

DB 쿼리는 평균 수 ms, Redis GET은 1ms 미만. 많이 읽히는 데이터를 Redis에 두면 DB 부하를 10~100배 감소시킬 수 있습니다.

Client
→
API Server
→
Redis (1ms)
miss →
PostgreSQL (20ms)

캐싱이 유리한 데이터

  • 읽기 >> 쓰기: 상품 정보, 카테고리, 설정값
  • TTL 허용: 수 초~수 분 지연 OK인 데이터
  • 재계산 비용 큼: 복잡한 집계, 외부 API 응답
  • 동일 응답 반복: 로그인 사용자 정보, 권한

캐싱 부적합

  • 실시간 정확성 필수 (잔고, 재고 최종 확인)
  • 쓰기가 읽기만큼 많음 (→ 무효화 비용)
  • 사용자별 고유 데이터 + 1회 조회 (캐시 히트율 낮음)

RedisTemplate Bean 설정

@Configuration
class RedisConfig {
    @Bean
    fun redisTemplate(
        connectionFactory: RedisConnectionFactory,
        objectMapper: ObjectMapper
    ): RedisTemplate<String, Any> {
        val template = RedisTemplate<String, Any>()
        template.connectionFactory = connectionFactory
        template.keySerializer = StringRedisSerializer()
        template.hashKeySerializer = StringRedisSerializer()

        // JSON 직렬화 (권장)
        val jsonSerializer = GenericJackson2JsonRedisSerializer(objectMapper)
        template.valueSerializer = jsonSerializer
        template.hashValueSerializer = jsonSerializer
        return template
    }
}

4 RedisTemplate 사용법

@Service
class UserCacheService(
    private val redis: RedisTemplate<String, Any>
) {
    // String 기반 간단 set/get
    fun set(key: String, value: Any, ttl: Duration) {
        redis.opsForValue().set(key, value, ttl)
    }

    @Suppress("UNCHECKED_CAST")
    fun <T : Any> get(key: String, type: Class<T>): T? {
        val raw = redis.opsForValue().get(key) ?: return null
        // Jackson이 JSON → 객체 변환한 상태
        return type.cast(raw)
    }

    // 카운터 (INCR)
    fun increment(key: String): Long =
        redis.opsForValue().increment(key) ?: 0

    // Hash — 객체의 필드 각각 저장
    fun setField(key: String, field: String, value: Any) {
        redis.opsForHash<String, Any>().put(key, field, value)
    }

    // List (큐로 쓰거나 최근 N개)
    fun pushAndTrim(key: String, value: Any, maxSize: Long) {
        redis.opsForList().leftPush(key, value)
        redis.opsForList().trim(key, 0, maxSize - 1)
    }

    // Set (유니크 태그 등)
    fun addTag(key: String, tag: String) {
        redis.opsForSet().add(key, tag)
    }

    // ZSet (랭킹, 시간 정렬)
    fun addToRanking(key: String, member: String, score: Double) {
        redis.opsForZSet().add(key, member, score)
    }

    fun topRanking(key: String, limit: Long): Set<Any> =
        redis.opsForZSet().reverseRange(key, 0, limit - 1) ?: emptySet()

    // Pub/Sub
    fun publish(channel: String, message: Any) {
        redis.convertAndSend(channel, message)
    }

    // 만료 + 삭제
    fun expire(key: String, duration: Duration) = redis.expire(key, duration)
    fun delete(key: String) = redis.delete(key)
}

5 @Cacheable 추상화 (권장)

RedisTemplate을 직접 쓰는 대신 Spring Cache 추상화를 쓰면 코드가 깔끔해집니다.

// build.gradle.kts
implementation("org.springframework.boot:spring-boot-starter-cache")

// Application.kt
@SpringBootApplication
@EnableCaching
class Application

// Cache manager 설정
@Configuration
class CacheConfig {
    @Bean
    fun cacheManager(connectionFactory: RedisConnectionFactory): CacheManager {
        val config = RedisCacheConfiguration.defaultCacheConfig()
            .entryTtl(Duration.ofMinutes(10))
            .serializeKeysWith(SerializationPair.fromSerializer(StringRedisSerializer()))
            .serializeValuesWith(
                SerializationPair.fromSerializer(GenericJackson2JsonRedisSerializer())
            )
        return RedisCacheManager.builder(connectionFactory)
            .cacheDefaults(config)
            .withCacheConfiguration("users",
                config.entryTtl(Duration.ofMinutes(30)))  // 캐시별 TTL
            .withCacheConfiguration("products",
                config.entryTtl(Duration.ofHours(2)))
            .build()
    }
}
@Service
class UserService(private val userRepo: UserRepository) {

    // @Cacheable: 결과를 캐시. 다음 호출시 캐시 있으면 메서드 스킵
    @Cacheable(value = ["users"], key = "#id")
    fun getUser(id: Long): User = userRepo.findById(id).orElseThrow()

    // @CachePut: 메서드는 항상 실행 + 결과를 캐시 갱신
    @CachePut(value = ["users"], key = "#user.id")
    fun update(user: User): User = userRepo.save(user)

    // @CacheEvict: 캐시 삭제
    @CacheEvict(value = ["users"], key = "#id")
    fun delete(id: Long) = userRepo.deleteById(id)

    // 전체 삭제
    @CacheEvict(value = ["users"], allEntries = true)
    fun clearAll() { }

    // 조건부 캐싱
    @Cacheable(
        value = ["users"],
        key = "#id",
        condition = "#id > 0",     // 파라미터 검증
        unless = "#result == null" // 결과 검증
    )
    fun findOrNull(id: Long): User? = userRepo.findById(id).orElse(null)

    // 복합 키
    @Cacheable(value = ["userSearch"], key = "#q + ':' + #page")
    fun search(q: String, page: Int): List<User> = userRepo.search(q, page)
}

6 캐싱 전략 3가지

Cache-Aside (Lazy Loading)

가장 흔한 패턴. @Cacheable이 이 방식.

  1. 1. 캐시 조회
  2. 2. miss면 DB 조회
  3. 3. 결과를 캐시에 저장 후 반환

Write-Through

쓰기시 DB + 캐시 동시 갱신.

  1. 1. Client → 서버
  2. 2. 서버가 DB 저장
  3. 3. 서버가 캐시도 갱신
  4. ✓ 캐시가 항상 최신 (stale 없음)

Write-Behind (Write-Back)

// 쓰기는 캐시에 먼저, DB 반영은 비동기로 배치
// 장점: 쓰기 throughput 높음
// 단점: 장애시 데이터 손실 위험
// 대부분의 서비스엔 over-engineering. 특수 케이스만.

실무 기본: Cache-aside + TTL

@Cacheable(value = ["products"], key = "#id")
fun getProduct(id: Long): Product = productRepo.findById(id).orElseThrow()

@CachePut(value = ["products"], key = "#product.id")
fun updateProduct(product: Product): Product {
    // DB 저장 → 캐시 갱신 (쓰기가 일관성 유지)
    return productRepo.save(product)
}

@CacheEvict(value = ["products"], key = "#id")
fun deleteProduct(id: Long) = productRepo.deleteById(id)

7 캐시 무효화

Phil Karlton: "캐시 무효화와 네이밍은 컴퓨터 과학의 2대 난제". 전략이 필요.

7.1 TTL (Time-To-Live)

// Redis 자체 TTL
redis.opsForValue().set(key, value, Duration.ofMinutes(10))

// @Cacheable은 CacheManager의 기본 TTL 사용
// 가장 단순. 데이터 신선도 vs 부하 trade-off

7.2 명시적 삭제 (Event-based)

@Service
class ProductService(
    private val productRepo: ProductRepository,
    private val cache: CacheManager
) {
    @Transactional
    fun update(id: Long, req: UpdateRequest) {
        productRepo.update(id, req)

        // 관련 캐시 삭제
        cache.getCache("products")?.evict(id)
        cache.getCache("productList")?.clear()  // 리스트 캐시도 날림
    }
}

7.3 버전 키 (Version Key)

// "products:v7:list" — 버전을 키에 포함
// update 시 버전 증가 → 이전 키는 자동으로 외면됨 → TTL로 GC

fun getProducts(): List<Product> {
    val version = redis.opsForValue().get("products:version") ?: "1"
    return cacheOrLoad("products:v$version:list") {
        productRepo.findAll()
    }
}

fun updateAnyProduct() {
    productRepo.save(...)
    redis.opsForValue().increment("products:version")  // 원자적
}

8 캐시 함정

1. Cache Stampede (캐시 쇄도)

인기 키의 TTL 만료 순간 수많은 요청이 동시에 DB로 몰려 과부하. 해결:

// ① Probabilistic Early Expiration — TTL 만료 직전 일부가 미리 갱신
// ② 분산 락 — 한 스레드만 DB 조회, 나머지는 대기
// ③ TTL jitter — 각 키마다 TTL에 랜덤 ±10% 추가

fun getWithLock(key: String): Any = redis.opsForValue().get(key) ?: run {
    val lockKey = "lock:$key"
    if (redis.opsForValue().setIfAbsent(lockKey, "1", Duration.ofSeconds(5)) == true) {
        try {
            val value = loadFromDb()
            redis.opsForValue().set(key, value, Duration.ofMinutes(10))
            value
        } finally {
            redis.delete(lockKey)
        }
    } else {
        Thread.sleep(50)   // 대기 후 재조회
        redis.opsForValue().get(key) ?: loadFromDb()
    }
}
2. Cache Penetration (캐시 관통)

존재하지 않는 키를 반복 조회 → DB까지 매번 도달. 해결:

  • Null 값도 짧은 TTL로 캐싱: @Cacheable(unless = "#result == null")을 빼면 null도 저장됨
  • Bloom filter: "이 키가 존재할 가능성" 사전 판단
3. Cache Avalanche (캐시 눈사태)

같은 시점에 대량 키가 동시 만료 → 한번에 DB로 몰림. 해결:

  • TTL에 jitter 추가: TTL + random(0, 60)초
  • Warm-up: 새 데이터 로드시 미리 캐시 채우기
4. Thundering Herd (@Transactional + @Cacheable 순서)
// ⚠️ 위험: 캐시가 먼저 업데이트되고 트랜잭션 롤백
@Transactional
@CachePut(value = ["users"], key = "#user.id")
fun update(user: User): User {
    val saved = userRepo.save(user)
    if (failCondition) throw RuntimeException()
    // ↑ 롤백되지만 캐시엔 이미 user 저장됨 → 불일치!
    return saved
}

// 해결: TransactionSynchronizationManager로 커밋 후 캐시 갱신

9 Spring Data MongoDB

Redis처럼 Spring Data 패턴으로 Repository를 자동 생성합니다.

// build.gradle.kts
implementation("org.springframework.boot:spring-boot-starter-data-mongodb")

// application.yml
// spring.data.mongodb.uri: mongodb://localhost:27017/myapp

@Document(collection = "products")
data class Product(
    @Id val id: String? = null,
    val name: String,
    val price: BigDecimal,

    @Indexed
    val category: String,

interface ProductRepository : MongoRepository<Product, String> {

    fun findByPriceBetween(min: BigDecimal, max: BigDecimal): List<Product>

    @Query("{ 'tags': { \$in: ?0 } }")
    fun findByAnyTag(tags: List<String>): List<Product>

    // Aggregation (고급)
    @Aggregation(pipeline = [
        "{ \$match: { 'category': ?0 } }",

MongoTemplate (복잡한 쿼리)

@Service
class ProductService(private val mongo: MongoTemplate) {
    fun search(criteria: SearchCriteria): List<Product> {
        val query = Query()
        criteria.name?.let { query.addCriteria(Criteria.where("name").regex(it, "i")) }
        criteria.category?.let { query.addCriteria(Criteria.where("category").`is`(it)) }
        criteria.maxPrice?.let { query.addCriteria(Criteria.where("price").lte(it)) }

        query.with(Sort.by(Sort.Direction.DESC, "createdAt"))
        query.limit(50)

        return mongo.find(query, Product::class.java)
    }
}

10 세션/사용자 상태 관리

Spring Session을 Redis와 결합해 수평 확장이 가능한 세션 스토어 구축.

// build.gradle.kts
implementation("org.springframework.session:spring-session-data-redis")

// application.yml
spring:
  session:
    store-type: redis
    timeout: 30m
    redis:
      namespace: "myapp:session"

이제 HttpSession에 저장한 데이터가 Redis로 자동 저장 → 서버 여러 대 있어도 세션 공유. 로그아웃 / 세션 만료도 Redis TTL로 자동.

JWT vs 세션

세션 (Spring Session + Redis): 서버가 상태 관리, 강제 로그아웃 쉬움, 요청마다 Redis lookup.
JWT: 서버 무상태, 수평 확장 유리, 서명 검증만으로 인증. 하지만 강제 로그아웃 어려움(블랙리스트 Redis 필요).
글로벌 서비스는 대개 JWT + refresh token + Redis 블랙리스트 조합.

11 예상 면접 질문

Q. Redis를 어떻게 사용해보셨나요?
A. 주 용도 5가지: ①캐싱: DB 조회 결과를 @Cacheable + TTL 10분으로 저장. 히트율 모니터링(Micrometer). ②세션 스토어: Spring Session + Redis로 서버 scale-out 시 세션 공유. ③레이트 리밋: INCR + EXPIRE로 "분당 N회" 제한. ④분산 락: SET key NX EX 5로 동시 처리 방지 (Redlock). ⑤실시간 랭킹: Sorted Set(ZSET)으로 top-N 빠른 조회. 모든 경우 키 네이밍 컨벤션({app}:{domain}:{id})과 메모리 사용량 모니터링을 병행.
Q. 캐시 무효화 전략을 설명해주세요.
A. 4가지 혼합 사용: ①TTL: 가장 단순. stale 허용 범위 내 설정(10분~1시간). ②명시적 evict: 쓰기 시점에 @CacheEvict로 해당 키 삭제. ③버전 키: 관련 데이터 그룹을 v1:list → v2:list로 버전 증가. 이전 키는 TTL로 자연 소멸. ④이벤트 기반: DB 변경 이벤트를 pub/sub으로 전달하여 다른 서버의 로컬 캐시도 갱신. 실무는 주로 TTL + 명시적 evict 조합. Stampede 방지는 TTL jitter + 분산 락.
Q. @Cacheable의 주의사항은?
A. 여러 함정: ①프록시 기반이라 같은 클래스 내부 메서드 호출시 캐시 적용 안 됨. ②@Transactional과 순서: 캐시가 먼저 업데이트되고 트랜잭션 롤백되면 불일치. @TransactionalEventListener(AFTER_COMMIT)으로 커밋 후 무효화 권장. ③컬렉션 반환: List<User>를 캐싱하면 전체를 통째로 저장. 각 User가 개별 업데이트되면 무효화 복잡. ④null 처리: 기본적으로 null도 캐싱(Penetration 방지). 원치 않으면 unless = "#result == null". ⑤키 설계: 파라미터 조합이 복잡하면 SpEL이 길어짐 → 커스텀 KeyGenerator 고려.
Q. RDB에 JSON 컬럼이 있는데 MongoDB를 쓸 이유가 있을까요?
A. 경계선이 모호해집니다. PostgreSQL의 JSONB는 인덱싱 + 부분 업데이트를 지원해 많은 경우 Mongo를 대체합니다. MongoDB가 여전히 유리한 경우: ①스키마가 심하게 heterogeneous(상품 A는 필드 10개, 상품 B는 30개). ②단일 문서 내 중첩 배열이 자주 쿼리됨(Mongo의 서브 도큐먼트 인덱스가 유리). ③샤딩: 수평 확장이 Mongo가 더 성숙. 하지만 일반 서비스는 PostgreSQL JSONB로 충분하며, 트랜잭션/조인/타입 안전이 훨씬 뛰어납니다.
Q. Redis의 영속성(persistence)을 설명해주세요.
A. 2가지 방식: ①RDB 스냅샷: 주기적으로 전체 메모리를 파일로. 복구 빠르지만 마지막 스냅샷 이후 데이터 손실 가능. ②AOF (Append-Only File): 모든 쓰기 명령을 로그로 기록. 손실 최소(fsync 설정에 따라) but 파일 크기 크고 재생 느림. 실무는 둘 다 활성(RDB + AOF)하되, Redis는 주요 데이터 저장소가 아니라 캐시/세션 용도로 쓰고, 장애 시엔 DB에서 다시 warm-up하는 설계가 일반적. Multi-AZ 복제(AWS ElastiCache)로 가용성 보강.
핵심 요약
  • RDB(기본) + Redis(캐시/세션/락) + 특수 Mongo(유연 스키마)
  • Spring Data Redis: RedisTemplate(저수준) / @Cacheable(추상화)
  • 캐싱 전략: Cache-aside + TTL이 실무 기본. Write-through는 일관성 강조시
  • 무효화: TTL + 명시적 evict + 버전 키
  • 함정: stampede / penetration / avalanche — jitter, 분산 락, bloom filter로 대응
  • 세션은 Spring Session + Redis로 scale-out 대응. JWT + 블랙리스트 조합도 흔함
