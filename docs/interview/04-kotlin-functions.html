1 함수 선언의 다양한 형태

Kotlin의 함수는 선언 위치에 따라 4가지: top-level, 클래스 멤버, 로컬(중첩), 확장.

# 모듈 레벨 함수 (Python 기본)
def add(a: int, b: int) -> int:
    return a + b

# 클래스 메서드
class Calculator:
    def add(self, a, b): return a + b

# 로컬 함수 (중첩)
def outer():
    def inner(x): return x * 2
    return inner(5)

# Python엔 "확장 메서드" 없음
# (monkey patching은 있지만 위험)
// 1. Top-level 함수 (파일 최상단)
fun add(a: Int, b: Int): Int = a + b

// 2. 클래스 멤버 함수
class Calculator {
    fun add(a: Int, b: Int) = a + b
}

// 3. 로컬 함수 (함수 안의 함수)
fun outer(): Int {
    fun inner(x: Int) = x * 2
    return inner(5)
}

// 4. 확장 함수 (기존 타입에 메서드 추가)
fun String.shout(): String = this.uppercase() + "!"
"hello".shout()  // "HELLO!"

단일 표현식 함수 (single-expression)

본문이 하나의 식이면 중괄호+return 생략:

// full form
fun square(x: Int): Int {
    return x * x
}

// single-expression (리턴 타입도 추론 가능)
fun square(x: Int) = x * x

// 공개 API는 명시 권장 (가독성)
fun square(x: Int): Int = x * x
실무 관용: 공개 함수는 타입 명시

단일 표현식이라도 public 함수는 리턴 타입을 적습니다. 구현이 바뀌어도 시그니처가 안 바뀌도록 고정하는 효과. private 함수는 추론에 맡겨도 무방.

2 파라미터: default · named · vararg

Python과 거의 동일한 풍부한 기능.

# default
def greet(name, lang="ko"):
    return f"{lang}: hi {name}"

# named (keyword args)
greet(name="Teddy", lang="en")

# *args / **kwargs
def log(level, *msgs, **meta):
    print(level, msgs, meta)
log("INFO", "hello", "world", user="teddy")
// default parameters
fun greet(name: String, lang: String = "ko") =
    "$lang: hi $name"

// named arguments
greet(name = "Teddy", lang = "en")
// 파라미터가 많을 때 named가 가독성 UP
createUser(id = 1, name = "T", email = "t@x", admin = false)

// vararg (가변 인자) — Python *args 대응
fun log(level: String, vararg msgs: String) {
    msgs.forEach { println("$level: $it") }
}
log("INFO", "hello", "world")

// spread 연산자 (*): 배열을 vararg로
val items = arrayOf("a", "b")
log("INFO", *items)

3 람다 표현식과 it

Kotlin 람다는 Python의 lambda보다 훨씬 자주 쓰입니다. 1줄 제한도 없고, 고차 함수 인자로 넘기는 관용구의 중심. 

square = lambda x: x * x
square(5)  # 25

# 1줄 제한 때문에 복잡하면 def 필요
add_and_log = lambda a, b: (
    print(f"adding"),
    a + b
)[1]  # 어색

# 그래서 list comprehension / map 등은
# 람다 대신 이해식 선호
result = [x * x for x in range(10)]
val square = { x: Int -> x * x }
square(5)  // 25

// 타입 명시 (파라미터 타입 추론이 어려울 때)
val adder: (Int, Int) -> Int = { a, b -> a + b }

// 여러 줄 OK — 마지막 식이 반환값
val process = { x: Int ->
    println("처리중: $x")
    val doubled = x * 2
    doubled + 1  // ← 반환값
}

// 파라미터 1개면 it으로 생략 가능
listOf(1, 2, 3).map { it * it }  // [1, 4, 9]

// 사용 안 하는 파라미터는 _
map.forEach { (_, value) -> println(value) }

람다의 타입 표기

Kotlin 람다는 일급 객체, 타입으로도 표현됩니다:

// 함수 타입: (파라미터 타입들) -> 리턴 타입
val noArgs: () -> Unit = { println("hi") }
val oneArg: (Int) -> String = { "num: $it" }
val twoArgs: (Int, Int) -> Int = { a, b -> a + b }

// 리시버 있는 함수 타입 (A.() -> B)
val appender: StringBuilder.() -> Unit = { append("world") }

// nullable 함수 타입
val maybeFn: ((Int) -> Int)? = null

4 고차 함수 (Higher-Order Function)

함수를 파라미터로 받거나, 리턴하는 함수. Kotlin 표준 라이브러리의 핵심 도구.

// 함수를 파라미터로
fun <T> retry(times: Int, action: () -> T): T {
    var lastException: Throwable? = null
    repeat(times) {
        try {
            return action()
        } catch (e: Exception) {
            lastException = e
        }
    }
    throw lastException!!
}

// 사용
val result = retry(3) {
    httpClient.get("https://api.example.com")
}

// 함수를 리턴
fun multiplier(factor: Int): (Int) -> Int = { it * factor }

val double = multiplier(2)
val triple = multiplier(3)
double(5)  // 10
triple(5)  // 15
프로젝트 사용 예: 고차 함수로 공통 처리 추출
src/main/kotlin/.../client/GateClient.kt 
// 모든 Gate.io API 호출의 공통: GateApiException → 도메인 예외 매핑 + retry
private inline fun <T> callApi(context: String, action: () -> T): T = withRetry(context) {
    try {
        action()
    } catch (e: GateApiException) {
        throw mapGateException(e, context)
    } catch (e: ApiException) {
        throw wrapApiException(e, context)
    }
}

// 호출부: 각 메서드가 자기 본질 로직만 action 안에 전달
fun createOrder(...): OrderResult = callApi("createOrder(...)") {
    val order = FuturesOrder().apply { /* ... */ }
    toOrderResult(futuresApi.createFuturesOrder(apiProperties.settle, order, null))
}

Python에선 데코레이터로 해결했을 패턴. Kotlin은 고차 함수 + 후행 람다로 동일한 "cross-cutting concern"을 처리합니다.

5 후행 람다 (Trailing Lambda)

람다가 함수의 마지막 파라미터라면 괄호 밖으로 뺄 수 있습니다. Kotlin 코드가 "블록처럼" 읽히는 핵심 이유. 

// 표준 호출
list.filter({ it > 10 })
list.map({ it * 2 })

// 후행 람다: 람다를 () 밖으로
list.filter { it > 10 }
list.map { it * 2 }

// 람다 외 다른 파라미터가 있어도 동일
fun <T> List<T>.firstOrElse(default: T, predicate: (T) -> Boolean): T =
    firstOrNull(predicate) ?: default

// 호출
val x = list.firstOrElse(0) { it > 100 }
//                    ^^^^   ^^^^^^^^^^^^^
//                    default  predicate

DSL 같은 API 설계

후행 람다 + 리시버 함수 타입으로 "빌더 DSL"을 만들 수 있습니다:

// Kotlin stdlib의 buildString
val s = buildString {
    // this는 StringBuilder
    append("Hello, ")
    append("world")
    appendLine()
    append("!")
}

// HTML DSL 예시 (kotlinx.html)
val html = html {
    head { title("My Page") }
    body {
        h1 { +"Hello" }
        p { +"Welcome" }
    }
}

// Spring의 routerFunction DSL
router {
    GET("/users/{id}") { req -> /* ... */ }
    POST("/users") { req -> /* ... */ }
}

6 확장 함수 (⭐ Kotlin의 킬러 기능)

기존 클래스의 소스를 건드리지 않고 메서드를 추가. Python monkey-patching과 비슷하지만 컴파일 타임에 정적 디스패치되어 안전. 

# Python: monkey-patching (위험)
def shout(self):
    return self.upper() + "!"

str.shout = shout  # 전역 영향, 런타임 패치
"hello".shout()     # "HELLO!"

# 대개는 유틸 함수로 작성
def shout(s: str) -> str:
    return s.upper() + "!"

shout("hello")
// Kotlin: 확장 함수 (안전)
fun String.shout(): String = this.uppercase() + "!"

"hello".shout()  // "HELLO!"

// this는 수신자(receiver) = 이 문자열
// 타입 외부에서 .메서드()처럼 호출 가능

// 확장 함수는 import로 scope 제어
// import com.mylib.shout
// → 이 파일에서만 유효

확장 함수가 유용한 상황

  1. 써드파티 타입에 유틸 추가 (Java 타입, stdlib 등 수정 불가한 대상)
  2. 유틸 클래스 탈피: StringUtils.isBlank(s) 대신 s.isBlank()
  3. 도메인 친화적 DSL: 3.days.ago, 100.USD 같은 표현
  4. "조건부" 메서드: 특정 타입일 때만 호출 가능한 메서드
// 1. 써드파티 Java 타입에 추가
fun LocalDate.isWeekend(): Boolean =
    dayOfWeek == DayOfWeek.SATURDAY || dayOfWeek == DayOfWeek.SUNDAY

LocalDate.now().isWeekend()

// 2. List에 유틸 (stdlib도 그렇게 만들어짐)
fun <T> List<T>.secondOrNull(): T? = if (size >= 2) this[1] else null

// 3. 제네릭 제약 (특정 타입만)
fun List<Int>.sumPositive(): Int = filter { it > 0 }.sum()

// 4. 확장 프로퍼티
val String.lastChar: Char
    get() = this[length - 1]

"hello".lastChar  // 'o'
확장 함수의 함정: 정적 디스패치

확장 함수는 컴파일 타임에 타입에 따라 결정되므로, 다형성이 동작하지 않습니다: 

open class Animal
class Dog : Animal()

fun Animal.sound() = "동물 소리"
fun Dog.sound() = "멍멍"

val a: Animal = Dog()
a.sound()  // "동물 소리" (Animal로 선언되어 있어서)
// 일반 메서드라면 "멍멍"이 호출되지만 확장은 아님

→ "실제 타입에 따라 동작이 달라져야 한다"면 확장 함수가 아니라 일반 메서드 + override를 쓰세요.

7 inline 함수는 언제 쓰나

람다는 내부적으로 객체 할당이 발생합니다. 고차 함수가 자주 호출되면 GC 부담이 될 수 있습니다. inline 키워드를 붙이면 컴파일러가 호출 위치에 코드를 직접 복붙하여 객체 생성을 없앱니다. 

// 일반 고차 함수: 호출할 때마다 람다 객체 생성
fun measure(block: () -> Unit) {
    val start = System.nanoTime()
    block()
    println("elapsed: ${System.nanoTime() - start}")
}

// inline: 람다가 호출 위치에 펼쳐짐
inline fun measure(block: () -> Unit) {
    val start = System.nanoTime()
    block()
    println("elapsed: ${System.nanoTime() - start}")
}

// inline의 부가 효과: return이 호출자를 탈출 가능 (non-local return)
inline fun forEach(list: List<Int>, action: (Int) -> Unit) {
    for (item in list) action(item)
}

fun findFirst(list: List<Int>): Int {
    forEach(list) { item ->
        if (item > 10) return item  // findFirst를 탈출!
    }
    return -1
}
언제 inline을 직접 쓰나

대개 안 씁니다. stdlib의 map, filter, let 등은 이미 inline이라 추가 비용 없음. 내가 만드는 일반 유틸은 inline 안 붙여도 OK. 핫루프에서 반복 호출되는 작은 고차 함수에만 고려.

8 스코프 함수 5종 (⭐ 필수)

let, run, with, apply, also. 객체에 대한 작업을 블록으로 묶어 표현하는 도구. 이름이 비슷해 헷갈리지만, it/this + 리턴값이 객체 자신/블록 결과 2축으로 분류하면 명확.

let

T.let { it: T -> R } : R
객체를 it으로 받아 변환/처리. 리턴은 블록 결과.
x?.let { println(it) }
val len = name?.let { it.length } ?: 0

run

T.run { this: T -> R } : R
객체를 this로 받아 처리. 리턴은 블록 결과.
val url = config.run {
  "$scheme://$host:$port"
}

with

with(T) { this: T -> R } : R
run과 동일하나 확장 함수 아님. 객체 null 아닐 때 사용.
with(stringBuilder) {
  append("a")
  append("b")
}

apply

T.apply { this: T -> Unit } : T
객체를 this로 설정 후 객체 자신 반환. 빌더 패턴에 최적.
val order = Order().apply {
  contract = "BTC"
  size = 1
}

also

T.also { it: T -> Unit } : T
객체를 it으로 부가 작업 후 객체 자신 반환. 로깅/디버깅.
val u = fetchUser()
  .also { logger.info("got $it") }
  .also { metrics.increment() }

선택 가이드 (표)

목적추천이유
null이 아닐 때만 실행 ?.let { } safe call과 조합. it으로 명시적 참조
객체 속성 설정 후 반환 apply this 접근, 빌더 스타일. Java Bean 초기화에 최적
값을 변환해서 반환 let / run it 선호(간단) vs this 선호(여러 멤버 호출)
체인 중간에 side effect also 로깅, 디버깅, 검증 등 부가 작업
여러 메서드 연달아 호출 with 객체를 반복 타이핑 안 해도 됨
프로젝트 사용 예: apply로 Java Bean 초기화

9.2 확장 함수로 공용 로직 재사용

src/main/kotlin/.../market/Indicator.kt 
// EntryCondition과 ExitCondition.IndicatorExit가 공유하는 평가 엔진
fun evaluateIndicator(
    indicator: Indicator, operator: ComparisonOp,
    value: Double, period: Int, prices: List<Double>
): Boolean {
    val current: Double = when (indicator) {
        Indicator.RSI -> calculateRsi(prices, period) ?: return false
        Indicator.SMA -> calculateSma(prices, period) ?: return false
        Indicator.EMA -> calculateEma(prices, period) ?: return false
        Indicator.PRICE -> prices.lastOrNull() ?: return false
    }
    return when (operator) {
        ComparisonOp.LT -> current < value
        ComparisonOp.LTE -> current <= value
        ComparisonOp.GT -> current > value
        ComparisonOp.GTE -> current >= value
    }
}

// EntryCondition
class EntryCondition(...) {
    fun evaluate(prices: List<Double>): Boolean =
        evaluateIndicator(indicator, operator, value, period, prices)
}

// ExitCondition.IndicatorExit도 동일 함수 호출 → DRY

9.3 고차 함수로 retry 공통화

src/main/kotlin/.../client/GateClient.kt 
private inline fun <T> callApi(context: String, action: () -> T): T = withRetry(context) {
    try {
        action()
    } catch (e: GateApiException) {
        throw mapGateException(e, context)
    } catch (e: ApiException) {
        throw wrapApiException(e, context)
    }
}

private inline fun <T> withRetry(context: String, action: () -> T): T {
    return try {
        action()
    } catch (e: RateLimitException) {
        logger.warn("Rate limit, 1.5초 후 재시도: {}", context)
        Thread.sleep(1500)
        action()
    }
}

10 예상 면접 질문

Q. apply와 also의 차이를 설명해주세요.
A. 둘 다 객체 자신을 리턴한다는 점은 같습니다. 차이는 블록 내부에서 객체를 참조하는 방식: apply는 this로, also는 it으로 받습니다. apply는 객체 속성을 설정할 때(setter 연속 호출, Java Bean 초기화), also는 체인 중간에 side-effect(로깅, 메트릭 기록)에 주로 씁니다. 예: User().apply { name = "a"; age = 30 } vs user.also { log.info("got $it") }.
Q. 확장 함수의 동작 원리는?
A. Kotlin 컴파일러는 확장 함수를 첫 번째 파라미터로 receiver를 받는 static 함수로 컴파일합니다. 예: fun String.shout() = ...은 JVM에서 public static String shout(String receiver)로 변환됩니다. 따라서 정적 디스패치이며, 런타임 타입이 아니라 컴파일 타임의 선언 타입을 기준으로 호출됩니다. 다형성이 필요한 경우엔 일반 메서드 + override를 사용해야 합니다.
Q. inline 함수는 언제 쓰나요?
A. 고차 함수에 람다를 넘길 때 발생하는 람다 객체 할당 오버헤드를 제거하고 싶을 때 씁니다. 또한 non-local return(람다 안에서 호출자 탈출) 같은 언어 기능을 가능하게 합니다. 하지만 본문이 긴 함수에 inline을 붙이면 바이트코드가 커져 역효과. 작고 자주 호출되는 고차 함수에만 적용. stdlib의 let/run/apply 등은 이미 inline입니다. 일반적인 비즈니스 로직에선 굳이 쓰지 않습니다.
Q. Python의 데코레이터에 해당하는 Kotlin 패턴은?
A. Kotlin엔 같은 이름의 기능은 없지만 고차 함수 + 후행 람다가 같은 목적을 수행합니다. 예: 재시도, 측정, 트랜잭션 같은 횡단 관심사는 fun <T> withRetry(block: () -> T): T로 구현하고 withRetry { callApi() }처럼 사용합니다. 또 다른 방법은 AOP(Spring AOP의 @Transactional 등)를 쓰는 것. 어노테이션 기반 횡단 관심사는 Spring이 프록시로 감싸서 처리합니다.
Q. let과 run은 어떻게 다른가요?
A. 리턴값(블록 결과)은 동일하지만 receiver 접근 방식이 다릅니다. let은 it으로, run은 this로. let은 주로 null-safe 체인(x?.let { ... })에 쓰고, run은 객체의 여러 멤버를 연속 호출해 결과를 만들 때 (config.run { "$host:$port" }) 씁니다. 외부 this와 충돌하는 컨텍스트에선 let이, 그렇지 않으면 run이 간결.
핵심 요약
  • 함수는 top-level / 멤버 / 로컬 / 확장 4가지. top-level + 확장이 Kotlin의 맛
  • default + named 파라미터로 Python 같은 호출 유연성 확보
  • 람다는 후행 람다 관용구로 "블록"처럼 읽히게 씀
  • 고차 함수로 "데코레이터" 역할의 횡단 관심사 추출
  • 확장 함수는 써드파티 타입에 유틸 추가 + import scope로 안전
  • 스코프 함수는 this/it × 객체 자신/블록 결과 2축으로 선택
