1 Gradle이란 무엇인가

Gradle은 빌드 시스템이자 의존성 매니저이자 태스크 러너입니다. Python 생태계에서 대응 도구를 하나로 묶으면:

역할PythonKotlin/JVM
패키지 설치pip installgradle dependencies (Maven Central에서 자동 다운로드)
의존성 선언requirements.txt / pyproject.tomlbuild.gradle.kts의 dependencies { } 블록
패키징python -m build / wheel./gradlew bootJar → 실행 가능 JAR
태스크 실행make test / tox./gradlew test
가상 환경venv / poetry불필요 (JVM은 분리된 classpath)

DSL이 Kotlin이라는 것이 큰 차이. build.gradle.kts 파일은 실제로 실행되는 Kotlin 스크립트이며, 복잡한 로직을 자유롭게 넣을 수 있습니다. Groovy DSL(build.gradle, 확장자 .kts 아님)도 있지만 신규 프로젝트는 대부분 Kotlin DSL.

2 Gradle Wrapper

프로젝트마다 Gradle 버전을 고정하는 메커니즘. gradlew(Unix), gradlew.bat(Windows) 스크립트가 저장소에 포함됩니다. 

# Python은 사용자가 Python 버전을 맞춰야 함
python3.11 -m pip install ...

# Gradle은 wrapper가 알아서 지정 버전을 다운로드
./gradlew build
# → gradle-wrapper.properties에 명시된 버전을 ~/.gradle에 다운로드 후 사용
# → 팀원 모두 같은 Gradle 버전 보장

프로젝트에 포함되는 wrapper 파일:

  • gradlew, gradlew.bat: 실행 스크립트
  • gradle/wrapper/gradle-wrapper.jar: wrapper 코어 (작음, 버전 체크 + 다운로드)
  • gradle/wrapper/gradle-wrapper.properties: 어떤 Gradle 버전 사용할지
Python 개발자를 위한 관용

Python 프로젝트에서 "시스템 Python vs venv"의 고민이 있는 것처럼, Java 프로젝트에선 "시스템 Gradle vs wrapper"를 결정해야 합니다. 답은 항상 wrapper. ./gradlew만 쓰고 절대 gradle(시스템)을 쓰지 않습니다.

3 build.gradle.kts 해부

가장 기본적인 Spring Boot 프로젝트의 build.gradle.kts:

plugins {
    kotlin("jvm") version "2.1.20"
    kotlin("plugin.spring") version "2.1.20"
    id("org.springframework.boot") version "3.5.3"
    id("io.spring.dependency-management") version "1.1.7"
    id("jacoco")
}

group = "com.parkj3onghoon"
version = "0.0.1-SNAPSHOT"

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(21)
    }
}

repositories {
    mavenCentral()
}

dependencies {
    implementation("org.springframework.boot:spring-boot-starter")
    implementation("org.jetbrains.kotlin:kotlin-reflect")

    testImplementation("org.springframework.boot:spring-boot-starter-test")
    testImplementation("io.mockk:mockk:1.13.16")
}

kotlin {
    compilerOptions {
        freeCompilerArgs.addAll("-Xjsr305=strict")
    }
}

tasks.withType<Test> {
    useJUnitPlatform()
}

블록별 역할

  • plugins { }: 사용할 빌드 기능(Kotlin 컴파일, Spring Boot 패키징 등)
  • group/version: 이 프로젝트가 라이브러리로 배포될 때의 Maven 좌표
  • java { toolchain }: JDK 버전 지정 — Gradle이 맞는 JDK 자동 다운로드/선택
  • repositories { }: 의존성을 어디서 가져올지 (Maven Central이 기본)
  • dependencies { }: 실제 라이브러리 목록 + 범위(scope)
  • tasks.withType<T> { }: 특정 타입 태스크의 설정

4 의존성 선언과 Scope

모든 의존성은 scope를 가집니다. Python requirements.txt는 단일 레벨이지만, Gradle은 컴파일/런타임/테스트를 구분.

Scope의미예시
implementation 컴파일 + 런타임 (기본 선택). 전이 의존은 내 사용자에게 숨김 대부분의 라이브러리
api 컴파일 + 런타임. 전이 의존이 공개됨 (라이브러리 모듈에서 주로) 공통 인터페이스를 모듈 간 공유할 때
runtimeOnly 런타임에만 필요 (컴파일시 불필요) DB 드라이버, logback-classic
compileOnly 컴파일에만 (런타임은 대상 환경이 제공) Servlet API, Lombok(옛날)
testImplementation 테스트 컴파일 + 런타임 JUnit, MockK, Testcontainers
testRuntimeOnly 테스트 런타임만 junit-platform-launcher
annotationProcessor 어노테이션 처리기 (Lombok, MapStruct 등 Java용) 대부분 kapt/ksp로 대체
kapt / ksp Kotlin 어노테이션 처리기 Hibernate, Dagger 등
실무 관용
  • 90%는 implementation. 나머지는 명시적 이유가 있을 때
  • DB 드라이버는 runtimeOnly (코드에서 직접 import하지 않음, JDBC URL로 로딩)
  • 공통 모듈에서 타입을 노출하고 싶으면 api
  • Lombok은 Kotlin에선 안 씀 (data class가 대체)

의존성 좌표 문법

// "group:name:version"
implementation("org.springframework.boot:spring-boot-starter-web:3.5.3")

// Spring Boot는 dependency-management plugin이 버전을 관리하므로
// 버전 생략 가능
implementation("org.springframework.boot:spring-boot-starter-web")

// 특정 전이 의존 제외
implementation("com.example:foo:1.0") {
    exclude(group = "org.slf4j", module = "slf4j-log4j12")
}

// 테스트 클래스가 있는 모듈 전용
implementation(project(":common"))
testImplementation(testFixtures(project(":common")))

5 Repository (Maven Central vs PyPI)

Java 생태계 중심 저장소는 Maven Central. PyPI처럼 공개 저장소 하나로 대부분 충족됩니다. 기업에선 사내 Nexus/Artifactory를 두어 프록시 + 내부 라이브러리 배포용으로 사용. 

repositories {
    mavenCentral()        // 공식 중앙 저장소

    // 추가 저장소
    gradlePluginPortal()                 // Gradle plugin
    maven("https://jitpack.io")          // GitHub 릴리즈 패키징
    maven("https://repo.example.com") {  // 사내
        credentials {
            username = System.getenv("NEXUS_USER")
            password = System.getenv("NEXUS_PASS")
        }
    }
}

6 Task와 라이프사이클

Gradle의 모든 작업은 Task. build, test, bootRun이 전부 Task입니다. 의존 관계가 있어서 ./gradlew build가 compileKotlin → compileTestKotlin → test → jar를 순차 실행.

clean
→
compileKotlin
→
processResources
→
compileTestKotlin
→
test
→
jar
→
bootJar

자주 쓰는 태스크

./gradlew build              # 컴파일 + 테스트 + 패키징
./gradlew test               # 테스트만
./gradlew test --tests "UserServiceTest"  # 특정 테스트
./gradlew bootRun            # Spring Boot 앱 실행
./gradlew bootJar            # 실행 가능 JAR 생성 (build/libs/*.jar)
./gradlew clean              # build 디렉토리 삭제
./gradlew dependencies       # 의존성 트리 출력
./gradlew tasks              # 사용 가능 태스크 목록
./gradlew jacocoTestReport   # 커버리지 리포트 (jacoco plugin 필요)

./gradlew build --no-daemon --rerun-tasks  # 캐시/데몬 무시 (디버깅)
./gradlew build -x test      # test만 건너뛰기

커스텀 Task 만들기

// 간단한 Task
tasks.register("hello") {
    group = "custom"
    description = "인사말 출력"
    doLast {
        println("Hello from Gradle!")
    }
}

// 기존 Task 수정
tasks.named<Test>("test") {
    useJUnitPlatform()
    maxParallelForks = 4
    testLogging {
        events("passed", "skipped", "failed")
    }
}

// 파일 복사 Task
tasks.register<Copy>("copyConfig") {
    from("src/main/resources/config")
    into("$buildDir/config")
}

7 Plugin

Gradle의 기능 확장 단위. 프로젝트 용도에 따라 조합.

plugins {
    // Kotlin 컴파일
    kotlin("jvm") version "2.1.20"

    // Spring 어노테이션 처리 (open 자동화)
    kotlin("plugin.spring") version "2.1.20"

    // JPA Entity의 no-arg 생성자 자동 생성
    kotlin("plugin.jpa") version "2.1.20"

    // Spring Boot fat JAR + bootRun
    id("org.springframework.boot") version "3.5.3"

    // Spring Boot 버전과 호환되는 의존성 버전 자동 관리
    id("io.spring.dependency-management") version "1.1.7"

    // 커버리지
    id("jacoco")

    // 정적 분석
    id("io.gitlab.arturbosch.detekt") version "1.23.7"

    // 포매팅
    id("org.jlleitschuh.gradle.ktlint") version "12.1.1"
}

8 Multi-module 프로젝트

큰 프로젝트는 모듈 단위로 분리합니다. Python의 "모노레포"와 유사하지만 Gradle은 모듈 간 의존 관계를 명시적으로 선언. 

// 루트: settings.gradle.kts
rootProject.name = "tnear-backend"
include(
    ":common",
    ":user-service",
    ":ads-service",
    ":admin-api"
)

// 루트: build.gradle.kts
subprojects {
    apply(plugin = "org.jetbrains.kotlin.jvm")

    repositories {
        mavenCentral()
    }

    dependencies {
        implementation("org.jetbrains.kotlin:kotlin-stdlib")
    }
}

// user-service/build.gradle.kts
dependencies {
    implementation(project(":common"))  // 다른 모듈 참조
    implementation("org.springframework.boot:spring-boot-starter-web")
}
언제 multi-module이 필요한가
  • 도메인 간 경계 강제: user → ads 방향만 허용, 반대는 컴파일 에러
  • 빌드 시간 단축: 변경된 모듈만 재빌드 (incremental)
  • 배포 단위 분리: 서비스별 별도 JAR 생성
  • 공통 코드 공유: common 모듈에 유틸/DTO

9 Version Catalog

최근 실무에선 gradle/libs.versions.toml로 의존성 버전을 중앙 관리합니다. multi-module에서 특히 유용. 

# gradle/libs.versions.toml
[versions]
kotlin = "2.1.20"
spring-boot = "3.5.3"
coroutines = "1.10.2"

[libraries]
spring-boot-starter-web = { module = "org.springframework.boot:spring-boot-starter-web", version.ref = "spring-boot" }
coroutines-core = { module = "org.jetbrains.kotlinx:kotlinx-coroutines-core", version.ref = "coroutines" }
mockk = { module = "io.mockk:mockk", version = "1.13.16" }

[plugins]
kotlin-jvm = { id = "org.jetbrains.kotlin.jvm", version.ref = "kotlin" }
spring-boot = { id = "org.springframework.boot", version.ref = "spring-boot" }
// build.gradle.kts
plugins {
    alias(libs.plugins.kotlin.jvm)
    alias(libs.plugins.spring.boot)
}

dependencies {
    implementation(libs.spring.boot.starter.web)
    implementation(libs.coroutines.core)
    testImplementation(libs.mockk)
}

10 gate-futures-bot build.gradle.kts 분석

build.gradle.kts 
plugins {
    kotlin("jvm") version "2.1.20"              // Kotlin JVM 컴파일
    kotlin("plugin.spring") version "2.1.20"    // @Component 등의 class를 자동 open
    id("org.springframework.boot") version "3.5.3"
    id("io.spring.dependency-management") version "1.1.7"
    id("jacoco")                                 // 커버리지
}

group = "com.parkj3onghoon"
version = "0.0.1-SNAPSHOT"

java {
    toolchain {
        languageVersion = JavaLanguageVersion.of(21)   // JDK 21 필수
    }
}

repositories {
    mavenCentral()
}

dependencies {
    // Spring Boot
    implementation("org.springframework.boot:spring-boot-starter")
    implementation("org.springframework.boot:spring-boot-starter-validation")   // @Validated
    implementation("org.springframework.boot:spring-boot-configuration-processor")  // @ConfigurationProperties IDE 힌트

    // Kotlin
    implementation("org.jetbrains.kotlin:kotlin-reflect")  // Spring이 필요
    implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.10.2")

    // Gate.io SDK
    implementation("io.gate:gate-api:7.1.8")

    // Test
    testImplementation("org.springframework.boot:spring-boot-starter-test")
    testImplementation("org.jetbrains.kotlin:kotlin-test-junit5")
    testImplementation("io.mockk:mockk:1.13.16")
    testImplementation("org.jetbrains.kotlinx:kotlinx-coroutines-test:1.10.2")
    testRuntimeOnly("org.junit.platform:junit-platform-launcher")
}

kotlin {
    compilerOptions {
        // Java의 JSR-305 어노테이션(@Nullable 등)을 엄격히 검사
        freeCompilerArgs.addAll("-Xjsr305=strict")
    }
}

tasks.withType<Test> {
    useJUnitPlatform()   // JUnit 5 엔진 사용
}

tasks.jacocoTestReport {
    dependsOn(tasks.test)
    reports {
        xml.required = true
        html.required = true
    }
}
포인트: kotlin("plugin.spring")

Kotlin 클래스는 기본 final(상속 불가)인데, Spring은 @Component 등을 프록시로 감싸기 위해 상속이 필요합니다. 이 플러그인이 Spring 어노테이션이 붙은 클래스를 자동으로 open으로 만듭니다. 없으면 런타임에 "클래스가 final이라 프록시 불가" 에러.

포인트: spring-boot-starter-validation

@Validated, @NotBlank 등 jakarta validation을 쓰려면 이 starter 필요. ApiProperties, BotProperties의 값 검증에 사용됩니다 (A+ 리팩터링에서 추가).

실행 명령어

./gradlew build               # 컴파일 + 테스트 + JAR
./gradlew test                # 테스트만
./gradlew jacocoTestReport    # 커버리지: build/reports/jacoco/test/html/index.html
./gradlew bootRun             # 봇 시작 (기본 프로파일)
./gradlew bootRun --args='--spring.profiles.active=dev'  # dev 프로파일

# bootJar 생성 후 java로 실행 (production 배포와 동일)
./gradlew bootJar
java -jar build/libs/gate-futures-bot-0.0.1-SNAPSHOT.jar

11 예상 면접 질문

Q. implementation과 api의 차이는?
A. 전이 의존(transitive dependency)의 공개 범위가 다릅니다. implementation으로 선언한 의존성은 해당 모듈 안에서만 보이고, 이 모듈을 의존하는 다른 모듈에는 숨겨집니다. api는 consuming 모듈에도 공개되어 그쪽에서도 import 가능.
실무 기준: 기본은 implementation, 공통 타입을 모듈 경계 밖에서도 써야 하면(예: DTO를 포함한 common 모듈) api. 과도한 api는 의존 그래프가 복잡해지고 재컴파일 범위가 커져 권장되지 않습니다.
Q. Gradle Wrapper는 왜 써야 하나요?
A. Gradle 버전을 저장소에 고정하기 위해. 시스템에 설치된 Gradle은 개발자마다 버전이 달라 "내 로컬에선 빌드되는데 동료 로컬에선 안 됨" 문제가 생깁니다. Wrapper 스크립트(gradlew)는 gradle-wrapper.properties에 명시된 버전을 자동 다운로드/실행하므로 모두 같은 버전을 사용. CI에서도 동일하게 동작.
Q. bootJar와 일반 jar의 차이는?
A. 일반 jar는 내 클래스만 담긴 아카이브로 의존 라이브러리를 별도로 classpath에 추가해야 실행됩니다. bootJar는 Spring Boot의 fat/uber JAR로, 의존 라이브러리를 모두 포함하고 내장 Tomcat까지 담아 java -jar 한 줄로 실행됩니다. 내부적으로는 BOOT-INF/lib/ 아래에 의존 JAR들을 중첩 배치하고 커스텀 MANIFEST.MF의 Main-Class가 Spring의 JarLauncher를 가리킵니다.
Q. Gradle과 Maven 중 어느 것을 선호하나요?
A. 신규 프로젝트는 Gradle (특히 Kotlin DSL)이 대세입니다. 이유: ① Kotlin DSL로 복잡한 로직 표현(커스텀 task, conditional dependency 등)이 자연스럽고, ② incremental build + build cache로 Maven보다 빠르며, ③ multi-module 구성이 더 유연합니다. 단, Maven은 XML이 선언적이라 툴 생태계 지원(GitHub Dependabot, 취약점 스캐너 등)이 더 잘되고 배우기 쉽습니다. 기존 Maven 프로젝트를 유지하는 회사도 많고, 둘 다 읽을 수 있어야 합니다.
Q. Gradle 빌드 속도가 느릴 때 개선 방법은?
A. ①Gradle daemon 활성(기본). 두 번째 실행부터 빨라짐. ②Build cache: org.gradle.caching=true — 동일 입력이면 출력 재사용. ③Parallel 실행: org.gradle.parallel=true — multi-module 병렬 빌드. ④Configuration cache (Gradle 6.6+): 구성 단계 캐싱. ⑤테스트 분할: maxParallelForks로 테스트 병렬. ⑥불필요한 의존성 정리: ./gradlew dependencies로 트리 점검. CI에선 --parallel --build-cache 기본 조합, 그리고 remote build cache (Gradle Enterprise 등)로 팀 단위 캐시 공유.
핵심 요약
  • Gradle = 빌드 + 의존성 + 태스크 러너 (Python의 pip+setup+make 합침)
  • 항상 Gradle Wrapper(./gradlew) 사용
  • 의존성 scope: implementation이 기본, 필요시 api/runtimeOnly/compileOnly
  • Spring Boot 프로젝트는 kotlin("plugin.spring") 필수 (final 자동 open)
  • Multi-module로 경계 강제 + 빌드 시간 단축
  • Version Catalog로 의존성 중앙 관리 (libs.versions.toml)
