1 JPA vs Hibernate vs Spring Data JPA

처음 Java 세계에 오면 이 3가지 용어가 섞여 혼란스럽습니다:

  • JPA (Java Persistence API): 자바 표준 스펙. 어노테이션 + 인터페이스 명세 (@Entity, EntityManager 등). 구현체는 없음.
  • Hibernate: JPA의 대표 구현체. 실제로 SQL을 생성하고 DB와 통신. Spring Boot 기본.
  • Spring Data JPA: Spring이 JPA 위에 얹은 추상화 레이어. JpaRepository 인터페이스만 선언하면 구현체 자동 생성.
한 줄 요약

JPA 어노테이션으로 Entity를 정의 → Hibernate가 SQL로 변환 → Spring Data JPA가 Repository 구현을 자동 생성. 개발자는 인터페이스만 선언하고 Hibernate의 엔진이 이걸 실행.

3 @Entity 선언

@Entity
@Table(name = "users", indexes = [
    Index(name = "idx_users_email", columnList = "email", unique = true)
])
class User(
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)  // AUTO_INCREMENT
    val id: Long = 0,

    @Column(nullable = false, length = 30)
    var name: String,

    @Column(nullable = false, unique = true)
    var email: String,

    @Enumerated(EnumType.STRING)   // ORDINAL 대신 STRING 필수!
    var status: UserStatus = UserStatus.ACTIVE,

    @Column(name = "created_at", updatable = false)
    val createdAt: Instant = Instant.now(),

    @Version                         // 낙관적 락
    var version: Long = 0
) {
    // 비즈니스 메서드 — Entity에 포함해도 OK (Rich Domain Model)
    fun deactivate() {
        require(status == UserStatus.ACTIVE) { "이미 비활성" }
        status = UserStatus.INACTIVE
    }
}

enum class UserStatus { ACTIVE, INACTIVE, BANNED }
@Enumerated(ORDINAL) 사용 금지

기본값 ORDINAL은 enum을 순서 기반 정수로 저장합니다. enum 값 순서를 바꾸거나 중간에 추가하면 DB 데이터가 모두 뒤틀림. 반드시 @Enumerated(EnumType.STRING)으로 문자열 저장하세요.

4 연관관계 매핑

1:N (OneToMany)

@Entity
class Order(
    @Id @GeneratedValue val id: Long = 0,

    @ManyToOne(fetch = FetchType.LAZY)     // ⭐ 항상 LAZY
    @JoinColumn(name = "user_id")
    val user: User,

    @OneToMany(
        mappedBy = "order",
        cascade = [CascadeType.ALL],
        orphanRemoval = true
    )
    val items: MutableList<OrderItem> = mutableListOf()
) {
    fun addItem(item: OrderItem) {
        items.add(item)
        item.order = this   // 양방향 동기화
    }
}

@Entity
class OrderItem(
    @Id @GeneratedValue val id: Long = 0,

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "order_id")
    var order: Order,

    val productId: String,
    val quantity: Int
)

M:N (ManyToMany) — 대개 중간 Entity로 풀어씀

// 권장 X: @ManyToMany
// 중간 테이블을 Entity로 만드는 게 유연성 높음

@Entity
class Post(
    @Id @GeneratedValue val id: Long = 0,
    val title: String,

    @OneToMany(mappedBy = "post", cascade = [CascadeType.ALL])
    val tags: MutableList<PostTag> = mutableListOf()
)

@Entity
@Table(name = "post_tags")
class PostTag(
    @EmbeddedId val id: PostTagId,

    @MapsId("postId")
    @ManyToOne(fetch = FetchType.LAZY)
    val post: Post,

    @MapsId("tagId")
    @ManyToOne(fetch = FetchType.LAZY)
    val tag: Tag,

    @Column(name = "created_at")
    val createdAt: Instant = Instant.now()  // 관계에 대한 추가 필드
)

@Embeddable
data class PostTagId(
    val postId: Long,
    val tagId: Long
) : Serializable
Fetch Type: 무조건 LAZY

@ManyToOne의 기본값은 EAGER(즉시 로딩) — NEVER USE. EAGER는 원하지 않는 타이밍에 JOIN을 일으켜 N+1, 불필요한 메모리 사용을 유발. 모든 관계는 LAZY로 명시하고, 필요시 fetch join으로 명시적으로 가져옵니다.

5 영속성 컨텍스트 (⭐ 가장 중요)

JPA의 심장. EntityManager가 관리하는 1차 캐시 + 변경 감지 공간. 트랜잭션 내에서 조회한 Entity는 모두 여기에 등록되고, 트랜잭션 커밋 시 변경 사항이 DB로 자동 반영됩니다.

영속성 컨텍스트 (트랜잭션 내)

EntityManager가 관리하는 1차 캐시

User(id=1, name="Teddy") ← dirty
Order(id=42) ← clean
User(id=2, name="Bob") ← new
OrderItem(id=7) ← removed

DB (실제 테이블)

커밋 시 flush되어 반영

UPDATE users SET name='Teddy' WHERE id=1
INSERT INTO users ...
DELETE FROM order_items WHERE id=7

Entity 생명주기 상태

Transient
(new)
— persist() →
Managed
(영속)
— detach() →
Detached
(준영속)

Managed — remove() →

Dirty Checking 예제

@Service
@Transactional
class UserService(private val userRepo: UserRepository) {

    fun updateName(id: Long, newName: String) {
        val user = userRepo.findById(id).orElseThrow()
        user.name = newName
        // save() 호출 없음!
        // 트랜잭션 커밋 시 Hibernate가 변경 감지 → UPDATE 자동 발행
    }
}
영속성 컨텍스트의 이점
  1. 1차 캐시: 같은 트랜잭션에서 같은 id 조회시 DB 안 감
  2. Dirty checking: 변경된 필드만 UPDATE 자동 발행
  3. 쓰기 지연: INSERT/UPDATE를 모아 커밋 시점에 한 번에 flush
  4. 동일성 보장: 같은 트랜잭션에서 같은 Entity 참조는 ===로 같음
Detached Entity 다루기의 함정
// 트랜잭션이 끝나면 Entity는 Detached 상태
@Transactional(readOnly = true)
fun getUser(id: Long): User = userRepo.findById(id).orElseThrow()
// ← 리턴되는 순간 Detached. Lazy 관계 접근시 LazyInitializationException!

// 해결:
// ① 트랜잭션 안에서 필요한 데이터 모두 로드 (fetch join)
// ② DTO로 변환 후 반환
// ③ @Transactional 범위 연장 (open-in-view, 권장 X)

6 JpaRepository 인터페이스

Spring Data JPA의 마법 — 인터페이스만 선언하면 구현체 자동 생성.

interface UserRepository : JpaRepository<User, Long> {
    // 1. 쿼리 메서드 — 메서드 이름으로 쿼리 자동 생성
    fun findByEmail(email: String): User?
    fun findByNameContaining(keyword: String, pageable: Pageable): Page<User>
    fun countByStatus(status: UserStatus): Long
    fun existsByEmail(email: String): Boolean
    fun deleteByCreatedAtBefore(before: Instant): Long

    // 2. @Query JPQL
    @Query("SELECT u FROM User u WHERE u.email LIKE %:domain")
    fun findByEmailDomain(@Param("domain") domain: String): List<User>

    // 3. Native SQL
    @Query(
        value = "SELECT * FROM users WHERE JSON_EXTRACT(metadata, '$.tier') = :tier",
        nativeQuery = true
    )
    fun findByTier(@Param("tier") tier: String): List<User>

    // 4. Projection (필요한 필드만)
    interface UserSummary {
        fun getId(): Long
        fun getName(): String
    }
    fun findAllProjectedBy(pageable: Pageable): Page<UserSummary>

    // 5. 벌크 업데이트/삭제
    @Modifying
    @Query("UPDATE User u SET u.status = :status WHERE u.lastLoginAt < :threshold")
    fun deactivateInactive(
        @Param("status") status: UserStatus,
        @Param("threshold") threshold: Instant
    ): Int
}

// - save(entity), saveAll(entities)
// - findById(id): Optional<T>
// - findAll(), findAllById(ids)
// - deleteById(id), delete(entity)
// - count()

7 쿼리 작성 방법 4가지

7.1 메서드 이름 (위에서 설명)

간단한 조회에 최적. 메서드 이름이 너무 길어지면 다른 방식.

7.2 JPQL (@Query)

JPA의 표준 쿼리 언어. 테이블이 아닌 Entity 기준으로 작성.

@Query("""
    SELECT o FROM Order o
    JOIN FETCH o.user
    WHERE o.status = :status
    AND o.createdAt >= :since
""")
fun findRecentByStatus(
    @Param("status") status: OrderStatus,
    @Param("since") since: Instant
): List<Order>

7.3 Native SQL

JPQL로 표현 어려운 DB 특화 기능 (Window 함수, JSON 연산 등).

@Query(
    value = """
        SELECT * FROM orders
        WHERE status = :status
        AND created_at >= NOW() - INTERVAL '7 days'
    """,
    nativeQuery = true
)
fun findRecent(@Param("status") status: String): List<Order>

7.4 QueryDSL (타입 안전 동적 쿼리)

복잡한 동적 쿼리(검색 필터 등)에 유용. 별도 의존성 필요.

@Repository
class OrderQueryRepository(
    private val queryFactory: JPAQueryFactory
) {
    fun search(filter: OrderFilter): List<Order> {
        val q = QOrder.order
        return queryFactory
            .selectFrom(q)
            .where(
                filter.status?.let { q.status.eq(it) },
                filter.minAmount?.let { q.amount.goe(it) },
                filter.userId?.let { q.user.id.eq(it) }
            )
            .orderBy(q.createdAt.desc())
            .fetch()
    }
}

    println(order.user.name)      // N번 쿼리: SELECT * FROM users WHERE id = ?

9 @Transactional

@Service
@Transactional  // 클래스 전체에 적용
class OrderService(
    private val orderRepo: OrderRepository,
    private val stockService: StockService
) {
    // 기본: REQUIRED — 기존 트랜잭션 있으면 참여, 없으면 새로 시작
    fun placeOrder(cmd: OrderCommand): Order {
        stockService.reserve(cmd.productId, cmd.quantity)  // 같은 트랜잭션
        val order = Order.create(cmd)
        return orderRepo.save(order)
    }

    // 읽기 전용 최적화: flush 안 함, 변경 감지 안 함
    @Transactional(readOnly = true)
    fun getOrder(id: Long): Order = orderRepo.findById(id).orElseThrow()

    // 새 트랜잭션 강제 (상위 트랜잭션과 분리)
    @Transactional(propagation = Propagation.REQUIRES_NEW)
    fun auditLog(msg: String) {
        // 상위 트랜잭션이 롤백되어도 이 로그는 남음
    }

    // 타임아웃 + 격리 수준
    @Transactional(
        timeout = 10,
        isolation = Isolation.READ_COMMITTED
    )
    fun complexOperation() { }
}

롤백 규칙

  • 기본: RuntimeException에만 롤백. Checked exception은 커밋됨 (Java 유래)
  • 명시: @Transactional(rollbackFor = [MyCheckedException::class])
  • 예외 안 던지고 롤백: TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()
@Transactional의 함정: 프록시 기반
@Service
class OrderService {
    fun outer() {
        this.inner()  // ❌ 프록시 bypass! inner의 @Transactional 무시
    }

    @Transactional
    fun inner() { /* ... */ }
}

// 해결: self-injection 또는 클래스 분리
@Service
class OrderService(
    @Lazy private val self: OrderService  // 자기 참조를 프록시로
) {
    fun outer() {
        self.inner()  // ✅ 프록시 통해 호출
    }
    @Transactional
    fun inner() { }
}

Offset vs Cursor 기반

  • Offset 기반(OFFSET N LIMIT M): 전체 count 가능하지만, offset이 클수록 느림
  • Cursor 기반(WHERE id < last_id LIMIT N): 빠르지만 전체 count 불가, 특정 페이지로 점프 불가

대량 데이터 + 무한 스크롤이면 Cursor 기반 권장.

11 Flyway / Liquibase

Django의 makemigrations/migrate에 대응. JPA의 ddl-auto: update는 개발용이며 프로덕션에선 절대 사용 금지. 대신 Flyway/Liquibase로 명시적 스키마 관리. 

# application.yml
spring:
  jpa:
    hibernate:
      ddl-auto: validate   # Entity ↔ 스키마 일치 여부만 검증

  flyway:
    enabled: true
    locations: classpath:db/migration

12 예상 면접 질문

Q. JPA와 Hibernate, Spring Data JPA의 관계는?
A. JPA는 자바 영속성 표준 스펙(어노테이션과 인터페이스 명세). Hibernate는 JPA의 대표 구현체로 실제 SQL을 생성하고 DB와 통신. Spring Data JPA는 JPA 위에 얹은 Spring의 추상화 레이어로, JpaRepository 인터페이스만 선언하면 구현체를 자동 생성합니다. 개발자는 JPA 어노테이션으로 Entity를 매핑하고, Spring Data JPA 인터페이스로 쿼리를 선언하며, 실제 실행은 Hibernate가 담당합니다.
Q. N+1 문제를 설명하고 해결 방법을 알려주세요.
A. Lazy 연관관계를 반복문에서 접근할 때 1번의 메인 쿼리 + N번의 추가 쿼리가 발생하는 문제. 예: orders.forEach { it.user.name }에서 각 order마다 user를 가져오는 쿼리가 발생. 해결책: ①fetch join(JPQL의 JOIN FETCH)로 한 번의 JOIN 쿼리로 해결. ②@EntityGraph로 선언적으로 eager 로딩 범위 지정. ③default_batch_fetch_size(예: 100)로 N을 N/100으로 감소 (IN 절). ④ DTO Projection으로 애초에 필요한 필드만 조회. 어떤 방법이든 "의도하지 않은 Lazy 접근을 제거"하는 것이 핵심.
Q. 영속성 컨텍스트는 무엇이고 왜 필요한가요?
A. EntityManager가 관리하는 Entity 저장소(트랜잭션 범위의 1차 캐시). 4가지 이점: ①1차 캐시: 같은 트랜잭션 내 동일 id 조회는 DB 재접근 없음. ②Dirty checking: 변경된 Entity를 감지해 커밋 시 UPDATE 자동 발행. save() 불필요. ③쓰기 지연: INSERT/UPDATE를 커밋 시점에 모아 한 번에 flush → DB 왕복 감소. ④동일성 보장: 같은 id는 ===로 동일 인스턴스. 단점은 트랜잭션 밖에서 Lazy 관계 접근시 LazyInitializationException이 발생하니 경계가 명확해야 합니다.
Q. JPA Entity는 data class로 만들어도 되나요?
A. 권장하지 않습니다. 이유: ① data class의 equals/hashCode는 모든 필드를 비교하는데, 영속성 컨텍스트에서 id만 일치하면 같은 Entity여야 합니다. 프록시 객체와 실제 객체의 equals가 깨질 수 있음. ② copy()가 DB 관점에서 의미가 없고 오히려 혼란. ③ Entity는 상태가 변화(dirty checking)하므로 "불변 값 객체" 개념과 맞지 않음. 일반 class로 선언하고 @Id 기반 equals를 명시적으로 구현합니다. DTO는 data class가 적합.
Q. @Transactional이 동작하지 않는 경우가 있나요?
A. 여러 경우: ①같은 클래스 내부 호출 — Spring AOP가 프록시 기반이라 this.inner()는 프록시를 거치지 않아 트랜잭션 적용 X. ②private 메서드에 붙임 — 프록시가 override 불가라 무시됨. ③checked exception 던짐 — 기본 롤백 규칙은 RuntimeException에만 적용. rollbackFor 명시 필요. ④예외 잡고 삼킴 — 예외가 외부로 나가지 않으면 롤백 신호도 없음. ⑤@Async 메서드에서 호출 — 다른 스레드라 트랜잭션 컨텍스트가 없음. 해결: self-injection, public 메서드 사용, 명시적 rollbackFor, AspectJ 모드 등.
Q. ddl-auto를 어떻게 설정하시나요?
A. 프로덕션은 validate(Entity ↔ 스키마 일치만 검증). 스키마 변경은 Flyway/Liquibase로 명시적 마이그레이션. 이유: ①update는 컬럼 타입 변경/삭제를 하지 않아 점진적 왜곡. ②create-drop은 재시작시 전체 날려 데이터 손실. ③스키마 변경 이력이 남지 않아 롤백/감사 불가. 개발은 update도 가능하지만 테스트는 Flyway를 타게 해서 프로덕션과 동일한 마이그레이션 경로 검증이 좋음.
핵심 요약
  • JPA(스펙) + Hibernate(구현) + Spring Data JPA(Repository 자동 생성)
  • Entity는 일반 class로, @Enumerated(STRING), fetch LAZY 기본
  • 영속성 컨텍스트의 dirty checking으로 save() 불필요, 단 트랜잭션 경계 주의
  • N+1 = fetch join / @EntityGraph / batch size / DTO projection
  • @Transactional: 내부 호출/private/checked exception 함정
  • 스키마는 Flyway/Liquibase, ddl-auto: validate
  • 페이징은 Pageable, 대량 데이터엔 cursor 기반 고려
