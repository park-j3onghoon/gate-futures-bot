1 세 기둥: Logs · Metrics · Traces

관측성(observability)의 3가지 데이터 종류. 역할이 다름.

Logs (로그)

사람이 읽을 수 있는 이벤트 기록. "무슨 일이 일어났나"의 상세 설명. 고비용 (디스크/저장소).
SLF4J, Logback, JSON Appender, ELK/Loki

Metrics (지표)

시계열 숫자 데이터. "얼마나 많이, 얼마나 빠르게" 집계된 값. 저비용, 대시보드/알림용.
Micrometer, Prometheus, CloudWatch, Grafana

Traces (추적)

요청이 여러 서비스를 거치는 경로 기록. "어디서 느려졌나"를 보여줌. MSA 필수.
Micrometer Tracing, Zipkin, Jaeger, AWS X-Ray
실무 조합

Logs → 사고 조사용 (post-mortem).
Metrics → 실시간 모니터링/알림 (PagerDuty).
Traces → 성능 분석/디버깅 (한 요청이 왜 5초 걸렸나).
셋 다 필요. 각각 다른 질문에 답함.

2 로깅 (SLF4J + Logback)

Spring Boot 기본 로깅 스택: SLF4J(API) + Logback(구현). Python의 logging과 유사하지만 API와 구현이 분리됨 → 런타임에 구현 교체 가능. 

import org.slf4j.LoggerFactory

class OrderService {
    private val logger = LoggerFactory.getLogger(OrderService::class.java)

    fun placeOrder(cmd: OrderCommand) {
        logger.trace("trace level")     // 가장 상세
        logger.debug("debug: {}", cmd)   // 개발용
        logger.info("주문 생성: id={}, amount={}", id, amount)  // 정상 이벤트
        logger.warn("재고 부족 경고: productId={}", productId)   // 주의
        logger.error("주문 실패", exception)                      // 에러
    }
}

// 또는 Kotlin 확장 (mu-kotlin-logging)
import io.github.oshai.kotlinlogging.KotlinLogging
private val logger = KotlinLogging.logger {}

logger.info { "주문 생성: id=$id" }  // 람다 — 로그 레벨 비활성시 평가 안 함 (성능)
SLF4J 플레이스홀더 사용 (성능)
// ❌ 문자열 연결이 항상 수행됨
logger.debug("user = " + user.toString())

// ✅ debug 비활성시 user.toString() 호출 안 됨
logger.debug("user = {}", user)

3 MDC로 구조화된 로깅

Mapped Diagnostic Context. 스레드-로컬 저장소에 컨텍스트(traceId, userId 등) 저장 → 모든 로그에 자동 포함. 분산 시스템에서 "이 요청의 모든 로그"를 한번에 grep할 수 있게 해줌. 

import org.slf4j.MDC

// 수동 MDC
MDC.put("userId", "42")
MDC.put("requestId", UUID.randomUUID().toString())
try {
    logger.info("주문 처리 시작")   // 로그에 userId, requestId 자동 포함
    processOrder()
} finally {
    MDC.clear()  // 반드시 정리 (thread pool 재사용 대비)
}

// Servlet Filter로 모든 요청에 traceId 자동 주입
@Component
class TraceIdFilter : OncePerRequestFilter() {
    override fun doFilterInternal(
        request: HttpServletRequest,
        response: HttpServletResponse,
        chain: FilterChain
    ) {
        val traceId = request.getHeader("X-Trace-Id") ?: UUID.randomUUID().toString()
        MDC.put("traceId", traceId)
        response.setHeader("X-Trace-Id", traceId)
        try {
            chain.doFilter(request, response)
        } finally {
            MDC.clear()
        }
    }
}

// application.yml에서 패턴에 MDC 포함
logging:
  pattern:
    console: "%d{HH:mm:ss.SSS} [%X{traceId}] [%X{userId}] %-5level %logger - %msg%n"
프로젝트 사용 예: contract를 MDC로
src/main/kotlin/.../worker/CoinWorker.kt 
suspend fun run() {
    MDC.put("contract", contract)   // 워커 시작시 주입
    try {
        while (currentCoroutineContext().isActive) {
            runOnce()
            delay(config.checkIntervalMillis)
        }
    } finally {
        logger.info("워커 종료")
        MDC.remove("contract")      // 종료시 정리
    }
}

// 로그 출력 (application.yml의 pattern에 [contract=%X{contract}] 포함)
// 2026-04-17 15:30:00 [contract=BTC_USDT] INFO  CoinWorker - 롱 진입 시그널
// 2026-04-17 15:30:05 [contract=ETH_USDT] INFO  CoinWorker - 청산 시그널
// → contract별 grep이 쉬움

4 Spring Boot Actuator

의존성 하나 추가하면 운영 엔드포인트가 자동 제공. 수동으로 구현할 필요 없음.

// build.gradle.kts
implementation("org.springframework.boot:spring-boot-starter-actuator")
# application.yml
management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics, prometheus, loggers, threaddump, heapdump
      base-path: /actuator
  endpoint:
    health:
      show-details: when-authorized  # 인증된 사용자만 상세
      probes:
        enabled: true   # liveness, readiness 분리
  metrics:
    export:
      prometheus:
        enabled: true
  info:
    git:
      enabled: true
      mode: full

6 커스텀 메트릭

@Service
class OrderService(
    private val orderRepo: OrderRepository,
    private val meterRegistry: MeterRegistry
) {
    // 1. Counter — 단조 증가하는 카운트
    private val orderCounter = Counter.builder("orders.created")
        .description("생성된 주문 수")
        .tag("region", "kr")
        .register(meterRegistry)

    // 2. Timer — 지연 시간 분포 + 호출 수
    private val orderTimer = Timer.builder("orders.placement.duration")
        .description("주문 처리 시간")
        .publishPercentiles(0.5, 0.95, 0.99)  // p50, p95, p99
        .register(meterRegistry)

    // 3. Gauge — 현재 값 (게이지)
    init {
        Gauge.builder("orders.pending.count") { orderRepo.countByStatus("PENDING") }
            .description("대기중인 주문 수")
            .register(meterRegistry)
    }

    fun placeOrder(cmd: OrderCommand): Order {
        return orderTimer.record<Order> {
            val order = orderRepo.save(Order.create(cmd))
            orderCounter.increment()
            // 태그 동적 추가
            Counter.builder("orders.created")
                .tag("status", order.status.name)
                .tag("channel", cmd.channel)
                .register(meterRegistry)
                .increment()
            order
        }
    }
}

8 분산 추적

한 요청이 여러 서비스를 거칠 때, 각 단계의 지연을 추적. MSA에선 필수. Spring Boot 3.x는 Micrometer Tracing이 표준 (이전 Spring Cloud Sleuth의 후속). 

// build.gradle.kts
implementation("io.micrometer:micrometer-tracing-bridge-brave")
implementation("io.zipkin.reporter2:zipkin-reporter-brave")

// application.yml
management:
  tracing:
    sampling:
      probability: 0.1   # 10% 샘플링 (prod)
  zipkin:
    tracing:
      endpoint: http://zipkin:9411/api/v2/spans
// 자동 계측: Spring MVC, WebClient, RestTemplate, JPA 등
// → HTTP 헤더(B3 또는 W3C Trace Context)에 traceId/spanId 자동 전파

// 수동 Span
@Component
class OrderService(private val tracer: Tracer) {
    fun placeOrder(cmd: OrderCommand): Order {
        val span = tracer.nextSpan().name("placeOrder").start()
        try {
            tracer.withSpan(span).use {
                return doPlaceOrder(cmd)
            }
        } catch (e: Exception) {
            span.error(e)
            throw e
        } finally {
            span.end()
        }
    }
}

결과: Zipkin/Jaeger UI에서 "주문 요청 → API Gateway → Order Service → Payment Service → DB"의 각 단계 소요 시간이 폭포수 차트로 표시됨. 병목 식별 쉬움.

9 알림 설계

메트릭이 있으면 임계값 초과시 자동 알림이 가능합니다. PagerDuty/Slack 연동.

SLO 기반 알림 설계 예시

# Prometheus Alertmanager
groups:
  - name: api
    interval: 30s
    rules:
      - alert: HighErrorRate
        expr: |
          sum(rate(http_server_requests_seconds_count{status=~"5.."}[5m]))
          / sum(rate(http_server_requests_seconds_count[5m])) > 0.01
        for: 5m   # 5분 이상 지속시
        labels:
          severity: critical
          team: backend
        annotations:
          summary: "에러율 1% 초과 ({{ $value | humanizePercentage }})"
          runbook: "https://wiki/runbook/high-error-rate"

      - alert: SlowResponse
        expr: |
          histogram_quantile(0.95,
            sum(rate(http_server_requests_seconds_bucket[5m])) by (le)
          ) > 1
        for: 10m
        annotations:
          summary: "p95 응답 시간 1초 초과"

      - alert: HighJvmHeap
        expr: jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.9
        for: 15m
알림 설계 원칙
  • Actionable: 알림 받으면 뭘 해야 하는지 명확 (runbook 링크)
  • Severity 구분: critical (즉시 대응) / warning (다음 영업일)
  • 알림 피로 방지: for: 5m으로 일시적 spike 무시
  • 근본 원인 지향: "에러율 높음"(증상) 대신 "DB 커넥션 고갈"(원인)
  • silence: 배포 중에는 알림 음소거 (Alertmanager inhibit rule)

11 예상 면접 질문

Q. 관측성의 3가지 기둥을 설명해주세요.
A. Logs(로그): 사람이 읽을 수 있는 이벤트 기록. "무슨 일이 일어났나"의 상세. 사고 post-mortem에 필수. Metrics(메트릭): 시계열 숫자 데이터. "얼마나 많이, 얼마나 빠르게". 저비용이고 실시간 대시보드/알림에 사용. Traces(분산 추적): 한 요청이 여러 서비스를 거치는 경로. "어디서 느려졌나". MSA에서 필수. 셋은 서로 다른 질문에 답하므로 함께 사용합니다. 로그로 사건 재구성, 메트릭으로 추세/알림, 트레이스로 병목 식별.
Q. MDC가 무엇이고 어떻게 활용하나요?
A. Mapped Diagnostic Context. 스레드-로컬 Map에 컨텍스트(traceId, userId, orderId 등)를 담아두면 모든 로그에 자동 포함됩니다. 활용: ①HTTP 요청마다 Filter에서 traceId 주입 → 해당 요청의 모든 로그가 연결됨. ②워커/배치 작업에 contract, batchId 같은 식별자. ③코루틴에서는 MDCContext로 dispatcher 전환시도 유지. 주의: thread pool 재사용을 고려해 try-finally로 반드시 MDC.clear(). 안 하면 이전 요청의 값이 섞임.
Q. Actuator를 프로덕션에서 안전하게 쓰려면?
A. 기본 설정으로 모두 노출하면 정보 누출 위험. 필수 조치: ①노출 엔드포인트 제한: management.endpoints.web.exposure.include: health,info,metrics,prometheus. ②별도 포트: management.server.port: 8081로 외부 공개 포트(8080)와 분리, 8081은 내부망만. ③인증: Spring Security로 /actuator/** 보호 (특히 loggers, heapdump, env). ④민감 정보 숨김: management.endpoint.env.show-values: when-authorized. ⑤Kubernetes: health/liveness, health/readiness 분리로 probe 용도 분리.
Q. 어떤 메트릭을 핵심으로 보시나요?
A. RED 방식으로 보통 정리: Rate(초당 요청 수), Errors(5xx 비율), Duration(p50/p95/p99 지연). 또는 USE(Utilization/Saturation/Errors) — 시스템 리소스 관점. 실무에선: ①http_server_requests_seconds의 rate/duration, ②jvm_memory_used 및 jvm_gc_pause, ③HikariCP hikaricp_connections_usage, ④비즈니스 카운터(주문수, 결제 성공률), ⑤외부 API 호출 성공률/지연. SLO(Service Level Objective)를 먼저 정의하고("p95 < 200ms, error rate < 0.1%") 거기서 필요한 메트릭을 결정.
Q. 로그 레벨은 어떻게 쓰시나요?
A. ①ERROR: 시스템에 문제 발생, 즉각 조치 필요. 예: DB 연결 실패, 중요 비즈니스 로직 실패. ②WARN: 문제는 아니지만 주의. 예: 재시도, deprecated API 사용, fallback 발생. ③INFO: 주요 이벤트. 예: 앱 시작/종료, 주문 생성/취소 같은 비즈니스 이벤트. ④DEBUG: 개발/디버깅 시 상세 정보. 프로덕션 기본은 INFO. ⑤TRACE: 매우 상세 (메서드 진입/종료 등). 거의 안 씀. 프로덕션은 INFO로 고정, 문제 발생시 /actuator/loggers로 특정 패키지만 DEBUG로 임시 변경.

  • MDC로 로그에 컨텍스트(traceId, userId) 자동 주입
  • 구조화 로깅 (JSON) + ELK/Loki로 검색 가능한 로그
  • Actuator + Micrometer로 /health, /metrics 자동 제공
  • Prometheus로 수집, Grafana로 시각화, Alertmanager로 알림
  • 분산 추적은 Micrometer Tracing + Zipkin/Jaeger
  • SLO 정의 → RED 메트릭 → actionable alert 설계
