1 왜 코루틴인가

동시성 처리 방식은 크게 3가지: OS 스레드(무겁고 비쌈), async/await(이벤트 루프 기반, 단일 스레드), 코루틴(경량 스레드 + 스레드 풀). Kotlin 코루틴은 Python asyncio의 "구조화된 버전 + 멀티 스레드 분산".

Thread (OS 스레드)

각 스레드 = ~1MB 스택. 수천 개 넘으면 OOM.
컨텍스트 스위칭 비용 큼.

← 5개 ~5MB
동시 처리: 수백 수준

Coroutine (경량)

각 코루틴 = ~수백 바이트. 스택 없음(continuation).
스레드 풀 재사용 → 적은 스레드로 많은 작업.

← 15개 ~수KB
동시 처리: 수십~수백만
import asyncio

async def fetch(url):
    # await로 논블로킹 대기
    async with httpx.AsyncClient() as c:
        r = await c.get(url)
        return r.text

async def main():
    # 여러 요청 동시 실행
    urls = ["u1", "u2", "u3"]
    results = await asyncio.gather(*[fetch(u) for u in urls])
    print(results)

asyncio.run(main())
import kotlinx.coroutines.*

// suspend = await 가능한 함수
suspend fun fetch(url: String): String {
    // HTTP 클라이언트의 suspend API
    return httpClient.get(url).body()
}

suspend fun main() = coroutineScope {
    val urls = listOf("u1", "u2", "u3")
    val results = urls.map { url ->
        async { fetch(url) }   // 동시 시작
    }.awaitAll()              // 모두 대기
    println(results)
}

2 suspend 함수

suspend 키워드가 붙은 함수는 중단/재개가 가능합니다. Python async def에 대응. 실행 중 delay, await, HTTP 호출 등을 만나면 스레드를 블록하지 않고 양보하고, 결과가 돌아오면 재개됩니다. 

// suspend 함수는 다른 suspend 함수나 코루틴 빌더에서만 호출
suspend fun loadUser(id: Long): User {
    delay(100)  // 논블로킹 대기 (Thread.sleep 아님!)
    return httpClient.get("/users/$id").body()
}

// 일반 함수에서 호출하려면 launch/runBlocking 등 필요
fun regularFunction() {
    // loadUser(1)  // ← 컴파일 에러

    runBlocking {           // 일반 → 코루틴 진입점
        val user = loadUser(1)  // OK
        println(user)
    }
}
suspend의 핵심 규칙
  • suspend 함수는 suspend 함수/코루틴 안에서만 호출
  • 내부적으로 Continuation<T>을 파라미터로 받는 일반 함수로 컴파일 (CPS transform)
  • Thread.sleep 대신 delay, blocking IO 대신 suspend IO 써야 효과 있음
  • suspend 함수 자체는 스레드를 블록하지 않는다는 "약속" — 개발자가 책임

3 CoroutineScope와 Job

모든 코루틴은 어떤 scope에 속해야 합니다 (고아 코루틴 금지). Scope는 "이 scope의 모든 자식 코루틴을 함께 관리"한다는 생명주기 단위. 

// Scope 만드는 법
val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())

// launch: 새 코루틴을 시작 (Job 반환)
val job: Job = scope.launch {
    loadAndProcess()
}

// Job으로 제어
job.cancel()            // 취소
job.join()              // 완료 대기
job.isActive            // 실행 중?
job.isCompleted         // 완료됨?
job.isCancelled         // 취소됨?

// scope 전체 취소 (자식까지)
scope.cancel()

Scope의 구조

CoroutineScope
Job (부모)
child job 1 (launch)
child job 2 (async)
child job 3

자식 Job 중 하나라도 실패하면 → 부모 전체 취소 → 형제들도 취소 (기본 Job()).
SupervisorJob은 이 전파를 막음.

4 launch vs async

두 가지 코루틴 빌더 — 결과값 유무로 구분.

import asyncio

# fire-and-forget
asyncio.create_task(do_work())

# 결과가 필요한 경우
task = asyncio.create_task(load_user(1))
user = await task

# 병렬 실행
results = await asyncio.gather(
    load_user(1),
    load_user(2),
    load_user(3)
)
// launch: fire-and-forget (Job 반환, 결과 없음)
scope.launch {
    doWork()
}

// async: 결과 필요 (Deferred<T> 반환)
val deferred: Deferred<User> = scope.async {
    loadUser(1)
}
val user = deferred.await()

// 병렬
coroutineScope {
    val a = async { loadUser(1) }
    val b = async { loadUser(2) }
    val c = async { loadUser(3) }
    listOf(a.await(), b.await(), c.await())
}

// or: awaitAll
listOf(1, 2, 3).map { id ->
    async { loadUser(id) }
}.awaitAll()

5 Dispatcher

"코루틴을 어떤 스레드 풀에서 실행할지" 결정. Python asyncio는 단일 이벤트 루프이지만, Kotlin은 용도별 스레드 풀을 선택할 수 있습니다.

Dispatchers.Default

CPU 연산용. 코어 수만큼 스레드. 정렬, 계산, 파싱 등.

Dispatchers.IO

IO 바운드용. 64+ 탄력적 스레드. HTTP, DB, 파일 IO.

Dispatchers.Main

UI 스레드(Android). 서버에선 거의 안 씀.

Dispatchers.Unconfined

특수 용도. 호출 스레드에서 즉시 실행, 첫 suspend 후 재개 스레드로 이동.

// scope 생성시 지정
val ioScope = CoroutineScope(Dispatchers.IO + SupervisorJob())

// 빌더에서 override
scope.launch(Dispatchers.Default) {
    heavyCpuWork()
}

// 중간에 전환 — withContext
suspend fun loadAndProcess() {
    val raw = withContext(Dispatchers.IO) {
        httpClient.get("/data").body<String>()  // IO 풀에서 실행
    }
    val parsed = withContext(Dispatchers.Default) {
        parseHeavyJson(raw)  // CPU 풀로 전환
    }
    // 함수 리턴 시점엔 원래 Dispatcher로 복귀
}

6 구조화된 동시성 (⭐ 철학의 핵심)

모든 코루틴은 부모 scope에 속하고, 부모는 자식이 끝날 때까지 기다린다는 원칙. asyncio의 "task leak" 문제를 구조적으로 차단. 

suspend fun loadUserData(userId: Long): UserData = coroutineScope {
    // 이 블록 안의 모든 자식이 끝나야 coroutineScope가 리턴
    val profile = async { loadProfile(userId) }
    val posts   = async { loadPosts(userId) }
    val friends = async { loadFriends(userId) }
    UserData(profile.await(), posts.await(), friends.await())
    // 하나라도 실패하면 형제들 cancel
    // coroutineScope도 예외 전파
}
구조화 동시성의 이점
  1. 누수 방지: 부모가 종료되면 자식도 자동 종료. "떠도는 task"가 없음
  2. 에러 전파: 자식 실패가 부모까지 올라감 — try/catch로 잡을 수 있음
  3. 취소 전파: 부모 취소시 자식 모두 취소
  4. 스코프 = 생명주기: 명시적 계층으로 추적 가능

7 coroutineScope vs supervisorScope

자식 실패를 형제에 전파할지가 차이. 

// coroutineScope: 자식 1개 실패 → 전체 취소
coroutineScope {
    launch { loadA() }        // 이 중 하나 실패하면
    launch { loadB() }        // 나머지도 모두 취소
    launch { loadC() }
}

// supervisorScope: 자식 실패가 형제에 전파 안 됨
supervisorScope {
    launch { loadA() }        // 실패해도
    launch { loadB() }        // B, C는 계속 실행
    launch { loadC() }
}
프로젝트 사용 예: 워커 에러 격리
src/main/kotlin/.../worker/WorkerOrchestrator.kt 
suspend fun runAll(strategyProvider: (String) -> TradingStrategy) =
    supervisorScope {
        botProperties.contracts.forEach { contract ->
            launch { createWorker(contract, strategyProvider(contract)).run() }
        }
    }

매매봇에서 BTC 워커가 예외로 죽어도 ETH/SOL 워커는 계속 돌아야 함. supervisorScope가 형제 워커를 보호.

8 withContext와 Dispatcher 전환

suspend 함수 내부에서 일시적으로 다른 Dispatcher에서 실행하고 결과만 받기. async+await과 비슷하지만 단일 작업용. 

suspend fun processImage(id: Long): Image {
    // IO로 다운로드
    val bytes: ByteArray = withContext(Dispatchers.IO) {
        httpClient.get("/image/$id").body()
    }
    // CPU로 디코딩
    val image: Image = withContext(Dispatchers.Default) {
        decodeImage(bytes)  // 무거운 연산
    }
    return image
}

9 취소 (Cancellation)

코루틴 취소는 협력적: 코루틴이 suspend point에서 체크하고 스스로 종료합니다. 

val job = scope.launch {
    while (isActive) {       // 취소 체크
        doWork()
        delay(100)           // suspend point — 취소 체크
    }
}

delay(500)
job.cancel()     // 다음 suspend point에서 CancellationException
job.join()       // 완전 종료 대기
// 또는 cancelAndJoin()

// 취소를 일반 예외처럼 잡으면 안 됨
suspend fun doWork() {
    try {
        riskyCall()
    } catch (e: CancellationException) {
        throw e   // 반드시 rethrow
    } catch (e: Exception) {
        logger.error(e)
    }
}
CancellationException을 삼키지 마라

catch (e: Exception)은 CancellationException도 잡습니다(Exception의 하위). 이걸 삼키면 취소 신호가 무시되어 좀비 코루틴이 됩니다. 항상 CancellationException을 따로 잡아 rethrow하거나, runCatching 대신 명시적 try/catch 사용.

프로젝트 사용 예: CancellationException rethrow
src/main/kotlin/.../worker/CoinWorker.kt 
internal suspend fun runOnce() {
    try {
        rateLimiter.acquire()
        // ... 비즈니스 로직
    } catch (e: CancellationException) {
        throw e   // ← 반드시 rethrow, 그래야 워커 graceful shutdown
    } catch (e: Exception) {
        logger.error("워커 iteration 에러", e)
    }
}

10 에러 처리

// 1. suspend 함수 내부: try/catch 그대로
suspend fun loadUser(id: Long): User = try {
    httpClient.get("/users/$id").body()
} catch (e: HttpException) {
    throw UserNotFoundException(id, e)
}

// 2. coroutineScope + 자식 실패: 부모로 전파
suspend fun loadAll(): List<User> = try {
    coroutineScope {
        (1..5).map { async { loadUser(it.toLong()) } }.awaitAll()
    }
} catch (e: HttpException) {
    // 자식 중 하나라도 실패하면 여기서 잡힘
    emptyList()
}

// 3. launch + 글로벌 에러 핸들러
val handler = CoroutineExceptionHandler { _, e ->
    logger.error("코루틴 에러", e)
}
scope.launch(handler) { /* ... */ }
// 주의: handler는 launch에만 작동, async의 예외는 await()에서 던져짐

11 Flow (개괄)

Cold stream. Python의 async generator와 유사. 시간에 따라 방출되는 값들의 시퀀스. RxJava/Reactor 대체, 서버에서도 많이 쓰임(Kafka 컨슈머, WebSocket, 실시간 데이터). 

// Flow 생성
fun priceUpdates(): Flow<Double> = flow {
    while (true) {
        val price = fetchPrice()
        emit(price)          // 방출
        delay(1000)
    }
}

// 수집 (consumer) — terminal operator
suspend fun main() {
    priceUpdates()
        .filter { it > 100 }
        .map { "현재가: $it" }
        .collect { println(it) }   // collect가 실제 실행 트리거
}

// Cold 특성: collect 할 때마다 새로 실행
// Hot이 필요하면 SharedFlow / StateFlow

// 조합
combine(priceFlow, volumeFlow) { p, v -> "$p x $v" }
    .collect { println(it) }

12 gate-futures-bot 실제 사용

12.1 BotRunner: Spring 생명주기 ↔ 코루틴 연결

src/main/kotlin/.../worker/BotRunner.kt 
@Component
class BotRunner(
    private val orchestrator: WorkerOrchestrator,
    private val assembler: StrategyAssembler,
    /* ... */
) {
    // IO dispatcher + SupervisorJob으로 scope 구성
    private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())
    private var job: Job? = null

    @EventListener(ApplicationReadyEvent::class)
    fun start() {
        job = scope.launch {
            orchestrator.runAll { contract -> assembler.forContract(contract) }
        }
    }

    @PreDestroy
    fun stop() {
        val current = job ?: return
        runBlocking { current.cancelAndJoin() }  // 동기 콜백에서 코루틴 정리
        scope.cancel()
    }
}

포인트: Spring은 동기 생명주기(@PreDestroy)를 기대하는데, 코루틴 정리는 suspend 작업. runBlocking으로 브리지하는 관용구.

12.3 CoinWorker: suspend 루프 + cancellation

src/main/kotlin/.../worker/CoinWorker.kt 
suspend fun run() {
    MDC.put("contract", contract)
    try {
        while (currentCoroutineContext().isActive) {   // 취소 체크
            runOnce()
            delay(config.checkIntervalMillis)          // suspend point
        }
    } finally {
        logger.info("워커 종료")
        MDC.remove("contract")
    }
}

internal suspend fun runOnce() {
    try {
        rateLimiter.acquire()  // suspend!
        // ... 로직
    } catch (e: CancellationException) {
        throw e  // graceful shutdown 위해 rethrow
    } catch (e: Exception) {
        logger.error("워커 iteration 에러", e)
    }
}

12.4 InMemoryRateLimiter: Mutex + delay

src/main/kotlin/.../ratelimit/InMemoryRateLimiter.kt 
@Component
class InMemoryRateLimiter(/* ... */) : RateLimiter {
    private val mutex = Mutex()   // kotlinx.coroutines.sync.Mutex (suspend lock)

    override suspend fun acquire() {
        while (true) {
            val waitMillis: Long? = mutex.withLock {
                refill()
                if (tokens >= 1.0) {
                    tokens -= 1.0
                    null   // 획득 성공
                } else {
                    // 부족한 토큰이 생길 때까지의 ms
                    ((1.0 - tokens) / refillPerSec * 1_000).toLong()
                }
            }
            if (waitMillis == null) return
            delay(waitMillis)  // ← lock 바깥에서 대기 (다른 코루틴 진입 허용)
        }
    }
}

핵심 포인트: Mutex 안에서 suspend 함수 호출 금지 관용구. lock은 원자적 state 수정에만 쓰고, delay는 lock 밖에서.

13 흔한 실수 / 안티패턴

1. GlobalScope 사용
// ❌ 안티
GlobalScope.launch { loadUser() }

// ✅ 좋음
val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())
scope.launch { loadUser() }
// 또는 suspend 함수 + coroutineScope

GlobalScope는 lifecycle이 앱 전체 — 취소/에러 관리 불가. 구조화 동시성 위반.

2. suspend에서 blocking 함수 호출
// ❌ Thread 블록 — 다른 코루틴이 이 스레드를 못 씀
suspend fun bad() {
    Thread.sleep(1000)     // Dispatchers.IO 스레드 블록!
    val r = blockingHttp.get("...")  // 동기 IO
}

// ✅ suspend API 사용
suspend fun good() {
    delay(1000)
    val r = suspendHttp.get("...")  // suspend API
}

// ✅ blocking이 불가피하면 IO 풀로
suspend fun acceptable() = withContext(Dispatchers.IO) {
    legacyBlockingCall()   // IO 풀은 블록 허용 (확장 가능)
}
3. CancellationException 삼키기
// ❌ 취소 신호 무시
try { work() } catch (e: Exception) { log(e) }

// ✅ rethrow
try {
    work()
} catch (e: CancellationException) {
    throw e
} catch (e: Exception) {
    log(e)
}

// 또는 catch 순서로
try { work() }
catch (e: CancellationException) { throw e }
catch (e: Exception) { log(e) }
4. launch 안 예외 무시
// launch의 예외는 자동 전파 → 부모 scope 관리
// async의 예외는 await() 해야 던져짐

val deferred = async { risky() }
// deferred.await() 안 하면 예외 소실!
// (또는 scope가 취소되면서 보고되긴 함)

Q. 코루틴과 스레드의 차이를 설명해주세요.
A. 스레드는 OS가 스케줄링하는 실행 단위로 생성 비용이 크고(~1MB 스택) 컨텍스트 스위칭이 비쌉니다. 코루틴은 JVM 레벨의 경량 실행 단위로 continuation(다음 실행 위치를 가진 객체)으로 표현됩니다. 하나의 스레드에 수천 개의 코루틴이 실행될 수 있고, suspend point에서 자발적으로 양보하는 방식입니다. 서버 애플리케이션에선 많은 IO 대기를 적은 스레드로 처리할 수 있어 메모리/GC 효율이 좋습니다.
Q. coroutineScope와 supervisorScope의 차이는?
A. 자식 실패의 전파 여부가 차이입니다. coroutineScope는 자식 중 하나가 실패하면 형제 모두 취소하고 부모에게 예외 전파. supervisorScope는 자식 실패가 형제에 전파되지 않음. 실전 예: 매매봇에서 여러 코인 워커를 병렬로 돌릴 때, BTC 워커가 실패해도 ETH 워커가 계속 돌아야 하므로 supervisorScope. 반면 "모든 데이터가 필요한 aggregation"에선 하나라도 실패하면 의미 없으니 coroutineScope.
Q. Dispatcher의 종류와 용도는?
A. Default: CPU 코어 수만큼의 스레드. 정렬/계산/파싱 등 CPU 바운드. IO: 64+ 탄력적 스레드(블록 허용). HTTP/DB/파일 IO. Main: Android UI 스레드. 서버에선 거의 안 씀. Unconfined: 호출 스레드에서 즉시 실행, 첫 suspend 후 재개 스레드로. 테스트나 특수 상황에만. 일반 백엔드는 IO dispatcher가 기본이고, 무거운 연산만 withContext(Default)로 전환.
Q. 구조화된 동시성(Structured Concurrency)이 해결하는 문제는?
A. 비구조적 동시성(GlobalScope, fire-and-forget)의 3가지 문제: ①leak: 부모 종료 후 자식이 계속 돌아 리소스 누수. ②에러 소실: 자식 예외가 부모로 전파 안 돼 디버깅 어려움. ③취소 관리 불가: 모든 관련 작업을 개별 추적해야 함. 구조화된 동시성은 "모든 코루틴은 scope에 속하고, 부모는 자식 완료를 기다린다"는 원칙으로 이 문제들을 해결. Kotlin은 coroutineScope/supervisorScope/runBlocking 빌더로 이 원칙을 강제합니다.
Q. async와 launch는 언제 각각 쓰나요?
A. launch는 fire-and-forget(결과 없음). Job을 반환하며 취소/완료 추적용. async는 결과가 필요할 때. Deferred<T>(Future 같은 것)를 반환하고 await()로 결과 수신. 예외 처리도 다름: launch의 예외는 부모 scope로 전파, async의 예외는 await() 시점에 던져집니다. 여러 작업을 병렬 실행 후 결과를 모으려면 async + awaitAll().
Q. Python asyncio와의 차이를 한 문장으로?
A. asyncio는 단일 이벤트 루프에서 async 함수를 스케줄링하는 반면, Kotlin 코루틴은 여러 Dispatcher(스레드 풀)에 걸친 경량 실행 단위로 구조화된 동시성과 CPU/IO 워크로드 분리를 언어 차원에서 지원합니다. 또한 Flow를 통해 cold/hot 스트림을 통합합니다.
핵심 요약
  • suspend = 중단/재개 가능 함수. 다른 suspend 또는 코루틴 빌더에서만 호출
  • CoroutineScope = 생명주기. 모든 코루틴은 scope에 속함 (구조화된 동시성)
  • launch = fire-and-forget, async = 결과가 있는 병렬 작업
  • Dispatchers: Default(CPU) / IO(블록 허용) / Main(UI)
  • supervisorScope: 자식 실패 격리 (워커 패턴에 필수)
  • CancellationException은 삼키지 말고 rethrow
  • Mutex는 원자적 state 변경만, delay는 lock 밖에서
  • 테스트는 runTest(virtual time) 또는 runBlocking(실시간)
