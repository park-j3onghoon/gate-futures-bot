1 철학: Django vs Spring Boot

Django는 "Batteries included" — 프레임워크 하나에 모든 기능이 내장. Spring Boot는 "Starter로 조합" — 필요한 Starter를 의존성에 넣으면 자동 설정.

Django (Monolith 설계)

URL Router (urls.py)
View (views.py)
Form / Serializer (DRF)
Model (ORM) — 모두 Django 내장
Admin — 무료 제공
Auth / Session — 무료 제공

한 패키지에 다 있음. 설정 간단. 관례 따르면 OK.

Spring Boot (Modular 설계)

starter-web (HTTP)
starter-data-jpa (ORM)
starter-security (인증)
starter-data-redis (캐시)
starter-actuator (운영)
... 필요한 것만 골라 추가

조합 자유. 대규모 프로젝트에서 유연. 학습 곡선은 더 높음.

개념 매핑 (Django → Spring Boot)

DjangoSpring Boot비고
settings.pyapplication.yml + @ConfigurationPropertiesProfile로 환경 분리
urls.py@RequestMapping / @GetMapping컨트롤러 클래스에 붙임
views.py@RestController / @ControllerDRF ViewSet ≈ RestController
models.pyJPA @Entity + JpaRepository영속성 컨텍스트가 있다는 게 큰 차이
serializers.pyDTO class + Jackson 자동 변환Kotlin data class가 DTO로 적합
middlewareServlet Filter / HandlerInterceptor / Spring AOP계층별로 용도 다름
signalsSpring Events + @EventListenerApplicationEvent 기반
management commandCommandLineRunner / ApplicationRunner앱 시작 시 1회 실행
Django Admin직접 구현 (보통 별도 프론트엔드)Spring에 내장 Admin 없음
Celery + Beat@Scheduled / Spring Batch분산 작업은 Spring Cloud Task

2 진입점: @SpringBootApplication

# manage.py (Django 진입점)
import os
import sys

if __name__ == "__main__":
    os.environ.setdefault(
        "DJANGO_SETTINGS_MODULE",
        "myapp.settings"
    )
    from django.core.management import (
        execute_from_command_line
    )
    execute_from_command_line(sys.argv)

# 실행: python manage.py runserver
// Application.kt
@SpringBootApplication
class MyApplication

fun main(args: Array<String>) {
    runApplication<MyApplication>(*args)
}

// 실행: ./gradlew bootRun
// 또는: java -jar app.jar

@SpringBootApplication 3가지 어노테이션의 합

  • @SpringBootConfiguration: 이 클래스를 설정 클래스로 표시
  • @EnableAutoConfiguration: classpath 기반 자동 설정 활성화
  • @ComponentScan: 이 패키지와 하위에서 @Component/@Service/@Repository/@Controller 찾아 등록
Component Scan 범위

@SpringBootApplication이 붙은 클래스의 패키지와 그 하위만 스캔됩니다. 최상위 패키지(com.example)에 두어야 전체 프로젝트가 스캔됩니다. 하위 패키지에 두면 상위/sibling은 인식 안 됨.

3 Auto-configuration 마법

Spring Boot의 핵심 혁신. classpath에 어떤 라이브러리가 있느냐에 따라 Bean을 자동 구성합니다.

예: H2 in-memory DB

// build.gradle.kts에 추가만 하면
dependencies {
    implementation("org.springframework.boot:spring-boot-starter-data-jpa")
    runtimeOnly("com.h2database:h2")
}

자동으로 일어나는 일:

  1. H2 JDBC 드라이버가 classpath에 있음 감지
  2. DataSource Bean 자동 생성 (in-memory H2)
  3. JPA EntityManagerFactory Bean 자동 생성
  4. JpaRepository를 구현한 인터페이스 자동 구현 생성
  5. 트랜잭션 매니저 자동 구성

코드 변경 전혀 없이 DB 기반 앱이 동작합니다. application.yml에서 spring.datasource.url=jdbc:postgresql://...로 바꾸면 PostgreSQL로 자동 전환.

Auto-configuration 원리 (조건부 Bean)

// Spring Boot 내부 코드 (의역)
@AutoConfiguration
@ConditionalOnClass(DataSource::class)      // DataSource 클래스가 classpath에 있을 때만
@ConditionalOnMissingBean(DataSource::class) // 사용자가 DataSource Bean을 안 만든 경우만
class DataSourceAutoConfiguration {
    @Bean
    fun dataSource(properties: DataSourceProperties): DataSource = ...
}

조건이 맞으면 활성, 사용자가 직접 Bean을 정의하면 skip.

4 Starter 생태계

Spring Boot의 "가장 많이 쓰는 조합"을 미리 묶어놓은 패키지. 의존성 하나 추가로 관련 라이브러리 다수가 함께 들어옴.

spring-boot-starter-web

REST API. 내장 Tomcat + Spring MVC + Jackson

spring-boot-starter-webflux

리액티브 웹. Netty + Reactor

spring-boot-starter-data-jpa

Hibernate + HikariCP + JPA

spring-boot-starter-data-redis

Lettuce(또는 Jedis) + Spring Data Redis

spring-boot-starter-security

Spring Security (filter chain)

spring-boot-starter-validation

jakarta.validation (@NotBlank, @Positive 등)

spring-boot-starter-actuator

/health, /metrics, /info 운영 엔드포인트

spring-boot-starter-test

JUnit 5, Mockito, AssertJ, Spring Test

spring-boot-starter-logging

Logback (기본 포함)

spring-boot-starter-mail

JavaMail + Spring Email

spring-boot-starter-quartz

스케줄러 (cron, 지속성)

spring-boot-starter-amqp

RabbitMQ 통합

5 application.yml 설정

Django의 settings.py에 해당. 다만 YAML이 기본이고 Python 코드가 아닙니다(선언적). application.properties(key=value 형식)도 가능하나 YAML이 가독성 우위. 

# application.yml
spring:
  application:
    name: my-service

  datasource:
    url: jdbc:postgresql://localhost:5432/app
    username: ${DB_USER}            # 환경변수 참조
    password: ${DB_PASSWORD:}       # 기본값 빈 문자열
    hikari:
      maximum-pool-size: 20         # HikariCP 커넥션 풀

  jpa:
    hibernate:
      ddl-auto: validate            # validate/update/create/create-drop
    show-sql: false
    properties:
      hibernate.format_sql: true

  data:
    redis:
      host: localhost
      port: 6379

server:
  port: 8080
  compression:
    enabled: true

logging:
  level:
    root: INFO
    com.tnear: DEBUG
  pattern:
    console: "%d{HH:mm:ss.SSS} [%thread] %-5level %logger - %msg%n"

# 커스텀 속성 (우리 앱의 설정)
app:
  weather-api-url: https://api.weather.com
  cache-ttl-seconds: 300

환경변수 치환 규칙

  • ${DB_USER} — 환경변수, 없으면 에러
  • ${DB_USER:default} — 기본값 제공
  • ${DB_USER:${DEFAULT_USER:admin}} — 중첩 기본값

6 Profile (환경 분리)

Django의 settings.production / settings.local에 해당. 프로파일별 yml 파일을 자동 병합.

application-dev.yml

spring:
  datasource:
    url: jdbc:h2:mem:dev
logging.level.root: DEBUG

application-staging.yml

spring:
  datasource:
    url: jdbc:postgresql://stage-db:5432/app
logging.level.root: INFO

application-prod.yml

spring:
  datasource:
    url: jdbc:postgresql://prod-db:5432/app
logging.level.root: WARN

프로파일 활성 방법

# 1. 커맨드라인
java -jar app.jar --spring.profiles.active=prod

# 2. 환경변수
SPRING_PROFILES_ACTIVE=prod java -jar app.jar

# 3. application.yml
spring:
  profiles:
    active: dev

# 4. gradle bootRun
./gradlew bootRun --args='--spring.profiles.active=dev'

조건부 Bean (프로파일별)

@Component
@Profile("prod")  // prod 프로파일에서만 Bean으로 등록
class SesEmailSender : EmailSender { /* ... */ }

@Component
@Profile("!prod")  // prod가 아닐 때만
class ConsoleEmailSender : EmailSender { /* ... */ }

7 @ConfigurationProperties

YAML 설정을 타입 안전한 Kotlin 객체로 바인딩. Django settings의 dict 접근보다 훨씬 안전. 

# Django settings.py
WEATHER_API_URL = os.getenv("WEATHER_API_URL")
CACHE_TTL = int(os.getenv("CACHE_TTL", "300"))

# 사용
from django.conf import settings
settings.WEATHER_API_URL  # 오타나도 None 반환, 런타임 에러
// yaml
// app:
//   weather-api-url: https://...
//   cache-ttl-seconds: 300

@ConfigurationProperties(prefix = "app")
data class AppProperties(
    val weatherApiUrl: String,
    val cacheTtlSeconds: Int = 300
)

// 등록
@SpringBootApplication
@EnableConfigurationProperties(AppProperties::class)
class App

// 사용 (DI)
@Service
class WeatherService(private val props: AppProperties) {
    fun fetch() = httpClient.get(props.weatherApiUrl)
}
@Validated로 부팅 시 검증
@Validated
@ConfigurationProperties(prefix = "bot")
data class BotProperties(
    @field:NotEmpty val contracts: List<String> = listOf("BTC_USDT"),
    @field:Positive val leverage: Int = 5,
    @field:Positive val orderSize: Int = 1,
    @field:Positive val checkIntervalSec: Long = 60
)
// 잘못된 값으로 부팅 시도시 BindException으로 앱 시작 실패 (fail-fast)

8 앱 라이프사이클

Spring Boot 앱의 시작/종료 흐름과 훅 포인트.

// 1. @PostConstruct: Bean 생성 직후 1회
@Component
class MyBean {
    @PostConstruct
    fun init() { /* ... */ }
}

// 2. ApplicationRunner / CommandLineRunner: 앱 완전 시작 후 1회
@Component
class StartupRunner : ApplicationRunner {
    override fun run(args: ApplicationArguments) {
        // args.getOptionValues("name"), args.containsOption("verbose")
    }
}

// 3. @EventListener: 특정 이벤트 발생시
@Component
class Listener {
    @EventListener(ApplicationReadyEvent::class)
    fun onReady() { /* 웹 서버까지 떴을 때 */ }

    @EventListener(ContextClosedEvent::class)
    fun onClose() { /* 종료 시작 */ }
}

// 4. @PreDestroy: Bean 소멸 직전
@Component
class MyBean {
    @PreDestroy
    fun cleanup() { /* 리소스 정리 */ }
}

// 이벤트 순서:
// ContextRefreshed → @PostConstruct 완료 →
// ApplicationStarted → ApplicationRunner →
// ApplicationReady → (서비스 중) →
// ContextClosed → @PreDestroy

9 Actuator (운영 엔드포인트)

의존성 하나 추가하면 운영에 필요한 엔드포인트가 자동 제공. Django에는 없는 기능으로 직접 만들어야 하는 것들. 

# build.gradle.kts: implementation("org.springframework.boot:spring-boot-starter-actuator")

management:
  endpoints:
    web:
      exposure:
        include: health, info, metrics, prometheus
  endpoint:
    health:
      show-details: when-authorized
  metrics:
    export:
      prometheus:
        enabled: true

제공되는 주요 엔드포인트:

  • /actuator/health — 앱 상태 (DB/Redis 연결 체크 포함)
  • /actuator/info — 앱 정보(버전, 빌드 시간)
  • /actuator/metrics — Micrometer 메트릭 (JVM, HTTP, 커스텀)
  • /actuator/prometheus — Prometheus 스크래핑용
  • /actuator/loggers — 런타임 로그 레벨 변경
  • /actuator/threaddump, /actuator/heapdump — 디버깅

계층형 vs 도메인형 구조

위는 계층형(controller/service/repository). 프로젝트가 커지면 도메인형으로 전환: 

com.tnear/
├── user/
│   ├── UserController.kt
│   ├── UserService.kt
│   ├── User.kt (Entity)
│   └── UserRepository.kt
├── ads/
│   ├── AdsController.kt
│   ├── ...
└── common/

기능 변경시 관련 파일이 한 패키지에 모여 있어 수정 범위가 명확. 큰 팀/MSA에 유리.

12 예상 면접 질문

Q. Spring Boot의 Auto-configuration 원리는?
A. @EnableAutoConfiguration이 spring.factories(Spring Boot 2.x) 또는 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports(3.x) 파일에 나열된 AutoConfiguration 클래스들을 로드합니다. 각 클래스는 @ConditionalOnClass, @ConditionalOnMissingBean, @ConditionalOnProperty 등의 조건부 어노테이션을 통해 classpath와 기존 Bean 상태를 확인하고 조건이 맞을 때만 Bean을 등록합니다. 사용자가 직접 Bean을 정의하면 자동 설정은 skip되어 커스터마이즈 가능.
Q. @Component와 @Service, @Repository의 차이는?
A. 기능상 동일합니다 — 모두 컴포넌트 스캔 대상. 차이는 의도 표현과 특수 기능: @Service는 비즈니스 로직, @Repository는 데이터 접근 계층, @Controller/@RestController는 웹 계층. @Repository는 특별히 JDBC/JPA 예외를 DataAccessException으로 변환하는 추가 기능이 있습니다. 의도를 명확히 하면 팀원이 책임을 한눈에 파악하기 좋습니다.
Q. Profile을 어떻게 설계하시나요?
A. 3~4개 정도로 단순화합니다: local(개발자 PC, H2/embedded), dev(공유 개발 서버), staging(QA), prod. 공통 설정은 application.yml, 환경별은 application-{profile}.yml로 override. 민감 정보(API 키, DB 비밀번호)는 yml에 하드코딩하지 않고 환경변수(${DB_PASSWORD})나 AWS Secrets Manager/Parameter Store에서 주입. profile-specific Bean은 @Profile("prod")로 대체 구현 제공(예: 실제 이메일 vs 로그 출력).
Q. application.yml과 @ConfigurationProperties 중 @Value는 언제 쓰나요?
A. @Value("${app.name}")은 단일 값 주입용 — 간단한 한두 개 설정에 쓰면 됩니다. 하지만 여러 속성을 그룹화하거나 계층적 설정을 다룰 땐 @ConfigurationProperties가 유리합니다. 이유: ① 타입 안전(String → Int 변환 명시), ② 기본값을 Kotlin data class에 표현, ③ @Validated로 부팅 시 검증, ④ IDE가 yml 편집 시 자동완성 제공(spring-boot-configuration-processor). 실무에선 @Value는 자주 변하는 디버깅 플래그 정도에만 쓰고 대부분 @ConfigurationProperties.
Q. Spring과 Spring Boot의 차이는?
A. Spring Framework(2003년~)는 DI 컨테이너 + 모듈 집합. 강력하지만 XML 설정이 복잡하고 서버(WAS) 별도 설치 필요. Spring Boot(2014년~)는 Spring 위에 올린 opinionated wrapper. ①Auto-configuration으로 설정 최소화, ②내장 Tomcat으로 java -jar 실행, ③Starter로 의존성 조합 단순화. Spring Boot는 Spring을 대체한 게 아니라 사용하기 쉽게 포장한 것. 내부는 여전히 Spring Framework.
핵심 요약
  • Django "all-in-one" vs Spring Boot "조합형 Starter"
  • @SpringBootApplication = Configuration + AutoConfig + ComponentScan
  • Auto-configuration은 classpath + 조건부 Bean으로 동작
  • 설정은 application.yml + @ConfigurationProperties (타입 안전)
  • Profile로 환경 분리 (dev/staging/prod)
  • 라이프사이클 훅: @PostConstruct / ApplicationRunner / @EventListener(ApplicationReadyEvent) / @PreDestroy
  • Actuator로 운영 엔드포인트 자동 제공
